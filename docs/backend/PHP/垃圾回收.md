## 概述

PHP 使用自动内存管理，主要采用**引用计数（Reference Counting）**机制来管理变量的内存。从 PHP 5.3 开始，为了解决循环引用导致的内存泄漏问题，引入了**垃圾回收器（Garbage Collection）**。

## 引用计数机制

### 基本原理

PHP 的每个变量都存储在一个名为 `zval` 的数据结构中，该结构包含：

- **value**：变量的实际值
- **type**：变量的类型
- **refcount**：引用计数，记录有多少个变量指向这个值
- **is_ref**：是否为引用类型

### 引用计数的变化规则

| 操作              | 对引用计数的影响 |
| ----------------- | ---------------- |
| 变量赋值          | `refcount++`     |
| 变量销毁/重新赋值 | `refcount--`     |
| `refcount` 变为 0 | 立即释放内存     |

### 代码示例

```php
<?php
// 创建一个变量
$a = "hello";  // refcount = 1

// 引用赋值
$b = $a;       // refcount = 2

// 取消引用
unset($b);     // refcount = 1

// 销毁变量
unset($a);     // refcount = 0，内存被释放
```

## 循环引用问题

### 什么是循环引用

当两个或多个对象相互引用，形成一个闭环，且外部没有变量引用它们时，就会出现循环引用。

### 循环引用示例

```php
<?php
class Node {
    public $parent;
    public $children = [];
}

$parent = new Node();
$child = new Node();

// 建立双向引用
$parent->children[] = $child;
$child->parent = $parent;

// 外部引用被销毁
unset($parent, $child);

// 此时两个对象互相引用，refcount 都为 1
// 但外部已无法访问，导致内存泄漏
```

### 数组的循环引用

```php
<?php
$a = [];
$a[] = &$a;  // 数组引用自身

unset($a);   // 循环引用导致 refcount 不为 0，内存无法释放
```

## 垃圾回收器（GC）

### 工作原理

PHP 5.3+ 引入的垃圾回收器采用**同步周期回收（Concurrent Cycle Collection）**算法：

1. **根缓冲区（Root Buffer）**：存储可能产生循环引用的 zval
2. **周期检测**：当根缓冲区满时（默认 10000 个），触发垃圾回收
3. **标记-清除**：从根节点开始遍历，标记可达对象，清除不可达对象

### 垃圾回收流程

```
1. 识别潜在的循环引用（对象、数组）
2. 将潜在根存入根缓冲区
3. 缓冲区满时触发垃圾回收
4. 深度遍历分析引用关系
5. 回收真正的垃圾内存
```

### 配置与调优

```php
<?php
// 查看垃圾回收是否启用
var_dump(gc_enabled());  // bool(true)

// 启用/禁用垃圾回收
gc_enable();
gc_disable();

// 手动触发垃圾回收
$collected = gc_collect_cycles();
echo "回收了 {$collected} 个循环引用\n";

// 获取根缓冲区中的数量
$status = gc_status();
/*
[
    'runs' => 5,           // GC 运行次数
    'collected' => 100,    // 回收的对象数
    'threshold' => 10000,  // 触发阈值
    'roots' => 50          // 当前根缓冲区数量
]
*/
```

## PHP 7 的变化

### zval 结构优化

PHP 7 对 zval 进行了重大重构：

| 特性         | PHP 5        | PHP 7            |
| ------------ | ------------ | ---------------- |
| 大小         | 24 字节      | 16 字节          |
| 引用计数位置 | zval 内部    | 复杂类型单独存储 |
| 简单类型     | 需要引用计数 | 不需要引用计数   |

### 改进点

1. **简单类型直接存储**：整数、浮点数等不再使用引用计数
2. **字符串优化**：小字符串直接嵌入 zval
3. **数组优化**：数组使用新的 `zend_array` 结构
4. **GC 效率提升**：减少了需要 GC 管理的对象数量

```php
<?php
// PHP 7 中，以下变量不再使用引用计数
$a = 1;      // 整数，直接值存储
$b = 3.14;   // 浮点数，直接值存储
$c = true;   // 布尔值，直接值存储

// 以下类型仍使用引用计数
$d = [];           // 数组
$e = new stdClass(); // 对象
$f = "长字符串..."; // 长字符串
```

## 性能考虑

### GC 对性能的影响

1. **触发时机**：GC 在缓冲区满时触发，可能导致短暂停顿
2. **内存 vs CPU**：GC 用 CPU 时间换取内存安全
3. **适用场景**：
   - 长生命周期脚本（CLI、Worker）
   - 大量对象操作的场景

### 优化建议

```php
<?php
// 1. 短时间脚本可以禁用 GC
if (php_sapi_name() === 'cli' && $shortRunning) {
    gc_disable();
}

// 2. 手动控制 GC 时机
$heavyOperation = function() {
    gc_disable();
    // 执行大量对象操作...
    gc_enable();
    gc_collect_cycles(); // 在合适时机手动回收
};

// 3. 避免不必要的循环引用
// 不好的做法
$a = [];
$a['self'] = &$a;

// 好的做法：使用弱引用（PHP 7.4+）
$weakRef = WeakReference::create($object);
```

## 调试与监控

### 内存使用监控

```php
<?php
// 查看当前内存使用
echo memory_get_usage() / 1024 / 1024 . " MB\n";

// 查看峰值内存
echo memory_get_peak_usage() / 1024 / 1024 . " MB\n";

// 监控 GC 状态
$before = gc_status();
// ... 执行代码 ...
$after = gc_status();

echo "GC 运行次数: " . ($after['runs'] - $before['runs']) . "\n";
echo "回收对象数: " . ($after['collected'] - $before['collected']) . "\n";
```

### 循环引用检测

```php
<?php
// 检测潜在的循环引用
function detectCircularRef($var, &$seen = [], $path = '') {
    if (!is_object($var) && !is_array($var)) {
        return;
    }
    
    $id = spl_object_id((object)$var);
    if (isset($seen[$id])) {
        echo "发现循环引用: {$path}\n";
        return;
    }
    
    $seen[$id] = true;
    
    foreach ($var as $key => $value) {
        detectCircularRef($value, $seen, $path . '.' . $key);
    }
}
```

## 总结

| 机制     | 适用场景 | 特点               |
| -------- | -------- | ------------------ |
| 引用计数 | 所有变量 | 即时回收，简单高效 |
| 垃圾回收 | 循环引用 | 定期扫描，防止泄漏 |

PHP 的垃圾回收机制在大多数情况下都能很好地工作，开发者通常不需要手动干预。但在处理大量数据或长时间运行的脚本时，了解 GC 的工作原理可以帮助优化内存使用。
