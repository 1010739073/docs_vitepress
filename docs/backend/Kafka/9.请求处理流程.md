# Kafka 请求处理流程

## 网络架构模型

Kafka 采用 **Reactor 模式** 处理网络请求：

```
                    ┌─────────────────────────────────────────────────────────┐
                    │                      Kafka Broker                        │
                    │                                                         │
┌──────────┐        │  ┌─────────────────┐                                   │
│  Client  │───────>│  │  Acceptor       │                                   │
└──────────┘        │  │  (1 Thread)     │                                   │
                    │  └────────┬────────┘                                   │
                    │           │                                            │
                    │           ▼                                            │
                    │  ┌─────────────────┐      ┌─────────────────────────┐  │
                    │  │  Processor      │      │  RequestQueue           │  │
                    │  │  (N Threads)    │─────>│  (LinkedBlockingQueue)  │  │
                    │  │  Selector       │      └───────────┬─────────────┘  │
                    │  └─────────────────┘                  │                │
                    │                                       │                │
                    │                                       ▼                │
                    │                           ┌─────────────────────────┐  │
                    │                           │  RequestHandler Pool    │  │
                    │                           │  (IO Threads)           │  │
                    │                           └───────────┬─────────────┘  │
                    │                                       │                │
                    │                                       ▼                │
                    │                           ┌─────────────────────────┐  │
                    │                           │  ResponseQueue          │  │
                    │                           └───────────┬─────────────┘  │
                    │                                       │                │
                    │                                       ▼                │
                    │                           ┌─────────────────────────┐  │
                    │                           │  Processor              │  │
                    │                           │  (Send Response)        │  │
                    │                           └─────────────────────────┘  │
                    └─────────────────────────────────────────────────────────┘
```

## 核心组件

### 1. Acceptor

- 监听新连接
- 将连接分配给 Processor
- 单线程运行

### 2. Processor

- 维护 Selector，处理 IO 事件
- 从 SocketChannel 读取请求
- 将请求放入 RequestQueue
- 从 ResponseQueue 取出响应，发送给客户端
- 多线程运行（`num.network.threads`）

### 3. RequestHandler

- 从 RequestQueue 取出请求
- 处理请求业务逻辑
- 将响应放入 ResponseQueue
- 多线程运行（`num.io.threads`）

## 请求处理流程

### 完整流程

```
1. Client 建立连接 → Acceptor 接收
2. Acceptor 将连接分配给 Processor
3. Processor 从 SocketChannel 读取请求数据
4. Processor 将请求放入 RequestQueue
5. RequestHandler 从 RequestQueue 取出请求
6. RequestHandler 处理请求（可能涉及磁盘 IO）
7. RequestHandler 将响应放入 ResponseQueue
8. Processor 从 ResponseQueue 取出响应
9. Processor 将响应写入 SocketChannel
```

### 请求格式

```

│                       Request Header                         │
├─────────────────────────────────────────────────────────────┤
│ api_key (2B) │ api_version (2B) │ correlation_id (4B)       │

```

## 内存池设计

### ByteBuffer 池

Kafka 使用内存池避免频繁的内存分配和 GC：

```java
// BufferPool 实现
public class BufferPool {
    private final long totalMemory;
    private final int poolableSize;
    private final Deque<ByteBuffer> free;
    private final ReentrantLock lock;
    private final Condition waiters;

    // 分配内存
    public ByteBuffer allocate(int size) {
        lock.lock();
        try {
            if (size == poolableSize && !free.isEmpty()) {
                return free.pollFirst();
            }
            // ...
        } finally {
            lock.unlock();
        }
    }

    // 释放内存
    public void deallocate(ByteBuffer buffer) {
        lock.lock();
        try {
            if (buffer.capacity() == poolableSize) {
                buffer.clear();
                free.add(buffer);
            }
            // ...
        } finally {
            lock.unlock();
        }
    }
}
```

## IO 多路复用

### Selector 使用

Kafka 使用 Java NIO 的 Selector 实现 IO 多路复用：

```java
// Processor 中的核心逻辑
public void run() {
    while (true) {
        try {
            // 等待 IO 事件
            selector.select();

            // 处理 SelectionKey
            Set<SelectionKey> keys = selector.selectedKeys();
            for (SelectionKey key : keys) {
                if (key.isAcceptable()) {
                    // 处理新连接
                } else if (key.isReadable()) {
                    // 读取请求数据
                    handleRead(key);
                } else if (key.isWritable()) {
                    // 发送响应数据
                    handleWrite(key);
                }
            }
            keys.clear();
        } catch (IOException e) {
            // 异常处理
        }
    }
}
```

### IO 事件类型

| 事件 | 描述 | 触发条件 |
|------|------|----------|
| OP_ACCEPT | 接受连接 | 有新连接请求 |
| OP_CONNECT | 连接完成 | 连接建立完成 |
| OP_READ | 可读 | 有数据可读 |
| OP_WRITE | 可写 | 缓冲区可写 |

```java
props.put("batch.size", 16384);    // 批量大小
```


## 请求优先级

### 优先级队列

Kafka 使用优先级队列处理请求：

1. **高优先级**：Controller 请求、Admin 请求
2. **普通优先级**：生产请求、消费请求
3. **低优先级**：心跳请求

## 完整请求处理流程示例

### Produce 请求流程

```
1. Producer 构造 ProduceRequest
2. Producer 将请求发送到 Broker
3. Acceptor 接收连接，分配给 Processor
4. Processor 读取请求数据，放入 RequestQueue
5. RequestHandler 从 RequestQueue 取出请求
6. RequestHandler 解析请求，写入日志
7. RequestHandler 更新 HW，返回响应
8. Processor 发送响应给 Producer
9. Producer 收到确认
```

### Fetch 请求流程

```
1. Consumer 构造 FetchRequest
2. Consumer 将请求发送到 Broker
3. Acceptor 接收连接，分配给 Processor
4. Processor 读取请求数据，放入 RequestQueue
5. RequestHandler 从 RequestQueue 取出请求
6. RequestHandler 读取日志，构造响应
7. RequestHandler 将响应放入 ResponseQueue
8. Processor 发送响应给 Consumer
9. Consumer 处理消息
```
