# Kafka 控制器与集群管理

## Controller 概述

### Controller 是什么

Controller 是 Kafka 集群中的一个特殊 Broker，负责管理整个集群的状态。

### Controller 的职责

1. **分区状态管理** - 管理 Partition 的 Leader 选举
2. **副本状态管理** - 管理副本的 ISR 变化
3. **Topic 管理** - 创建、删除、修改 Topic
4. **Broker 管理** - 处理 Broker 的上线、下线
5. **重分配** - 执行 Partition 重分配
6. **Preferred Leader 选举** - 执行优先副本选举

## Controller 选举

### 选举流程

```
1. 所有 Broker 启动时，尝试在 ZooKeeper 创建 /controller 临时节点
2. 第一个成功创建的 Broker 成为 Controller
3. 其他 Broker 监听 /controller 节点变化
4. Controller 故障时，触发重新选举
```

### Controller Epoch

每次 Controller 变更后，epoch 递增：

```bash
# ZooKeeper 中的 Controller 信息
/controller
{
  "version": 1,
  "brokerid": 1,
  "timestamp": "1609459200000"
}

/controller_epoch
{
  "version": 1,
  "data": 5    # Controller 第5次变更
}
```

## Controller 工作原理

### ZooKeeper 监听

Controller 监听以下 ZooKeeper 路径：

| 路径 | 监听事件 | 处理动作 |
|------|----------|----------|
| /brokers/ids | Broker 上线/下线 | 触发 Partition 重分配 |
| /brokers/topics | Topic 创建/删除 | 创建/删除 Partition |
| /admin/reassign_partitions | 分区重分配 | 执行重分配 |
| /admin/preferred_replica_election | 优先副本选举 | 执行选举 |
| /isr_change_notification | ISR 变化 | 更新元数据 |

### 分区状态机

Controller 维护每个 Partition 的状态：

```
状态转换：
NonExistent → NewPartition
NewPartition → OnlinePartition
OnlinePartition → OfflinePartition
OfflinePartition → OnlinePartition
OfflinePartition → NonExistent
```

### 副本状态机

Controller 维护每个副本的状态：

```
状态转换：
NewReplica → OnlineReplica
OnlineReplica → OfflineReplica
OfflineReplica → OnlineReplica
OfflineReplica → ReplicaDeletionStarted
ReplicaDeletionStarted → ReplicaDeletionSuccessful
ReplicaDeletionSuccessful → NonExistentReplica
```

## Controller 故障转移

### 故障检测

```
1. Broker 监听 /controller 节点
2. Controller 故障时，/controller 节点被删除
3. 其他 Broker 收到通知，触发选举
```

### 故障转移流程

```
1. 检测到 Controller 故障
2. 所有 Broker 尝试创建 /controller 节点
3. 第一个成功的 Broker 成为新 Controller
4. 新 Controller 初始化：
   - 读取集群元数据
   - 初始化状态机
   - 恢复 Partition 的 Leader
   - 通知所有 Broker 更新元数据
```

## 元数据同步

### 元数据内容

- 所有 Topic 和 Partition 信息
- 每个 Partition 的 AR、ISR、Leader
- 所有 Broker 信息

### 同步机制

```
Controller ──[UpdateMetadata]──> Broker 1
           ├──[UpdateMetadata]──> Broker 2
           └──[UpdateMetadata]──> Broker 3
```

## 集群管理操作

### 查看 Broker 信息

```bash
# 查看集群 Broker 列表
bin/kafka-broker-api-versions.sh --bootstrap-server localhost:9092

# 查看集群元数据
bin/kafka-metadata-quorum.sh describe --bootstrap-server localhost:9092
```

### Topic 管理

```bash
# 创建 Topic
bin/kafka-topics.sh --create \n  --bootstrap-server localhost:9092 \n  --topic my-topic \n  --partitions 3 \n  --replication-factor 3

# 查看 Topic 详情
bin/kafka-topics.sh --describe \n  --bootstrap-server localhost:9092 \n  --topic my-topic

# 修改 Topic 配置
bin/kafka-configs.sh --alter \n  --bootstrap-server localhost:9092 \n  --entity-type topics \n  --entity-name my-topic \n  --add-config retention.ms=86400000

# 删除 Topic
bin/kafka-topics.sh --delete \n  --bootstrap-server localhost:9092 \n  --topic my-topic
```

### 分区重分配

```bash
# 生成分区重分配计划
bin/kafka-reassign-partitions.sh --generate \n  --bootstrap-server localhost:9092 \n  --topics-to-move-json-file topics.json \n  --broker-list "1,2,3"

# 执行分区重分配
bin/kafka-reassign-partitions.sh --execute \n  --bootstrap-server localhost:9092 \n  --reassignment-json-file reassignment.json

# 验证分区重分配
bin/kafka-reassign-partitions.sh --verify \n  --bootstrap-server localhost:9092 \n  --reassignment-json-file reassignment.json
```

### 增加分区

```bash
bin/kafka-topics.sh --alter \n  --bootstrap-server localhost:9092 \n  --topic my-topic \n  --partitions 6
```

## 集群扩缩容

### 扩容

```
1. 部署新 Broker
2. 启动新 Broker，自动注册到 ZooKeeper
3. Controller 检测到新 Broker
4. 执行分区重分配，将 Partition 迁移到新 Broker
```

### 缩容

```
1. 执行分区重分配，将 Partition 迁移走
2. 确认 Partition 已全部迁移
3. 停止 Broker
4. Broker 自动从集群中移除
```

## ZooKeeper 依赖

### ZooKeeper 存储的数据

```
/zookeeper
├── /brokers
│   ├── /ids          # Broker 列表
│   ├── /topics       # Topic 信息
│   └── /seqid
├── /controller       # Controller 信息
├── /controller_epoch # Controller 代际
├── /config
│   ├── /topics       # Topic 配置
│   ├── /clients      # 客户端配额
│   └── /users
├── /admin
│   ├── /delete_topics # 待删除 Topic
│   └── /reassign_partitions # 分区重分配
└── /isr_change_notification # ISR 变化通知
```

### ZooKeeper 配置

```properties
# ZooKeeper 连接地址
zookeeper.connect=localhost:2181

# ZooKeeper 会话超时
zookeeper.session.timeout.ms=18000

# ZooKeeper 连接超时
zookeeper.connection.timeout.ms=3000

# ZooKeeper 同步时间
zookeeper.sync.time.ms=2000
```

## KRaft 模式（无 ZooKeeper）

### 架构变化

Kafka 2.8+ 引入 KRaft 模式，不再依赖 ZooKeeper：

```
传统模式：
Producer/Consumer → Broker → ZooKeeper

KRaft 模式：
Producer/Consumer → Broker（内置 Raft 共识）
```

### KRaft 配置

```properties
# KRaft 模式配置
process.roles=broker,controller
node.id=1
controller.quorum.voters=1@localhost:9093,2@localhost:9094,3@localhost:9095
controller.listener.names=CONTROLLER
listeners=PLAINTEXT://:9092,CONTROLLER://:9093
inter.broker.listener.name=PLAINTEXT
```

### KRaft 优势

1. **简化部署** - 无需单独部署 ZooKeeper
2. **提高扩展性** - 元数据管理更高效
3. **减少故障点** - 不依赖外部系统
4. **统一监控** - 只需监控 Kafka

## Controller 最佳实践

### 1. 部署建议

- 奇数个 Broker，避免脑裂
- 跨机架部署，提高可用性
- 为 Controller 分配足够资源

### 2. 配置优化

```properties
# Controller 相关配置
controller.socket.timeout.ms=30000
controller.request.queue.size=50

# 分区重分配配置
num.recovery.threads.per.data.dir=1

# 元数据配置
metadata.max.age.ms=300000
```

### 3. 监控告警

```
- 监控 ActiveControllerCount，确保始终为 1
- 监控 OfflinePartitionsCount，及时处理故障
- 监控 Controller 性能，避免成为瓶颈
```
