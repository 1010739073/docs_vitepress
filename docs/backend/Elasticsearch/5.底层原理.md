# 5. 底层原理

## Lucene 核心原理

### 倒排索引 (Inverted Index)

倒排索引是 Lucene 的核心数据结构，实现了从词项到文档的快速映射。

**数据结构**：
```
┌─────────────────────────────────────────────────────────────┐
│                    倒排索引结构                              │
├──────────────┬──────────────────────────────────────────────┤
│   Term       │           Posting List                       │
├──────────────┼──────────────────────────────────────────────┤
│ "apple"      │ [doc1, doc3, doc5, ...]                      │
│ "iphone"     │ [doc1, doc2, doc4, ...]                      │
│ "pro"        │ [doc1, doc3, ...]                            │
│ "手机"       │ [doc1, doc2, ...]                            │
└──────────────┴──────────────────────────────────────────────┘
```

**Posting List 包含**：
- **文档ID**：包含该词项的文档
- **词频(TF)**：词项在文档中出现的次数
- **位置信息**：词项在文档中的位置（用于短语查询）
- **偏移信息**：词项在文档中的字符偏移

**创建过程**：
1. **分词**：将文档文本拆分为词项
2. **标准化**：转换为小写、去除停用词
3. **构建**：建立词项到文档的映射
4. **合并**：多个段(segment)合并优化

### FST (Finite State Transducer)

FST 是 Lucene 用于存储词项字典的高效数据结构。

**特点**：
- **前缀共享**：公共前缀只存储一次
- **空间高效**：压缩率可达 10:1
- **查询高效**：O(n) 时间复杂度

**FST 结构示例**：
```
    a
   /|\
  p  p  p
 /  |  \
p   p   p
|   |   |
l   p   l
|   |   |
e   l   e
    |
    e
```

**用途**：
- 快速词项查找
- 前缀匹配
- 范围查询

### BM25 评分算法

BM25 是 Elasticsearch 默认的相关性评分算法。

**公式**：
```
Score(q, d) = Σ (IDF(q_i) * TF(q_i, d) * (k1 + 1)) / (TF(q_i, d) + k1 * (1 - b + b * |d| / avgdl))
```

**参数说明**：
| 参数 | 说明 | 默认值 |
|------|------|--------|
| `k1` | 词频饱和系数 | 1.2 |
| `b` | 文档长度归一化系数 | 0.75 |
| `TF` | 词频 | - |
| `IDF` | 逆文档频率 | - |
| `|d|` | 文档长度 | - |
| `avgdl` | 平均文档长度 | - |

**IDF 计算**：
```
IDF(q_i) = log((N - n + 0.5) / (n + 0.5) + 1)
```
其中 N 是总文档数，n 是包含词项 q_i 的文档数。

### 索引合并策略

Lucene 使用分段(segment)存储，定期合并以优化查询性能。

**合并策略**：
| 策略 | 说明 | 适用场景 |
|------|------|----------|
| `TieredMergePolicy` | 分层合并策略 | 默认，通用场景 |
| `LogMergePolicy` | 对数合并策略 | 写入密集场景 |
| `LogByteSizeMergePolicy` | 按字节数合并 | 大索引场景 |

**合并过程**：
```
┌─────────────────────────────────────────────────────────────┐
│  Segment 合并过程                                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Segments:  [s1] [s2] [s3] [s4] [s5] [s6]                  │
│               │    │    │    │    │    │                    │
│               └────┴────┘    └────┴────┘                    │
│                       │            │                        │
│                       v            v                        │
│                  [merged1]    [merged2]                    │
│                       │            │                        │
│                       └────────────┘                        │
│                               │                             │
│                               v                             │
│                         [final_segment]                     │
└─────────────────────────────────────────────────────────────┘
```

## Elasticsearch 分布式原理

### 分片机制

**分片分配**：
```
┌─────────────────────────────────────────────────────────────┐
│                     集群分片分布                             │
├──────────┬──────────┬──────────┬──────────┬──────────┐     │
│   Node1  │   Node2  │   Node3  │   Node4  │   Node5  │     │
├──────────┼──────────┼──────────┼──────────┼──────────┤     │
│ P0  R1   │ P1  R2   │ P2  R3   │ P3  R4   │ P4  R0   │     │
│          │          │          │          │          │     │
│ (主)副本 │ (主)副本 │ (主)副本 │ (主)副本 │ (主)副本 │     │
└──────────┴──────────┴──────────┴──────────┴──────────┘     │
│                                                             │
│  P=Primary Shard  R=Replica Shard                           │
└─────────────────────────────────────────────────────────────┘
```

**分片分配规则**：
- 主分片均匀分布在不同节点
- 副本分片不与主分片在同一节点
- 副本分片均匀分布在不同节点

### 写操作流程

**文档写入流程**：
```
Client → Coordinating Node → Primary Shard → Replica Shards → Response
```

**详细步骤**：
1. **路由计算**：根据文档ID计算目标分片
   ```
   shard_id = hash(routing) % number_of_shards
   ```
2. **转发请求**：协调节点将请求转发到主分片所在节点
3. **主分片处理**：
   - 写入内存缓冲区 (Memory Buffer)
   - 写入事务日志 (Translog)
4. **复制到副本**：主分片将操作复制到所有副本分片
5. **返回响应**：所有副本确认后返回成功

**Translog 机制**：
- **作用**：保证数据持久化，防止节点宕机数据丢失
- **刷新策略**：
  - 默认 5 秒自动刷新
  - 可手动调用 `_flush` 强制刷新
  - 缓冲区满时自动刷新

### 读操作流程

**搜索请求流程**：
```
Client → Coordinating Node → All Shards → Aggregate → Response
```

**详细步骤**：
1. **查询解析**：协调节点解析查询语句
2. **分片选择**：选择所有相关分片（主分片或副本分片）
3. **并行查询**：向所有选中分片发送查询请求
4. **结果聚合**：协调节点合并各分片结果
5. **返回响应**：返回最终结果给客户端

**搜索类型**：
| 类型 | 说明 |
|------|------|
| `query_then_fetch` | 默认，先查询后获取 |
| `dfs_query_then_fetch` | 分布式频率计分 |
| `query_and_fetch` | 查询并立即获取 |
| `dfs_query_and_fetch` | 分布式频率计分并获取 |

### 近实时搜索

**Refresh 机制**：
- **内存缓冲区** → **Segment** → **磁盘**
- 默认 1 秒自动 refresh
- 可手动调用 `_refresh`

**Refresh 流程**：
```
┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐
│   Memory Buffer     │───→│      Segment        │───→│      磁盘           │
│   (可搜索)          │    │   (内存中,可搜索)   │    │   (持久化)          │
└─────────────────────┘    └─────────────────────┘    └─────────────────────┘
         │                          │
         └──────────────────────────┘
                Refresh 操作
```

**近实时性保证**：
- 数据写入后约 1 秒可搜索
- 牺牲强一致性换取高性能

### 分布式一致性

**一致性级别**：
| 级别 | 说明 |
|------|------|
| `one` | 仅主分片确认即可 |
| `quorum` | 过半分片确认 |
| `all` | 所有分片确认 |

**公式**：
```
quorum = (number_of_replicas + 1) / 2 + 1
```

**脑裂问题**：
- **原因**：网络分区导致多个主节点
- **解决方案**：
  - 最小主节点数配置
  - 投票机制
  - 脑裂检测

## 内存管理

### Heap 内存分配

**内存布局**：
```
┌─────────────────────────────────────────────────────────────┐
│                    JVM Heap                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                    Young Generation                   │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────────────┐   │   │
│  │  │  Eden    │→│  Survivor │→│  Survivor        │   │   │
│  │  │  Space   │  │  Space 0 │  │  Space 1        │   │   │
│  │  └──────────┘  └──────────┘  └──────────────────┘   │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                    Old Generation                     │   │
│  │  ┌────────────────────────────────────────────────┐  │   │
│  │  │  Lucene Segment Cache, Field Data Cache, etc. │  │   │
│  │  └────────────────────────────────────────────────┘  │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**GC 调优策略**：
- **G1GC**：推荐，适用于大堆内存
- **CMS**：低延迟优先，Java 8 默认
- **堆内存大小**：不超过物理内存的 50%，不超过 32GB

### 文件系统缓存

**操作系统缓存**：
- Lucene 段文件被操作系统缓存
- 热数据常驻内存
- 提升查询性能

**内存映射文件**：
```java
MappedByteBuffer buffer = fileChannel.map(
    MapMode.READ_ONLY, 0, fileSize);
```

**Page Cache 利用**：
- 操作系统自动管理
- 频繁访问的数据自动缓存
- 减少磁盘 I/O

## 线程池

### 线程池类型

| 线程池 | 用途 | 队列类型 |
|--------|------|----------|
| `search` | 搜索请求 | 有界队列 |
| `get` | 获取请求 | 有界队列 |
| `index` | 索引请求 | 有界队列 |
| `bulk` | 批量请求 | 有界队列 |
| `flush` | 刷新操作 | 有界队列 |
| `merge` | 合并操作 | 无界队列 |

### 线程池配置

```yaml
thread_pool:
  search:
    size: 10
    queue_size: 1000
    max_queue_size: 5000
    auto_queue_frame_size: 2000
```

### 线程池调优

**搜索密集型**：
- 增大 search 线程池
- 减小 index 线程池

**写入密集型**：
- 增大 index/bulk 线程池
- 适当减小 search 线程池

## 熔断机制

### 内存熔断

**熔断类型**：
| 类型 | 触发条件 | 保护对象 |
|------|----------|----------|
| `fielddata` | fielddata 内存超过阈值 | fielddata 缓存 |
| `request` | 请求内存超过阈值 | 单个请求 |
| `circuit_breaker` | 总内存超过阈值 | 整个节点 |

**熔断配置**：
```yaml
indices.breaker.fielddata.limit: 60%
indices.breaker.request.limit: 40%
indices.breaker.total.limit: 70%
```

**熔断处理流程**：
```
请求 → 检查内存 → 超过阈值 → 拒绝请求 → 返回 429 错误
                       │
                       ↓
                  正常处理
```

## 分片恢复

### 恢复流程

**分片恢复阶段**：
```
┌─────────────────────────────────────────────────────────────┐
│                    分片恢复流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 初始化 → 2. 主分片恢复 → 3. 副本分片恢复 → 4. 完成      │
│       │            │                  │                     │
│       ↓            ↓                  ↓                     │
│  检查状态    从磁盘加载        从主分片同步                   │
│              或从快照恢复      或从其他副本同步               │
└─────────────────────────────────────────────────────────────┘
```

**恢复策略**：
| 策略 | 说明 |
|------|------|
| `local` | 优先本地恢复 |
| `remote` | 从远程节点恢复 |
| `snapshot` | 从快照恢复 |

### 恢复优先级

**优先级顺序**：
1. **主分片**优先于副本分片
2. **小分片**优先于大片段
3. **活跃索引**优先于关闭索引

**恢复限速**：
```yaml
indices.recovery.max_bytes_per_sec: 50mb
indices.recovery.concurrent_streams: 2
```
