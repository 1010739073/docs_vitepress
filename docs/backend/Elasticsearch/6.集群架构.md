# 6. 集群架构

## 节点类型

### 主节点 (Master Node)

主节点负责管理集群的元数据和协调集群操作。

**职责**：
- 维护集群状态
- 管理分片分配
- 处理索引创建/删除
- 协调节点加入/退出

**配置**：
```yaml
node.name: master-node-1
node.roles: [master]
node.master: true
```

**主节点选举**：
- 使用 Bully 算法
- 需要过半节点同意
- 配置最小主节点数防止脑裂

### 数据节点 (Data Node)

数据节点负责存储数据和处理数据相关操作。

**职责**：
- 存储分片数据
- 处理搜索请求
- 执行聚合操作
- 参与分片恢复

**配置**：
```yaml
node.name: data-node-1
node.roles: [data]
node.data: true
```

### 协调节点 (Coordinating Node)

协调节点负责路由请求和聚合结果。

**职责**：
- 接收客户端请求
- 解析查询语句
- 路由请求到目标分片
- 聚合各分片结果

**配置**：
```yaml
node.name: coordinating-node-1
node.roles: []  # 不配置任何角色
```

### Ingest 节点

Ingest 节点负责数据预处理。

**职责**：
- 处理管道 (Pipeline)
- 数据转换和增强
- 文档预处理

**配置**：
```yaml
node.name: ingest-node-1
node.roles: [ingest]
```

### 机器学习节点 (ML Node)

机器学习节点负责运行机器学习任务。

**职责**：
- 执行异常检测
- 数据预测分析
- 模型训练和部署

**配置**：
```yaml
node.name: ml-node-1
node.roles: [ml]
```

## 集群状态

### 集群状态组成

```
┌─────────────────────────────────────────────────────────────┐
│                    Cluster State                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────┐  ┌─────────────────┐                  │
│  │   Nodes Info    │  │   Indices Info  │                  │
│  │  - 节点列表     │  │  - 索引列表     │                  │
│  │  - 节点状态     │  │  - 映射信息     │                  │
│  │  - 节点角色     │  │  - 设置信息     │                  │
│  └─────────────────┘  └─────────────────┘                  │
│                                                             │
│  ┌─────────────────┐  ┌─────────────────┐                  │
│  │   Shards Info   │  │   Routing Info  │                  │
│  │  - 分片位置     │  │  - 分片分配     │                  │
│  │  - 分片状态     │  │  - 恢复状态     │                  │
│  └─────────────────┘  └─────────────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

### 集群状态同步

**状态更新流程**：
1. 主节点更新本地状态
2. 通过 Publish-Subscribe 机制广播到所有节点
3. 各节点验证并应用状态
4. 确认收到更新

**一致性保证**：
- 使用版本号防止旧状态覆盖新状态
- 确保状态顺序一致性

## 分片分配

### 分配策略

**分片分配器**：
- **AwarenessAllocationDecider**：机架感知
- **DiskThresholdDecider**：磁盘空间阈值
- **FilterAllocationDecider**：节点过滤
- **SameShardAllocationDecider**：同分片避免

**分配规则**：
| 规则 | 说明 |
|------|------|
| 主分片均匀分布 | 防止单点故障 |
| 副本不与主分片同节点 | 提高可用性 |
| 副本均匀分布 | 负载均衡 |

### 机架感知

**配置**：
```yaml
cluster.routing.allocation.awareness.attributes: rack_id
node.attr.rack_id: rack1
```

**效果**：
- 主分片和副本分片分配到不同机架
- 防止单机架故障导致数据丢失
- 提高跨机架容错能力

### 磁盘感知

**配置**：
```yaml
cluster.routing.allocation.disk.watermark.low: 85%
cluster.routing.allocation.disk.watermark.high: 90%
cluster.routing.allocation.disk.watermark.flood_stage: 95%
```

**行为**：
| 水位 | 行为 |
|------|------|
| low | 停止分片分配到该节点 |
| high | 将分片迁移出该节点 |
| flood_stage | 只读模式 |

## 故障转移

### 节点故障检测

**检测机制**：
- **Ping 机制**：定期发送心跳包
- **超时检测**：无响应超过阈值标记为故障
- **Master 确认**：主节点确认节点状态

**故障判定**：
```
节点无响应 → 超过超时时间 → 主节点标记为故障 → 触发故障转移
```

### 分片恢复

**恢复流程**：
```
节点故障 → 主节点检测 → 重新分配分片 → 数据恢复
```

**恢复类型**：
| 类型 | 说明 |
|------|------|
| **Primary 恢复** | 从副本提升为主分片 |
| **Replica 恢复** | 从主分片同步数据 |
| **快照恢复** | 从快照恢复数据 |

### 脑裂处理

**原因**：
- 网络分区导致集群分裂
- 分裂后的集群各自选举主节点

**预防措施**：
```yaml
discovery.zen.minimum_master_nodes: 3  # (节点数/2)+1
```

**处理策略**：
- 检测脑裂
- 比较集群大小
- 较小集群停止服务
- 等待网络恢复

## 集群扩展

### 水平扩展

**添加节点流程**：
```
启动新节点 → 自动发现集群 → 加入集群 → 接收分片分配 → 数据同步 → 完成
```

**分片重平衡**：
```yaml
cluster.routing.rebalance.enable: all
cluster.routing.allocation.cluster_concurrent_rebalance: 2
```

### 垂直扩展

**升级节点配置**：
- 增加 CPU 核数
- 增加内存容量
- 增加磁盘容量

**注意事项**：
- 需重启节点
- 可能影响服务
- 建议逐步升级

## 跨集群复制

### 配置方式

**主动-被动模式**：
```yaml
PUT /_ccr/follow/my_follower_index
{
  "remote_cluster": "remote_cluster",
  "leader_index": "my_leader_index"
}
```

**主动-主动模式**：
- 双向复制
- 需要处理冲突
- 适合多数据中心

### 使用场景

| 场景 | 说明 |
|------|------|
| **灾难恢复** | 异地备份 |
| **数据迁移** | 平滑迁移 |
| **只读副本** | 分担查询压力 |
| **多活部署** | 跨区域高可用 |

## 监控与告警

### 监控指标

| 指标类型 | 关键指标 |
|----------|----------|
| **集群** | 健康状态、节点数、分片数 |
| **节点** | CPU、内存、磁盘、网络 |
| **索引** | 文档数、大小、查询速率 |
| **搜索** | 查询延迟、吞吐量、缓存命中率 |
| **写入** | 写入速率、失败率、刷新次数 |

### 监控工具

**内置工具**：
- `_cat` API
- `_cluster/health`
- `_nodes/stats`

**外部工具**：
- **Elasticsearch Monitoring**
- **Metricbeat**
- **Kibana Dashboard**

### 告警配置

**Watcher**：
```json
PUT _watcher/watch/disk_usage
{
  "trigger": {
    "schedule": { "interval": "10m" }
  },
  "input": {
    "http": {
      "request": {
        "url": "http://localhost:9200/_nodes/stats/fs",
        "method": "GET"
      }
    }
  },
  "condition": {
    "compare": {
      "ctx.payload.nodes.*.fs.total.used_percent": { "gt": 85 }
    }
  },
  "actions": {
    "send_email": {
      "email": {
        "to": "admin@example.com",
        "subject": "Disk Usage Alert",
        "body": "Disk usage exceeded 85%"
      }
    }
  }
}
```

## 安全机制

### 身份认证

**内置用户**：
| 用户 | 权限 |
|------|------|
| `elastic` | 超级管理员 |
| `kibana` | Kibana 专用 |
| `logstash_system` | Logstash 专用 |

**认证方式**：
- **内置用户认证**
- **LDAP 认证**
- **Active Directory 认证**
- **PKI 证书认证**

### 角色授权

**内置角色**：
| 角色 | 权限 |
|------|------|
| `superuser` | 所有权限 |
| `monitoring` | 监控权限 |
| `logstash_writer` | Logstash 写入权限 |

**自定义角色**：
```json
PUT /_security/role/my_role
{
  "indices": [
    {
      "names": ["my_index"],
      "privileges": ["read", "write"]
    }
  ]
}
```

### 传输加密

**配置**：
```yaml
xpack.security.transport.ssl.enabled: true
xpack.security.transport.ssl.verification_mode: certificate
xpack.security.transport.ssl.keystore.path: certs/elastic-certificates.p12
xpack.security.transport.ssl.truststore.path: certs/elastic-certificates.p12
```

**HTTPS 配置**：
```yaml
xpack.security.http.ssl.enabled: true
xpack.security.http.ssl.keystore.path: certs/http.p12
```
