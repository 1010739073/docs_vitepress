# Redis重要知识点及面试考点总结

## 一、Redis基础特性

### 1. 核心特点
- **内存数据库**：数据存储在内存中，读写速度极快
- **持久化**：支持RDB和AOF两种持久化方式
- **丰富的数据类型**：String、Hash、List、Set、Sorted Set等
- **单线程模型**：避免了多线程并发问题，简化设计
- **高并发**：单实例可支持10万QPS
- **原子操作**：支持事务和CAS操作

### 2. 数据结构及应用场景
| 数据类型 | 内部实现 | 应用场景 |
|---------|---------|--------|
| String | 动态字符串 | 缓存、计数器、分布式锁 |
| Hash | 哈希表 | 存储对象、用户信息 |
| List | 双向链表 | 消息队列、时间线 |
| Set | 哈希表 + 集合 | 去重、交集并集 |
| Sorted Set | 跳表 | 排行榜、范围查询 |
| HyperLogLog | 概率算法 | 基数统计 |
| Geo | 地理空间索引 | 附近的人、位置查询 |
| Stream | 消息流 | 消息队列、事件流 |

## 二、持久化机制

### 1. RDB（Redis Database）
- **原理**：定期将内存数据快照写入磁盘
- **优点**：恢复速度快，文件体积小
- **缺点**：可能丢失最后一次快照后的修改
- **配置**：`save 900 1`、`save 300 10`等

### 2. AOF（Append Only File）
- **原理**：记录所有写操作到日志文件
- **优点**：数据安全性高，最多丢失1秒数据
- **缺点**：文件体积大，恢复速度慢
- **配置**：`appendonly yes`、`appendfsync everysec`

### 3. 混合持久化
- **原理**：RDB + AOF结合，先写入RDB快照，再记录增量AOF
- **优势**：兼具RDB的恢复速度和AOF的数据安全性

## 三、高可用架构

### 1. 主从复制
- **原理**：主节点将数据同步到从节点
- **作用**：读写分离、故障转移
- **复制方式**：全量复制和增量复制
- **复制偏移量**：确保数据同步一致性

### 2. 哨兵机制（Sentinel）
- **功能**：监控、自动故障转移、配置中心
- **工作原理**：多个哨兵节点监控主从节点状态
- **故障转移流程**：发现主节点故障 → 选举新主节点 → 通知从节点切换

### 3. 集群模式（Cluster）
- **架构**：去中心化设计，数据分片
- **分片机制**：16384个哈希槽，每个节点负责部分槽
- **故障检测**：Gossip协议
- **水平扩展**：支持动态添加节点

## 四、缓存设计与问题

### 1. 缓存穿透
- **问题**：查询不存在的数据，导致请求直达数据库
- **解决方案**：布隆过滤器、缓存空值

### 2. 缓存雪崩
- **问题**：大量缓存同时过期，导致数据库压力骤增
- **解决方案**：过期时间随机化、缓存预热、分布式锁

### 3. 缓存击穿
- **问题**：热点key过期，大量请求同时访问
- **解决方案**：热点key永不过期、互斥锁、双层缓存

### 4. 缓存一致性
- **策略**：更新缓存策略（先更新数据库再更新缓存、先删除缓存再更新数据库）
- **延迟双删**：删除缓存 → 更新数据库 → 再次删除缓存

## 十、常见面试题详细解答与代码示例

### 1. 基础概念

#### 1.1 Redis与Memcached的区别

**详细解答**：
- **数据类型**：Redis支持String、Hash、List、Set、Sorted Set等多种数据类型，而Memcached只支持String类型
- **持久化**：Redis支持RDB和AOF持久化，Memcached不支持持久化
- **内存管理**：Redis使用jemalloc内存分配器，Memcached使用Slab Allocation
- **高可用**：Redis支持主从复制、哨兵机制和集群模式，Memcached需要依赖客户端实现高可用
- **性能**：两者在简单场景下性能相近，但Redis在复杂操作下更有优势

#### 1.2 Redis的单线程模型为什么快

**详细解答**：
- **内存操作**：数据存储在内存中，避免了磁盘I/O的开销
- **单线程优势**：避免了多线程上下文切换的开销，避免了线程安全问题
- **I/O多路复用**：使用epoll/select/kqueue等I/O多路复用技术，高效处理并发连接
- **非阻塞I/O**：采用非阻塞I/O操作，提高了I/O利用率
- **优化的数据结构**：针对不同数据类型采用了高效的数据结构实现

#### 1.3 Redis的持久化机制对比

**详细解答**：
| 特性 | RDB | AOF |
|------|-----|-----|
| 原理 | 定期快照 | 记录写操作 |
| 文件大小 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢失数据 | 最多丢失1秒数据 |
| 性能影响 | 快照时可能阻塞 | 写操作时有I/O开销 |
| 适用场景 | 备份、灾难恢复 | 对数据安全性要求高的场景 |

### 2. 数据结构

#### 2.1 Redis的String类型最大容量是多少

**详细解答**：
- Redis的String类型最大容量是512MB
- 实际使用中应避免存储过大的String值，会影响性能

#### 2.2 Hash类型的实现原理

**详细解答**：
- Hash类型在底层有两种实现：
  1. **ziplist**：当哈希表的元素个数小于`hash-max-ziplist-entries`（默认512）且所有值的长度小于`hash-max-ziplist-value`（默认64字节）时使用
  2. **hashtable**：当不满足ziplist条件时使用
- ziplist是一种紧凑的编码格式，节省内存但操作效率较低
- hashtable操作效率高但内存占用较大

**代码示例**：
```bash
# 查看Hash类型的编码方式
redis> OBJECT ENCODING key
```

#### 2.3 Sorted Set的跳表实现原理

**详细解答**：
- 跳表是一种有序数据结构，通过维护多层索引来加速查找
- 每个节点包含多个指针，指向不同层级的其他节点
- 查找时从最高层开始，逐层向下，类似于二分查找
- 时间复杂度：查找、插入、删除均为O(logN)
- 空间复杂度：O(N)

### 3. 高可用

#### 3.1 主从复制的原理

**详细解答**：
- **全量复制**：从节点首次连接主节点时，主节点生成RDB文件并发送给从节点，从节点加载RDB文件后，主节点再发送增量命令
- **增量复制**：主节点将写操作记录到复制缓冲区，从节点通过复制偏移量获取增量数据
- **复制偏移量**：主从节点各自维护复制偏移量，确保数据同步一致性
- **复制积压缓冲区**：主节点维护一个固定大小的缓冲区，用于存储最近的写命令，当从节点网络中断后重连时，可以通过偏移量获取丢失的命令

**代码示例**：
```bash
# 配置从节点
redis> SLAVEOF master_ip master_port

# 查看复制状态
redis> INFO replication
```

#### 3.2 哨兵机制的工作流程

**详细解答**：
1. **监控**：哨兵节点定期向主从节点发送PING命令，检测节点状态
2. **判断故障**：当主节点无响应时，哨兵节点标记其为主观下线
3. **故障确认**：多个哨兵节点确认主节点故障，标记为客观下线
4. **选举领导者**：哨兵节点通过Raft算法选举出领导者
5. **故障转移**：领导者哨兵选择一个从节点作为新主节点，其他从节点指向新主节点
6. **通知**：更新配置并通知客户端

**配置示例**：
```conf
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel failover-timeout mymaster 180000
sentinel parallel-syncs mymaster 1
```

#### 3.3 集群模式的数据分片策略

**详细解答**：
- Redis Cluster使用哈希槽（Hash Slot）进行数据分片
- 共有16384个哈希槽，每个节点负责一部分槽
- 数据存储时，通过`CRC16(key) % 16384`计算键对应的槽位
- 节点间通过Gossip协议交换槽位信息
- 当节点添加或删除时，槽位会在节点间重新分配

**代码示例**：
```bash
# 查看集群信息
redis> CLUSTER INFO

# 查看槽位分配
redis> CLUSTER SLOTS

# 手动分配槽位
redis> CLUSTER ADDSLOTS 0 1 2 ... 5460
```

### 4. 缓存问题

#### 4.1 缓存穿透、雪崩、击穿的区别与解决方案

**详细解答**：
- **缓存穿透**：查询不存在的数据，导致请求直达数据库
  - 解决方案：布隆过滤器、缓存空值、接口层参数校验
- **缓存雪崩**：大量缓存同时过期，导致数据库压力骤增
  - 解决方案：过期时间随机化、缓存预热、多级缓存、分布式锁
- **缓存击穿**：热点key过期，大量请求同时访问
  - 解决方案：热点key永不过期、互斥锁、双层缓存

**代码示例**：
```java
// 布隆过滤器解决缓存穿透
public boolean mightContain(String key) {
    return bloomFilter.mightContain(key);
}

// 互斥锁解决缓存击穿
public String getCache(String key) {
    String value = redis.get(key);
    if (value == null) {
        if (redis.setnx("lock:" + key, "1", 10) == 1) {
            try {
                value = db.query(key);
                redis.set(key, value, 3600);
            } finally {
                redis.del("lock:" + key);
            }
        } else {
            Thread.sleep(100);
            return getCache(key);
        }
    }
    return value;
}
```

#### 4.2 缓存一致性的处理策略

**详细解答**：
- **先更新数据库，再更新缓存**：可能导致缓存不一致（并发更新时）
- **先删除缓存，再更新数据库**：可能导致缓存不一致（更新期间有查询）
- **延迟双删**：删除缓存 → 更新数据库 → 再次删除缓存（解决并发问题）
- **异步更新**：通过消息队列异步更新缓存

**代码示例**：
```java
// 延迟双删策略
public void updateData(String key, String value) {
    // 1. 删除缓存
    redis.del(key);
    
    // 2. 更新数据库
    db.update(key, value);
    
    // 3. 延迟删除缓存（解决读写并发问题）
    executor.schedule(() -> {
        redis.del(key);
    }, 500, TimeUnit.MILLISECONDS);
}
```

#### 6.1 分布式锁的实现（基于SETNX）

**详细解答**：
- 使用`SET key value NX EX seconds`命令实现分布式锁
- 保证锁的原子性操作
- 设置合理的过期时间，避免死锁
- 可使用RedLock算法提高可靠性

**代码示例**：
```java
public boolean acquireLock(String lockKey, String requestId, int expireTime) {
    return redis.set(lockKey, requestId, "NX", "EX", expireTime) != null;
}

public boolean releaseLock(String lockKey, String requestId) {
    String script = "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end";
    Object result = redis.eval(script, Collections.singletonList(lockKey), Collections.singletonList(requestId));
    return result.equals(1L);
}
```

#### 6.2 限流方案（基于滑动窗口）

**详细解答**：
- 使用Sorted Set实现滑动窗口限流
- 记录每个请求的时间戳
- 定期清理过期的请求记录
- 统计窗口内的请求数量，判断是否超过阈值

**代码示例**：
```java
public boolean isAllowed(String key, int limit, int windowSeconds) {
    long currentTime = System.currentTimeMillis();
    long windowStart = currentTime - windowSeconds * 1000;
    
    // 移除窗口外的请求
    redis.zremrangeByScore(key, 0, windowStart);
    
    // 获取当前窗口内的请求数
    long count = redis.zcard(key);
    
    if (count < limit) {
        // 添加当前请求
        redis.zadd(key, currentTime, String.valueOf(currentTime));
        // 设置过期时间，避免内存泄漏
        redis.expire(key, windowSeconds + 1);
        return true;
    }
    return false;
}
```

#### 6.3 消息队列的实现（基于List或Stream）

**详细解答**：
- **基于List**：使用LPUSH和BRPOP命令实现简单的消息队列
- **基于Stream**：Redis 5.0+引入的Stream数据类型，支持消息持久化、消费者组等特性

**代码示例**：
```java
// 基于List的消息队列
public void sendMessage(String queue, String message) {
    redis.lpush(queue, message);
}

public String receiveMessage(String queue) {
    return redis.brpop(0, queue)[1];
}

// 基于Stream的消息队列
public void sendStreamMessage(String stream, Map<String, String> message) {
    redis.xadd(stream, StreamEntryID.NEW_ENTRY, message);
}

public List<MapRecord<String, String, String>> receiveStreamMessage(String stream, String group, String consumer) {
    // 创建消费者组（如果不存在）
    try {
        redis.xgroupCreate(stream, group, StreamEntryID.LAST_ENTRY, true);
    } catch (Exception e) {
        // 消费者组已存在
    }
    // 消费消息
    return redis.xreadgroup(Consumer.from(group, consumer), 
                           StreamOffset.create(stream, ReadOffset.lastConsumed()));
}
```
