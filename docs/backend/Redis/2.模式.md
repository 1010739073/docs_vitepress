# Redis运行模式及相关面试题

## 1. Redis运行模式

### 1.1 单机模式
- **描述**：单个Redis实例独立运行
- **优点**：配置简单，部署方便
- **缺点**：单点故障，无法水平扩展
- **适用场景**：开发环境、测试环境或低流量应用

### 1.2 主从复制（Master-Slave）
- **描述**：一个Master节点，多个Slave节点，数据从Master复制到Slave
- **工作原理**：
  - Slave启动时会发送SYNC命令到Master
  - Master执行BGSAVE生成RDB文件并发送给Slave
  - Slave加载RDB文件，之后通过增量复制同步数据
- **优点**：
  - 提高读性能（读操作可分摊到Slave）
  - 提供数据备份
  - 实现故障转移基础
- **缺点**：
  - 主节点故障需要手动切换
  - 写操作仍集中在Master

### 1.3 哨兵模式（Sentinel）
- **描述**：在主从复制基础上，增加哨兵节点监控集群状态
- **功能**：
  - 监控Master和Slave的健康状态
  - 自动故障转移（Master故障时选举新Master）
  - 通知客户端新的Master地址
- **部署建议**：至少3个哨兵节点，分布在不同机器上
- **优点**：实现自动故障转移，提高可用性
- **缺点**：
  - 配置复杂度增加
  - 写操作仍集中在单个Master

### 1.4 集群模式（Redis Cluster）
- **描述**：将数据分布在多个节点上，每个节点负责一部分哈希槽
- **工作原理**：
  - 将16384个哈希槽分配到不同节点
  - 通过CRC16(key) % 16384计算键所属的哈希槽
  - 支持自动分片和故障转移
- **部署建议**：至少3个主节点，每个主节点至少1个从节点
- **优点**：
  - 水平扩展（增加节点可提高容量）
  - 自动故障转移
  - 高可用性
- **缺点**：
  - 配置复杂
  - 不支持多键操作跨槽（如MSET多个不同槽的键）

### 1.5 伪集群模式
- **描述**：在单台机器上运行多个Redis实例模拟集群
- **适用场景**：开发测试环境
- **注意**：生产环境不建议使用

## 2. 相关知识点

### 2.1 持久化
- **RDB**：
  - 原理：定期生成内存快照
  - 优点：恢复速度快，文件小
  - 缺点：可能丢失最近数据
- **AOF**：
  - 原理：记录所有写操作
  - 优点：数据安全性高
  - 缺点：文件大，恢复慢
- **混合持久化**：结合RDB和AOF的优点

### 2.2 复制原理
- **全量复制**：首次同步或从节点断线时间较长时
- **增量复制**：从节点正常运行时的同步
- **复制偏移量**：用于标识复制进度
- **复制积压缓冲区**：用于增量复制
- **PSYNC**：Redis 2.8+的复制命令，支持部分重同步

### 2.3 哨兵机制
- **Quorum**：判断Master故障所需的哨兵数量
- **Leader选举**：哨兵之间选举出一个领导者执行故障转移
- **故障转移过程**：
  1. 确认Master故障
  2. 选举新Master
  3. 通知Slave成为新Master的从节点
  4. 通知客户端新的Master地址

### 2.4 集群原理
- **哈希槽**：16384个槽，用于数据分布
- **节点间通信**：Gossip协议
- **故障检测**：集群内节点互相监控
- **故障转移**：主节点故障时，从节点晋升为主节点
- **槽迁移**：添加或删除节点时的数据重分布

## 3. 面试题

### 3.1 基础概念
1. **Redis有哪些运行模式？各有什么优缺点？**
   - 单机模式：配置简单，单点故障
   - 主从复制：提高读性能，手动故障转移
   - 哨兵模式：自动故障转移，配置复杂
   - 集群模式：水平扩展，不支持跨槽多键操作

2. **Redis主从复制的工作原理是什么？**
   - 全量复制：Slave发送SYNC，Master生成RDB并发送
   - 增量复制：Master将写操作发送给Slave
   - 复制偏移量和复制积压缓冲区

3. **Redis哨兵的作用是什么？**
   - 监控节点健康状态
   - 自动故障转移
   - 通知客户端

4. **Redis集群如何实现数据分片？**
   - 使用16384个哈希槽
   - 通过CRC16(key) % 16384计算槽位
   - 每个节点负责一部分槽

### 3.2 进阶问题
1. **Redis主从复制中，从节点可以写数据吗？**
   - 默认情况下从节点是只读的，可通过slave-read-only配置修改
   - 但不建议在从节点写数据，因为会被主节点的复制覆盖

2. **Redis哨兵集群至少需要几个节点？为什么？**
   - 至少3个节点
   - 因为哨兵需要通过投票机制判断Master故障，2个节点可能出现脑裂

3. **Redis集群中，如何处理跨槽的多键操作？**
   - 使用哈希标签（hash tag）将多个键映射到同一槽
   - 例如：{user:1000}:name 和 {user:1000}:age 会被映射到同一槽

4. **Redis集群的最大节点数是多少？**
   - 理论上没有限制，但受限于哈希槽数量（16384）
   - 实际建议不超过1000个节点

### 3.3 实践问题
1. **如何优化Redis主从复制的性能？**
   - 使用SSD存储
   - 调整repl-backlog-size参数
   - 避免在主节点上执行大键操作
   - 合理设置复制缓冲区大小

2. **Redis哨兵故障转移的过程是怎样的？**
   - 哨兵检测到Master故障
   - 哨兵之间选举领导者
   - 领导者选择一个Slave作为新Master
   - 通知其他Slave连接新Master
   - 通知客户端新的Master地址

3. **Redis集群扩容的步骤是什么？**
   - 添加新节点
   - 将哈希槽从现有节点迁移到新节点
   - 等待槽迁移完成
   - 验证集群状态

4. **如何监控Redis集群的健康状态？**
   - 使用redis-cli --cluster check命令
   - 监控节点状态、槽分布、复制状态
   - 使用监控工具如Prometheus + Grafana

5. **Redis集群中，主节点故障后，从节点晋升为主节点的条件是什么？**
   - 从节点与原主节点的复制偏移量差距不能太大
   - 从节点的优先级（slave-priority）较高
   - 从节点处于在线状态

## 4. 总结

Redis的不同运行模式各有优缺点，选择合适的模式需要根据业务需求、数据量、可用性要求等因素综合考虑：

- **单机模式**：适合开发测试环境或低流量应用
- **主从复制**：适合读多写少的场景，提高读性能
- **哨兵模式**：适合需要高可用性的场景，实现自动故障转移
- **集群模式**：适合大规模应用，需要水平扩展的场景

在实际部署中，还需要考虑持久化策略、监控方案、备份策略等因素，确保Redis服务的稳定运行。
