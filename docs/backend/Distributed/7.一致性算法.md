# 7. 一致性算法

## 一、分布式一致性概述

### 1.1 什么是一致性

在分布式系统中，一致性是指多个节点对同一数据的视图保持一致。

**一致性级别**：

| 级别 | 说明 |
|------|------|
| **强一致性** | 所有节点在同一时间看到相同的数据 |
| **弱一致性** | 节点可能看到不同的数据，但最终会一致 |
| **最终一致性** | 经过一段时间后，所有节点数据会一致 |

### 1.2 一致性面临的挑战

- **网络延迟**：节点间通信需要时间
- **网络分区**：节点间无法通信
- **节点故障**：节点宕机或数据损坏
- **并发写入**：多个节点同时修改同一数据

## 二、Paxos 算法

### 2.1 Paxos 简介

Paxos 是 Leslie Lamport 提出的分布式一致性算法，用于解决分布式系统中的共识问题。

**核心思想**：
- 通过多轮投票达成共识
- 保证在存在故障和网络延迟的情况下仍能达成一致

### 2.2 Paxos 角色

| 角色 | 职责 |
|------|------|
| **Proposer** | 提出提案 |
| **Acceptor** | 接受或拒绝提案 |
| **Learner** | 学习最终达成的共识 |

### 2.3 Paxos 流程

**Phase 1：准备阶段**

```
Proposer                     Acceptor
    |                            |
    |  Prepare(n)               |
    |───────────────────────────►|
    |                            |
    |      Promise(n, v)        |
    |◄──────────────────────────|
```

1. Proposer 选择一个提案编号 n
2. 向所有 Acceptor 发送 Prepare(n) 请求
3. Acceptor 如果没有接受过编号大于 n 的提案，返回 Promise(n, v)
   - v 是 Acceptor 已经接受的最大编号提案的值

**Phase 2：接受阶段**

```
Proposer                     Acceptor
    |                            |
    |  Accept(n, v)             |
    |───────────────────────────►|
    |                            |
    |        Accepted(n)        |
    |◄──────────────────────────|
```

1. Proposer 收到过半数 Acceptor 的 Promise 响应
2. 如果收到了 v 值，使用 v；否则使用自己提议的值
3. 向所有 Acceptor 发送 Accept(n, v) 请求
4. Acceptor 如果没有接受过编号大于 n 的提案，接受该提案

**Phase 3：学习阶段**

```
Proposer                     Learner
    |                            |
    |    Learn(n, v)            |
    |───────────────────────────►|
```

1. Proposer 收到过半数 Acceptor 的 Accepted 响应
2. 将结果通知所有 Learner

### 2.4 Paxos 特性

- **安全性**：只有被提议的值才能被选定，且只有一个值被选定
- **活性**：最终会有一个值被选定
- **容错性**：可以容忍少于一半节点故障

### 2.5 Multi-Paxos

标准 Paxos 每选定一个值都需要经过两轮 RPC，效率较低。Multi-Paxos 通过选举一个 Leader 来优化：

1. Leader 负责提出所有提案
2. Leader 直接进入 Phase 2，跳过 Phase 1
3. 提高了吞吐量

## 三、Raft 算法

### 3.1 Raft 简介

Raft 是一种更易于理解的分布式一致性算法，由 Stanford 大学提出。

**核心思想**：
- 通过选举 Leader 简化共识过程
- 将问题分解为几个独立的子问题

### 3.2 Raft 角色

| 角色 | 状态 | 职责 |
|------|------|------|
| **Leader** | 领导者 | 处理客户端请求，复制日志到 Followers |
| **Follower** | 跟随者 | 接收 Leader 的日志复制，响应请求 |
| **Candidate** | 候选人 | 竞选 Leader |

### 3.3 Raft 状态机

```
          timeout
Follower ──────────────────► Candidate ───────► Leader
    │                            │              │
    │         receive votes      │              │
    │◄───────────────────────────┘              │
    │                                           │
    │            receive heartbeat              │
    │◄──────────────────────────────────────────┘
```

### 3.4 Leader 选举

**触发条件**：
- Follower 在选举超时时间内没有收到 Leader 的心跳

**选举流程**：

```
Candidate                    Follower
    |                            |
    |  RequestVote(rpc)          |
    |───────────────────────────►|
    |                            |
    |       VoteGranted          |
    |◄──────────────────────────|
```

1. Candidate 增加自己的任期号
2. 向所有其他节点发送 RequestVote RPC
3. 如果获得过半数投票，成为 Leader
4. 向所有节点发送心跳，维持 Leader 地位

**选举约束**：
- **安全性**：只有包含最新日志的 Candidate 才能当选
- **活性**：选举超时时间随机化，避免选票分裂

### 3.5 日志复制

**流程**：

```
Client          Leader          Follower1         Follower2
    |              |                 |                  |
    |  Set x=5     |                 |                  |
    |─────────────►|                 |                  |
    |              |  AppendEntries  |                  |
    |              |────────────────►|                  |
    |              |                 |                  |
    |              |                 |   AppendEntries  |
    |              |                 |◄─────────────────|
    |              |                 |                  |
    |              |    Success      |                  |
    |              |◄────────────────|                  |
    |              |                 |                  |
    |              |    Success      |                  |
    |              |◄───────────────────────────────────|
    |              |                 |                  |
    |  OK          |                 |                  |
    |◄─────────────|                 |                  |
```

**日志特性**：
- **一致性**：如果两个日志条目具有相同的索引和任期号，则它们存储相同的命令
- **完整性**：如果某个日志条目在某个节点上被提交，则该条目在所有更高任期的 Leader 上也存在

### 3.6 安全性保证

**Election Safety**：每个任期最多只有一个 Leader

**Leader Append-Only**：Leader 只追加日志，不删除或修改

**Log Matching**：如果两个日志条目具有相同的索引和任期号，则它们之间的所有条目都相同

**Leader Completeness**：如果某个日志条目在某个任期内被提交，则它会出现在所有更高任期的 Leader 中

### 3.7 Raft vs Paxos

| 特性 | Paxos | Raft |
|------|-------|------|
| 复杂度 | 较高 | 较低 |
| 可理解性 | 难 | 易 |
| 实现难度 | 大 | 小 |
| 容错能力 | 相同 | 相同 |
| 性能 | 较高 | 略低 |

## 四、ZAB 协议

### 4.1 ZAB 简介

ZAB（ZooKeeper Atomic Broadcast）是 ZooKeeper 使用的一致性协议。

**设计目标**：
- 保证顺序一致性
- 支持崩溃恢复
- 高性能

### 4.2 ZAB 阶段

**Phase 1：发现阶段**

1. 选举新的 Leader
2. 确定最新的事务日志

**Phase 2：同步阶段**

1. Leader 将自己的日志同步到所有 Follower
2. Follower 应用所有未应用的事务

**Phase 3：广播阶段**

1. Leader 接收客户端请求
2. 将事务广播到所有 Follower
3. 过半数确认后提交

### 4.3 ZAB 特性

- **原子性**：事务要么全部提交，要么全部回滚
- **顺序性**：事务按顺序执行
- **可靠性**：已提交的事务不会丢失

## 五、Gossip 协议

### 5.1 Gossip 简介

Gossip 协议（又称 Epidemic Protocol）是一种去中心化的一致性协议。

**核心思想**：
- 节点定期与随机选择的邻居节点交换信息
- 信息像病毒一样传播到整个网络

### 5.2 Gossip 类型

| 类型 | 说明 |
|------|------|
| **Push** | 节点将自己的信息推送给邻居 |
| **Pull** | 节点从邻居拉取信息 |
| **Push-Pull** | 双向交换信息 |

### 5.3 Gossip 流程

```
Node A          Node B          Node C          Node D
   |               |               |               |
   |  gossip(x)    |               |               |
   |──────────────►|               |               |
   |               |  gossip(x)    |               |
   |               |──────────────►|               |
   |               |               |  gossip(x)    |
   |               |               |──────────────►|
```

### 5.4 Gossip 特性

- **去中心化**：没有单点故障
- **容错性**：部分节点故障不影响整体
- **最终一致性**：最终所有节点会达成一致
- **传播时间**：O(log n)

### 5.5 Gossip 应用场景

- **集群成员管理**：发现新节点、检测故障节点
- **数据同步**：复制数据到所有节点
- **分布式哈希表**：路由信息传播

## 六、一致性哈希

### 6.1 一致性哈希简介

一致性哈希是一种特殊的哈希算法，用于分布式系统中的数据分片。

**核心思想**：
- 将节点和数据映射到一个环形空间
- 数据存储到顺时针方向最近的节点

### 6.2 一致性哈希流程

```
        0°
       / \
      /   \
     /     \
   NodeA   NodeB
     \     /
      \   /
       \ /
      180°
```

**步骤**：
1. 计算数据 key 的哈希值
2. 沿顺时针方向找到第一个节点
3. 将数据存储到该节点

### 6.3 虚拟节点

**问题**：节点数量较少时，数据分布不均匀

**解决方案**：为每个节点创建多个虚拟节点

```
        0°
       /|\
      / | \
     /  |  \
   VNode1 VNode2
     \   / \
      \ /   \
   NodeA     NodeB
```

### 6.4 一致性哈希特性

- **平衡性**：数据均匀分布
- **单调性**：节点增减时，只有少量数据需要迁移
- **分散性**：避免热点

## 七、分布式共识算法对比

| 算法 | 适用场景 | 复杂度 | 容错能力 | 性能 |
|------|---------|--------|----------|------|
| **Paxos** | 通用场景 | 高 | 高 | 中 |
| **Raft** | 通用场景 | 中 | 高 | 中 |
| **ZAB** | ZooKeeper | 中 | 高 | 中 |
| **Gossip** | 大规模集群 | 低 | 高 | 高 |
| **一致性哈希** | 数据分片 | 低 | 中 | 高 |

## 八、实践建议

### 8.1 选择合适的算法

| 场景 | 推荐算法 |
|------|---------|
| 需要强一致性 | Raft、ZAB |
| 需要最终一致性 | Gossip |
| 数据分片 | 一致性哈希 |
| 小规模集群 | Raft |
| 大规模集群 | Gossip |

### 8.2 部署建议

- **节点数量**：建议奇数个节点（3、5、7）
- **网络环境**：确保节点间网络稳定
- **监控告警**：监控节点状态、选举次数、日志同步延迟

### 8.3 性能优化

- **减少网络往返**：批处理请求
- **优化日志同步**：异步复制
- **合理配置超时时间**：避免频繁选举
