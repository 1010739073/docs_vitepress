# 分布式面试题

## 一、分布式系统基础

### Q1: 什么是分布式系统？分布式系统有哪些特点？

**答案**：

分布式系统是由多个独立的计算机节点组成的系统，这些节点通过网络进行通信和协调，共同完成任务。

**特点**：
1. **透明性**：用户无需了解系统内部结构
2. **开放性**：遵循标准协议，便于扩展
3. **可扩展性**：可以方便地增加节点
4. **可靠性**：单个节点故障不影响整体
5. **并发性**：多个节点可以并发处理

### Q2: CAP 理论是什么？为什么说只能三选二？

**答案**：

CAP 理论指出，分布式系统不可能同时满足以下三个要素：

- **C (Consistency) 一致性**：所有节点在同一时间看到的数据是一致的
- **A (Availability) 可用性**：每个请求都能得到响应
- **P (Partition Tolerance) 分区容错性**：系统在网络分区的情况下仍能继续运行

**为什么只能三选二**：

由于网络分区（P）是分布式系统固有的属性，无法避免，因此实际只能在 CP 和 AP 之间权衡：

- **CP 系统**：保证一致性和分区容错，牺牲可用性（如 ZooKeeper、etcd）
- **AP 系统**：保证可用性和分区容错，牺牲一致性（如 Eureka、Cassandra）

### Q3: 什么是 BASE 理论？

**答案**：

BASE 理论是对 CAP 理论的延伸，核心思想是**即使无法做到强一致性，每个应用也可以根据自身业务特点，采用适当的方式来使系统达到最终一致性**。

- **BA (Basically Available) 基本可用**：允许系统出现局部不可用
- **S (Soft state) 软状态**：允许系统中的数据存在中间状态
- **E (Eventually consistent) 最终一致性**：经过一定时间后，所有节点数据最终一致

### Q4: 分布式事务有哪些解决方案？分别适用于什么场景？

**答案**：

| 方案 | 适用场景 | 优缺点 |
|------|---------|--------|
| **2PC** | 对一致性要求高的场景 | 优点：强一致性；缺点：同步阻塞、单点故障 |
| **TCC** | 对性能要求高的场景 | 优点：性能好；缺点：代码侵入性强 |
| **本地消息表** | 消息队列场景 | 优点：实现简单；缺点：需要额外的消息表 |
| **事务消息** | RocketMQ 场景 | 优点：保证最终一致性；缺点：依赖特定 MQ |
| **Saga** | 长事务场景 | 优点：避免长事务锁定资源；缺点：状态管理复杂 |

**TCC 详解**：

Try-Confirm-Cancel 三个阶段：
- **Try**：资源预留，检查资源是否充足
- **Confirm**：确认执行，使用 Try 阶段预留的资源
- **Cancel**：取消执行，释放 Try 阶段预留的资源

**需要处理的问题**：
- 幂等性
- 空回滚（Try 未执行，Cancel 执行）
- 悬挂（Cancel 先于 Try 执行）

### Q5: 什么是分布式锁？有哪些实现方式？

**答案**：

分布式锁是在分布式环境下，多个进程/机器互斥地访问共享资源的一种机制。

**实现方式**：

1. **Redis 实现**：
   ```java
   // SET key value NX EX 30
   boolean locked = redisTemplate.setNX(lockKey, "1", 30, TimeUnit.SECONDS);
   if (locked) {
       try {
           // 执行业务
       } finally {
           // 释放锁（使用 Lua 脚本保证原子性）
           redisTemplate.eval("if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end",
               Collections.singletonList(lockKey), "1");
       }
   }
   ```

2. **ZooKeeper 实现**：
   ```java
   // 创建临时顺序节点
   String lockPath = zk.create("/locks/lock-", null, ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL_SEQUENTIAL);
   // 判断是否是最小的节点
   List<String> children = zk.getChildren("/locks", false);
   if (isSmallestNode(lockPath, children)) {
       // 获取锁成功
   } else {
       // 监听前一个节点
   }
   ```

3. **数据库实现**：
   ```sql
   -- 使用唯一索引
   INSERT INTO lock (lock_name, expire_time) VALUES ('my_lock', NOW() + INTERVAL 30 SECOND);
   -- 释放锁
   DELETE FROM lock WHERE lock_name = 'my_lock';
   ```

**对比**：
- Redis：性能高，但主从切换可能丢失锁
- ZooKeeper：可靠性高，但性能较低
- 数据库：简单，但性能差

### Q6: 分布式 ID 生成有哪些方案？

**答案**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **UUID** | 简单、唯一 | 无序、长度长 | 不需要有序的场景 |
| **数据库自增** | 简单、有序 | 性能差、扩展困难 | 小规模系统 |
| **Redis INCR** | 高性能、有序 | 持久化问题 | 中小规模 |
| **雪花算法** | 有序、高性能 | 时钟回拨问题 | 大规模分布式系统 |
| **号段模式** | 高性能 | ID 不连续 | 高并发场景 |

**雪花算法详解**：

64 bit 结构：
- 1 bit：符号位（始终为 0）
- 41 bit：毫秒级时间戳（可用约 69 年）
- 10 bit：机器 ID（5 bit 数据中心 + 5 bit 机器）
- 12 bit：序列号（每毫秒可生成 4096 个 ID）

**时钟回拨处理**：
1. 等待时钟追上
2. 抛出异常
3. 使用备用机器 ID

## 二、分布式缓存

### Q7: 什么是缓存穿透？如何解决？

**答案**：

缓存穿透是指查询一个不存在的数据，由于缓存没有命中，每次请求都会查询数据库，可能导致数据库压力过大。

**解决方案**：

1. **缓存空值**：
   ```java
   User user = redisTemplate.get(key);
   if (user != null) {
       if (user == NULL_USER) {
           return null; // 缓存的是空值
       }
       return user;
   }
   user = userDao.findById(id);
   if (user == null) {
       redisTemplate.set(key, NULL_USER, 5, TimeUnit.MINUTES);
   } else {
       redisTemplate.set(key, user);
   }
   ```

2. **布隆过滤器**：
   ```java
   BloomFilter<Long> filter = BloomFilter.create(Funnels.longFunnel(), 1000000, 0.01);
   // 加载所有存在的 ID
   for (Long id : allIds) {
       filter.put(id);
   }
   // 查询前先判断
   if (!filter.mightContain(id)) {
       return null;
   }
   ```

3. **参数校验**：在接口层进行参数校验，过滤非法请求

### Q8: 什么是缓存击穿？如何解决？

**答案**：

缓存击穿是指某个热点 key 在缓存过期的瞬间，大量请求同时访问，导致所有请求都打到数据库。

**解决方案**：

1. **互斥锁**：
   ```java
   boolean locked = redisTemplate.setNX(lockKey, "1", 10, TimeUnit.SECONDS);
   if (locked) {
       // 双重检查
       user = redisTemplate.get(key);
       if (user != null) {
           return user;
       }
       // 查询数据库
       user = userDao.findById(id);
       redisTemplate.set(key, user, 30, TimeUnit.MINUTES);
   } else {
       // 等待并重试
       Thread.sleep(100);
       return getUserById(id);
   }
   ```

2. **逻辑过期**：
   ```java
   class CacheValue<T> {
       T data;
       long expireTime; // 逻辑过期时间
   }
   if (cacheValue.getExpireTime() > System.currentTimeMillis()) {
       return cacheValue.getData();
   }
   // 已过期，返回旧数据，并异步更新
   executorService.submit(() -> rebuildCache(id));
   return cacheValue.getData();
   ```

### Q9: 什么是缓存雪崩？如何解决？

**答案**：

缓存雪崩是指大量缓存同时失效或缓存服务器宕机，导致所有请求都打到数据库。

**解决方案**：

1. **随机过期时间**：
   ```java
   int baseExpire = 30 * 60;
   int randomOffset = new Random().nextInt(5 * 60);
   int expire = baseExpire + randomOffset;
   redisTemplate.set(key, value, expire, TimeUnit.SECONDS);
   ```

2. **缓存预热**：系统启动时或定时任务，提前加载热点数据

3. **高可用缓存架构**：使用 Redis 哨兵或 Redis Cluster

4. **限流降级**：对请求进行限流或降级处理

### Q10: 如何保证缓存和数据库的一致性？

**答案**：

**方案一：先更新数据库，再删除缓存（推荐）**

```java
public void updateUser(User user) {
    userDao.update(user);
    redisTemplate.delete("user:" + user.getId());
}
```

**问题**：缓存删除失败，导致数据不一致

**改进**：添加重试机制或使用消息队列

**方案二：延迟双删**

```java
public void updateUser(User user) {
    redisTemplate.delete("user:" + user.getId());
    userDao.update(user);
    Thread.sleep(500);
    redisTemplate.delete("user:" + user.getId());
}
```

**方案三：消息队列保证最终一致性**

```java
public void updateUser(User user) {
    userDao.update(user);
    mqService.send("cache-invalidate", "user:" + user.getId());
}

@Consumer(topic = "cache-invalidate")
public void invalidateCache(String key) {
    redisTemplate.delete(key);
}
```

### Q11: Redis 为什么快？

**答案**：

1. **基于内存**：内存访问速度远快于磁盘
2. **单线程模型**：避免线程切换和锁竞争
3. **IO 多路复用**：使用 epoll/kqueue 实现高并发
4. **高效的数据结构**：如 SDS、跳跃表等
5. **操作系统优化**：利用操作系统的零拷贝、页面缓存

### Q12: Redis 的数据结构有哪些？底层如何实现？

**答案**：

| 类型 | 编码 | 底层结构 |
|------|------|---------|
| String | int | 整数 |
| String | embstr | 简单动态字符串（≤ 39 字节）|
| String | raw | 简单动态字符串（> 39 字节）|
| List | quicklist | quickList |
| Hash | ziplist | 压缩列表（< 512 个，value < 64 字节）|
| Hash | hashtable | 字典 |
| Set | intset | 整数集合（元素都是整数）|
| Set | hashtable | 字典 |
| ZSet | ziplist | 压缩列表（< 128 个，value < 64 字节）|
| ZSet | skiplist + dict | 跳跃表 + 字典 |

**关键数据结构**：
- **SDS**：Redis 自己实现的字符串，支持 O(1) 获取长度
- **字典**：使用哈希表实现，支持渐进式 rehash
- **跳跃表**：ZSet 底层实现，查找、插入、删除时间复杂度均为 O(log n)

### Q13: Redis 持久化有哪些方式？有什么区别？

**答案**：

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **RDB** | 生成数据集的时间点快照 | 文件紧凑，恢复速度快 | 可能丢失数据，fork 时内存占用翻倍 |
| **AOF** | 记录每个写操作命令 | 数据安全性高，可读性好 | 文件较大，恢复速度慢 |
| **混合持久化** | RDB + AOF | 兼顾速度和安全性 | 需要配置 |

**AOF 同步策略**：
- `always`：每次写入都同步，最安全但性能最差
- `everysec`：每秒同步一次，折中方案
- `no`：由操作系统决定，性能最好但可能丢失数据

## 三、消息队列

### Q14: 消息队列的作用是什么？

**答案**：

1. **解耦**：降低系统间的耦合度
2. **异步**：提高系统响应速度
3. **削峰**：应对流量高峰
4. **可靠**：保证消息不丢失

### Q15: 如何保证消息不丢失？

**答案**：

1. **发送阶段**：
   - 使用同步发送
   - 使用本地消息表
   - 发送成功后删除本地消息

2. **存储阶段**：
   - 开启消息持久化
   - 配置刷盘策略（如 RocketMQ 的 SYNC_FLUSH）
   - 配置多副本

3. **消费阶段**：
   - 手动 ACK
   - 消费成功后再 ACK
   - 消费失败不 ACK，消息重新入队

### Q16: 如何保证消息不重复消费？

**答案**：

**核心思想**：保证幂等性

1. **数据库唯一索引**：
   ```java
   try {
       orderDao.insert(order); // order_id 有唯一索引
   } catch (DuplicateKeyException e) {
       log.warn("Duplicate order: {}", order.getId());
   }
   ```

2. **分布式锁**：
   ```java
   RLock lock = redisson.getLock("order:" + order.getId());
   if (lock.tryLock(10, TimeUnit.SECONDS)) {
       if (isProcessed(order.getId())) {
           return;
       }
       doProcess(order);
       markProcessed(order.getId());
   }
   ```

3. **状态机**：
   ```java
   int updated = orderDao.updateStatus(order.getId(), OrderStatus.CREATED, OrderStatus.PROCESSING);
   if (updated == 0) {
       return; // 已处理或正在处理
   }
   ```

### Q17: 如何保证消息的顺序性？

**答案**：

1. **单队列单消费者**：所有消息发送到同一个队列，由同一个消费者消费

2. **分区有序（Kafka）**：
   ```java
   // 根据 key 分区
   ProducerRecord<String, String> record = new ProducerRecord<>("topic", orderId, message);
   producer.send(record);
   ```

3. **RocketMQ 顺序消息**：
   ```java
   // 发送顺序消息
   producer.send(msg, new MessageQueueSelector() {
       @Override
       public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
           Long orderId = (Long) arg;
           int index = (int) (orderId % mqs.size());
           return mqs.get(index);
       }
   }, orderId);

   // 消费顺序消息
   consumer.registerMessageListener(new MessageListenerOrderly() {
       @Override
       public ConsumeOrderlyStatus consumeMessage(List<MessageExt> msgs, ConsumeOrderlyContext context) {
           // 单线程消费，保证顺序
           for (MessageExt msg : msgs) {
               process(msg);
           }
           return ConsumeOrderlyStatus.SUCCESS;
       }
   });
   ```

### Q18: 消息积压如何处理？

**答案**：

1. **增加消费者**：增加 Consumer 数量（注意不能超过分区数）

2. **临时扩容**：
   - 将积压的消息转发到一个新的 Topic（分区数扩大 10 倍）
   - 部署 10 倍的消费者消费新 Topic
   - 消费完成后恢复原配置

3. **丢弃不重要的消息**：对于日志、统计类消息，可以直接丢弃部分

4. **异步批量处理**：批量消费提高吞吐量

### Q19: 如何实现延迟消息？

**答案**：

1. **RabbitMQ 延迟插件**：
   ```java
   rabbitTemplate.convertAndSend("delayed.exchange", "delayed.key", message, msg -> {
       msg.getMessageProperties().setDelay(30000); // 延迟 30 秒
       return msg;
   });
   ```

2. **RocketMQ 延迟消息**：
   ```java
   msg.setDelayTimeLevel(4); // 30 秒
   producer.send(msg);
   ```

3. **Redis 过期通知**：
   ```java
   redisTemplate.opsForValue().set("delay:order:" + orderId, orderId, 30, TimeUnit.MINUTES);
   @RedisKeyExpirationEventMessageListener
   public void onKeyExpired(String expiredKey) {
       cancelOrder(orderId);
   }
   ```

4. **时间轮（HashedWheelTimer）**：适用于单机应用

## 四、RPC 框架

### Q20: RPC 和 HTTP/RESTful 有什么区别？

**答案**：

| 特性 | RPC | RESTful |
|------|-----|---------|
| 传输协议 | TCP（自定义）| HTTP/HTTPS |
| 数据格式 | 二进制（高效）| JSON/XML（可读） |
| 性能 | 高 | 较低 |
| 调用方式 | 方法调用 | HTTP 请求（GET/POST）|
| 接口定义 | IDL（Thrift/Protobuf）| OpenAPI/Swagger |
| 跨语言 | 需要代码生成 | 原生支持 |
| 测试 | 需要客户端 | 浏览器可直接测试 |
| 适用场景 | 内部服务调用 | 对外 API |

### Q21: Dubbo 的工作原理是什么？

**答案**：

**核心流程**：

1. **服务启动**：Provider 启动时，向注册中心注册服务
2. **服务发现**：Consumer 启动时，从注册中心订阅服务
3. **服务调用**：Consumer 调用服务时，通过负载均衡选择一个 Provider
4. **服务通信**：通过 Netty 进行网络通信
5. **服务容错**：如果调用失败，根据容错策略处理

**核心组件**：
- **Container**：服务容器
- **Provider**：服务提供者
- **Consumer**：服务消费者
- **Registry**：注册中心
- **Monitor**：监控中心
- **Protocol**：远程调用层
- **Cluster**：集群容错层

### Q22: gRPC 有什么特点？

**答案**：

1. **基于 HTTP/2**：支持多路复用、流式传输
2. **使用 Protobuf**：高效的序列化方式
3. **支持多种语言**：自动生成客户端和服务端代码
4. **支持流式传输**：
   - 简单 RPC
   - 服务端流式 RPC
   - 客户端流式 RPC
   - 双向流式 RPC
5. **双向流通信**：支持实时通信

### Q23: RPC 的负载均衡有哪些策略？

**答案**：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **随机** | 随机选择 | 通用场景 |
| **轮询** | 依次选择 | 节点性能相近 |
| **加权轮询** | 按权重选择 | 节点性能不同 |
| **最小连接数** | 选择连接数最少的 | 长连接场景 |
| **一致性哈希** | 根据 key 哈希 | 需要会话保持 |

## 五、服务治理

### Q24: 什么是服务注册与发现？

**答案**：

**服务注册**：服务启动时将自己的网络信息（IP、端口）注册到注册中心

**服务发现**：服务消费者从注册中心获取服务提供者的地址列表

**主流注册中心对比**：

| 特性 | Eureka | Consul | Nacos | ZooKeeper |
|------|--------|--------|-------|-----------|
| CAP | AP | CP/AP | CP/AP | CP |
| 健康检查 | 客户端心跳 | TCP/HTTP/gRPC | TCP/HTTP | TCP |
| 配置中心 | 不支持 | 支持 | 支持 | 不支持 |

### Q25: Dubbo 和 Spring Cloud 有什么区别？

**答案**：

| 特性 | Dubbo | Spring Cloud |
|------|-------|--------------|
| 定位 | RPC 框架 | 微服务全家桶 |
| 通信协议 | Dubbo 协议（TCP）| HTTP/RESTful |
| 注册中心 | ZooKeeper/Nacos | Eureka/Nacos/Consul |
| 服务治理 | 完善 | 完善 |
| 配置中心 | 需要单独配置 | Spring Cloud Config |
| 调用方式 | 方法调用 | HTTP 调用 |
| 性能 | 高 | 较低 |
| 学习曲线 | 较陡 | 较平缓 |

**选择建议**：
- 内部微服务通信 → Dubbo
- 对外暴露 API → Spring Cloud
- 已经使用 Spring 生态 → Spring Cloud

### Q26: 什么是熔断降级？

**答案**：

**熔断**：当服务调用失败率达到一定阈值时，自动切断对该服务的调用，避免雪崩。

**降级**：当服务不可用时，提供一个兜底的响应，保证系统基本可用。

**实现方式**：
- **Sentinel**：阿里巴巴开源的流量控制组件
- **Hystrix**：Netflix 开源的熔断组件（已停止维护）

**示例**：
```java
@SentinelResource(value = "getUserById",
    fallback = "getUserByIdFallback",
    blockHandler = "getUserByIdBlockHandler")
public User getUserById(Long id) {
    return userService.getUserById(id);
}

// 降级方法
public User getUserByIdFallback(Long id, Throwable e) {
    return getDefaultUser();
}

// 限流方法
public User getUserByIdBlockHandler(Long id, BlockException e) {
    throw new ApiException("请求过于频繁");
}
```

### Q29: 如何实现分布式 session？

**答案**：

**方案一：Session 同步**

- 用户登录时，将 session 同步到所有节点
- 优点：实现简单
- 缺点：节点多时同步开销大

**方案二：客户端存储**

- 将 session 加密后存储在 Cookie 中
- 优点：不需要服务端存储
- 缺点：有安全风险，存储大小有限

**方案三：Redis 存储（推荐）**

- 将 session 存储在 Redis 中
- 所有节点共享 session
- 优点：集中管理、易于扩展
- 缺点：需要额外的 Redis

**方案四：粘性会话**

- 使用负载均衡的会话保持功能
- 同一用户的请求转发到同一节点
- 优点：实现简单
- 缺点：节点故障时 session 丢失

### Q30: 什么是分布式链路追踪？

**答案**：

分布式链路追踪是追踪一个请求在分布式系统中经过的所有服务的技术。

**核心概念**：
- **Trace**：一个完整的请求链路
- **Span**：链路中的每个环节
- **TraceID**：链路的唯一标识
- **SpanID**：环节的唯一标识

**主流工具**：
- **Zipkin**：Twitter 开源
- **Jaeger**：Uber 开源，兼容 Zipkin
- **SkyWalking**：国产，支持多种语言
- **Sleuth**：Spring Cloud 生态

**作用**：
- 故障定位
- 性能分析
- 依赖分析
