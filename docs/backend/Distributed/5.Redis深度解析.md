# Redis 深度解析

## 一、Redis 基础

### 1.1 Redis 特点

| 特性 | 说明 |
|------|------|
| **内存存储** | 所有数据存储在内存中，读写速度快 |
| **键值对存储** | 支持 String、List、Set、ZSet、Hash 等数据结构 |
| **单线程模型** | 主要命令处理是单线程，避免锁竞争 |
| **IO 多路复用** | 使用 epoll/kqueue 实现高并发 |
| **持久化** | 支持 RDB 和 AOF 两种持久化方式 |
| **支持事务** | MULTI/EXEC/DISCARD/WATCH |
| **支持发布订阅** | Pub/Sub 消息模式 |
| **支持 Lua 脚本** | 原子性执行多个命令 |
| **主从复制** | 支持主从复制和读写分离 |
| **集群** | Redis Cluster 支持数据分片 |

### 1.2 为什么 Redis 快

**核心原因**：

1. **基于内存**：内存访问速度远快于磁盘
2. **单线程模型**：避免线程切换和锁竞争
3. **IO 多路复用**：一个线程可以处理多个连接
4. **高效的数据结构**：如 SDS、跳跃表等
5. **操作系统优化**：利用操作系统的零拷贝、页面缓存

## 二、Redis 数据结构底层实现

### 2.1 SDS（Simple Dynamic String）

Redis 没有直接使用 C 语言的字符串，而是实现了 SDS。

**C 字符串的问题**：
- 获取字符串长度需要 O(n) 时间
- 二进制不安全（不能存储 \0）
- 每次修改可能需要重新分配内存
- 缓冲区溢出风险

**SDS 结构**：

```c
struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;        // 已使用长度
    uint8_t alloc;      // 总长度（不包括 NULL）
    unsigned char flags; // 标志位，标识 SDS 类型
    char buf[];         // 字节数组
};
```

**SDS 优势**：
- O(1) 获取字符串长度
- 二进制安全
- 减少内存重分配次数
- 兼容 C 字符串函数

### 2.2 链表

Redis 的 List 底层使用链表实现。

```c
typedef struct listNode {
    struct listNode *prev;
    struct listNode *next;
    void *value;
} listNode;

typedef struct list {
    listNode *head;
    listNode *tail;
    unsigned long len;
    void *(*dup)(void *ptr);
    void (*free)(void *ptr);
    int (*match)(void *ptr, void *key);
} list;
```

**特点**：
- 双向链表
- 无环
- 带表头和表尾指针
- 带链表长度计数器

### 2.3 字典（Dict）

Redis 的 Hash 和整体键空间都使用字典实现。

**字典结构**：

```c
// 哈希表节点
typedef struct dictEntry {
    void *key;
    union {
        void *val;
        uint64_t u64;
        int64_t s64;
        double d;
    } v;
    struct dictEntry *next;
} dictEntry;

// 哈希表
typedef struct dictht {
    dictEntry **table;      // 哈希表数组
    unsigned long size;     // 哈希表大小
    unsigned long sizemask; // 哈希表大小掩码
    unsigned long used;     // 已有节点数量
} dictht;

// 字典
typedef struct dict {
    dictType *type;     // 类型特定函数
    void *privdata;     // 私有数据
    dictht ht[2];       // 两个哈希表（用于 rehash）
    long rehashidx;     // rehash 索引
    unsigned long iterators; // 迭代器数量
} dict;
```

**渐进式 rehash**：

为了避免 rehash 对性能的影响，Redis 采用渐进式 rehash：

1. 为 ht[1] 分配空间
2. 维持 rehashidx 索引，从 0 开始
3. 每次执行增删改查时，将 ht[0] 上 rehashidx 索引的值迁移到 ht[1]
4. rehashidx 自增
5. 全部迁移完成后，将 ht[1] 设为 ht[0]，ht[0] 清空

**rehash 触发条件**：
- 负载因子 >= 1 且没有 BGSAVE/BGREWRITEAOF
- 负载因子 >= 5

### 2.4 跳跃表（Skiplist）

Redis 的 ZSet 底层使用跳跃表。

**跳跃表结构**：

```c
// 跳跃表节点
typedef struct zskiplistNode {
    sds ele;                  // 成员对象
    double score;              // 分值
    struct zskiplistNode *backward; // 后退指针
    struct zskiplistLevel {
        struct zskiplistNode *forward; // 前进指针
        unsigned long span;    // 跨度
    } level[];                 // 层
} zskiplistNode;

// 跳跃表
typedef struct zskiplist {
    struct zskiplistNode *header, *tail; // 表头和表尾
    unsigned long length;    // 节点数量
    int level;               // 最大层数
} zskiplist;
```

**跳跃表原理**：

```
Level 4:  1 ------------------------------------> 9
Level 3:  1 ------------------> 7 --------------> 9
Level 2:  1 --------> 4 -------> 7 --------> 8 --> 9
Level 1:  1 --> 2 --> 4 --> 5 --> 7 --> 8 --> 9 --> 10
          ↓    ↓    ↓    ↓    ↓    ↓    ↓    ↓
         val  val  val  val  val  val  val  val
```

**优势**：
- 查找、插入、删除时间复杂度均为 O(log n)
- 实现简单

### 2.5 整数集合（IntSet）

当 Set 中只包含整数且元素数量较少时，使用 intset。

```c
typedef struct intset {
    uint32_t encoding;  // 编码方式：int16/int32/int64
    uint32_t length;    // 元素数量
    int8_t contents[];  // 整数数组
} intset;
```

**升级机制**：
当新元素类型超过当前编码时，进行升级：
1. 分配新空间
2. 将元素从旧编码转换到新编码
3. 添加新元素

### 2.6 压缩列表（ZipList）

当 List、Hash、ZSet 元素数量较少且元素较小时，使用 ziplist。

**ziplist 结构**：

```
<zlbytes> <zltail> <zllen> <entry> <entry> ... <zlend>
```

| 字段 | 说明 |
|------|------|
| zlbytes | ziplist 总字节数 |
| zltail | 最后一个节点的偏移量 |
| zllen | 节点数量 |
| entry | 节点 |
| zlend | 结束标记（0xFF） |

**entry 结构**：

```
<previous_entry_length> <encoding> <content>
```

**问题**：
- 连续内存，修改时可能需要重新分配
- 级联更新问题（连锁反应）

### 2.7 ListPack（Redis 7.0+）

ListPack 是 ziplist 的改进版，解决了级联更新问题。

**改进**：
- 使用绝对偏移量代替相对偏移量
- 避免级联更新

### 2.8 QuickList

Redis 3.2+ 的 List 使用 quickList 实现。

**结构**：双向链表 + ziplist

```
[ziplist] <-> [ziplist] <-> [ziplist] <-> [ziplist]
```

**优势**：
- 兼顾内存效率和性能
- 避免单个 ziplist 过大

### 2.9 RedisObject

所有数据类型都由 redisObject 包装。

```c
typedef struct redisObject {
    unsigned type:4;        // 类型：string/list/hash/set/zset
    unsigned encoding:4;    // 编码方式
    unsigned lru:LRU_BITS;  // LRU 时间
    int refcount;           // 引用计数
    void *ptr;              // 指向底层数据结构
} robj;
```

**类型与编码对应关系**：

| 类型 | 编码 | 底层结构 |
|------|------|---------|
| String | int | 整数 |
| String | embstr | 简单动态字符串（≤ 39 字节）|
| String | raw | 简单动态字符串（> 39 字节）|
| List | quicklist | quickList |
| Hash | ziplist | 压缩列表（< 512 个，value < 64 字节）|
| Hash | hashtable | 字典 |
| Set | intset | 整数集合（元素都是整数）|
| Set | hashtable | 字典 |
| ZSet | ziplist | 压缩列表（< 128 个，value < 64 字节）|
| ZSet | skiplist + dict | 跳跃表 + 字典 |

## 三、Redis 持久化

### 3.1 RDB（Redis Database）

**原理**：在指定时间间隔内生成数据集的时间点快照。

**触发方式**：

1. **手动触发**：
   - `SAVE`：阻塞式，在主线程执行
   - `BGSAVE`：非阻塞，fork 子进程执行

2. **自动触发**：
   - 配置 save 规则：`save 900 1`（900 秒内至少 1 次写操作）
   - 主从复制时，从节点发起全量复制
   - 执行 FLUSHALL 命令
   - 退出 Redis 时

**RDB 文件结构**：

```
REDIS + RDB_VERSION + DBS + EOF + CHECKSUM
```

**优点**：
- 文件紧凑，恢复速度快
- 适合备份
- 主进程不影响

**缺点**：
- 可能丢失数据（最后一次快照后的数据）
- fork 时内存占用翻倍

**配置**：

```conf
# RDB 配置
save 900 1      # 900 秒内至少 1 次修改
save 300 10     # 300 秒内至少 10 次修改
save 60 10000   # 60 秒内至少 10000 次修改

# 文件名
dbfilename dump.rdb

# 文件目录
dir /var/lib/redis

# 是否压缩
rdbcompression yes

# 是否校验
rdbchecksum yes
```

### 3.2 AOF（Append Only File）

**原理**：记录每个写操作命令，恢复时重新执行。

**AOF 持久化流程**：

```
命令写入 → AOF 缓冲区 → AOF 文件
                       ↓
                   同步策略
```

**同步策略**：

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| always | 每次写入都同步 | 数据最安全 | 性能差 |
| everysec | 每秒同步一次 | 性能和安全折中 | 可能丢失 1 秒数据 |
| no | 由操作系统决定 | 性能最好 | 可能丢失较多数据 |

**AOF 重写**：

**目的**：合并重复命令，减少文件大小

**触发条件**：
- 手动触发：`BGREWRITEAOF`
- 自动触发：`auto-aof-rewrite-min-size` 和 `auto-aof-rewrite-percentage`

**重写过程**：
1. fork 子进程
2. 子进程遍历内存数据，生成新的 AOF 文件
3. 父进程继续处理命令，将新命令写入 AOF 重写缓冲区
4. 子进程完成重写，通知父进程
5. 父进程将重写缓冲区内容追加到新 AOF 文件
6. 原子替换旧 AOF 文件

**AOF 文件格式**：

```
*2              # 参数个数
$6              # 第一个参数长度
SELECT
$1              # 第二个参数长度
0
*3
$3
SET
$3
key
$5
value
```

**优点**：
- 数据安全性高
- 可读性好

**缺点**：
- 文件较大
- 恢复速度慢

**配置**：

```conf
# 开启 AOF
appendonly yes

# AOF 文件名
appendfilename "appendonly.aof"

# 同步策略
appendfsync everysec

# AOF 重写配置
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# AOF 加载出错时是否继续
aof-load-truncated yes
```

### 3.3 RDB vs AOF

| 特性 | RDB | AOF |
|------|-----|-----|
| 持久化方式 | 快照 | 记录命令 |
| 文件大小 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢失 | 安全 |
| 性能影响 | fork 时影响 | 根据策略 |
| 适用场景 | 备份 | 数据安全 |

### 3.4 混合持久化（Redis 4.0+）

**原理**：RDB + AOF 混合使用
- AOF 重写时使用 RDB 格式写入
- 重写后的新命令以 AOF 格式追加

**优点**：
- RDB 格式，恢复速度快
- AOF 格式，数据安全

**配置**：

```conf
aof-use-rdb-preamble yes
```

## 四、Redis 主从复制

### 4.1 复制流程

**全量复制**：

```
1. 从节点发送 SYNC 命令
2. 主节点执行 BGSAVE，生成 RDB 文件
3. 主节点将 RDB 文件发送给从节点
4. 从节点加载 RDB 文件
5. 主节点将缓冲区的写命令发送给从节点
6. 从节点执行写命令
```

**增量复制**：

```
1. 从节点发送偏移量和复制 ID
2. 主节点根据偏移量发送增量命令
```

### 4.2 复制 ID（replid）

**作用**：标识主节点

- `replid`：主节点 ID
- `replid2`：上一次主节点 ID（用于故障转移）

### 4.3 复制偏移量（replication offset）

**作用**：记录复制进度

- 主节点和从节点各自维护 offset
- 通过 offset 判断是否需要全量复制

### 4.4 复制积压缓冲区（replication backlog）

**作用**：存储最近执行的写命令

**大小**：`repl-backlog-size`

**用途**：从节点断线重连后，如果 offset 在缓冲区内，使用增量复制

### 4.5 复制类型

| 类型 | 说明 |
|------|------|
| 全量复制 | 第一次连接或 offset 不在缓冲区内 |
| 增量复制 | 断线重连且 offset 在缓冲区内 |

### 4.6 主从复制配置

```conf
# 主节点配置
port 6379

# 从节点配置
replicaof <masterip> <masterport>
masterauth <password>

# 只读
replica-read-only yes
```

## 五、Redis 哨兵（Sentinel）

### 5.1 哨兵作用

- **监控**：监控主从节点运行状态
- **通知**：通知管理员节点故障
- **自动故障转移**：将从节点升级为主节点
- **配置提供者**：提供主节点信息

### 5.2 哨兵架构

```
┌─────────┐
│ Sentinel │ ───┐
└─────────┘    │ 监控
┌─────────┐    │
│ Sentinel │ ───┼──→ ┌─────────┐
└─────────┘    │    │ Master  │
┌─────────┐    │    └────┬────┘
│ Sentinel │ ───┘         │
└─────────┘         ┌─────┴─────┐
                    │  Replica  │
                    └───────────┘
```

### 5.3 哨兵工作原理

**主观下线（SDOWN）**：
- 单个哨兵认为主节点下线

**客观下线（ODOWN）**：
- 超过 `quorum` 数量的哨兵认为主节点下线

**故障转移**：
1. 选举 leader 哨兵
2. 选择新的主节点（优先级高、offset 大、runid 小）
3. 执行 `SLAVEOF NO ONE`
4. 其他从节点指向新主节点
5. 通知客户端

### 5.4 哨兵配置

```conf
# 哨兵端口
port 26379

# 监控主节点
sentinel monitor mymaster <ip> <port> <quorum>

# 主节点密码
sentinel auth-pass mymaster <password>

# 故障转移超时
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1
sentinel failover-timeout mymaster 180000

# 哨兵间通信密码
sentinel auth-pass <sentinel-password>
```

## 六、Redis Cluster

### 6.1 集群架构

```
客户端
   │
   ↓
┌─────────────────────────────────────────┐
│         Redis Cluster (16384 slots)     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ Node A  │  │ Node B  │  │ Node C  │  │
│  │ 0-5460  │  │ 5461-10922│ 10923-16383│
│  └────┬────┘  └────┬────┘  └────┬────┘  │
│       └────────────┼──────────────┘      │
└────────────────────┼────────────────────┘
                     │
              槽位迁移、故障检测
```

### 6.2 槽位分配

- 16384 个槽位（为什么是 16384？）
- 通过 CRC16(key) % 16384 计算槽位
- 每个节点负责一部分槽位

### 6.3 哈希标签

**作用**：将相关 key 放到同一个节点

```bash
# user:1001:info 和 user:1001:orders 在同一节点
SET user:1001:info "..."
SET user:1001:orders "..."
```

### 6.4 集群通信

**Gossip 协议**：
- 节点间通过 ping/pong 消息通信
- 定期交换节点状态
- 检测节点故障

### 6.5 故障转移

1. 节点标记为 PFAIL（主观故障）
2. 超过半数主节点标记为 FAIL（客观故障）
3. 从节点发起选举
4. 获得过半票数的从节点成为新主节点
5. 其他从节点指向新主节点

### 6.6 集群配置

```conf
# 启用集群
cluster-enabled yes

# 集群配置文件
cluster-config-file nodes.conf

# 集群节点超时
cluster-node-timeout 15000

# 集群总线端口 = port + 10000
```

## 七、Redis 内存管理

### 7.1 内存淘汰策略

| 策略 | 说明 |
|------|------|
| noeviction | 不淘汰，内存满时返回错误 |
| allkeys-lru | 淘汰最少使用的 key |
| allkeys-lfu | 淘汰使用频率最低的 key（Redis 4.0+）|
| allkeys-random | 随机淘汰 |
| volatile-lru | 淘汰设置了过期时间的最少使用的 key |
| volatile-lfu | 淘汰设置了过期时间的使用频率最低的 key |
| volatile-random | 淘汰设置了过期时间的随机 key |
| volatile-ttl | 淘汰即将过期的 key |

### 7.2 LRU 算法实现

Redis 使用近似的 LRU 算法（不是精确的 LRU）：

- 每个 key 记录访问时间
- 随机采样 key，淘汰最久未访问的
- 默认采样 5 个，可通过 `maxmemory-samples` 配置

**原因**：精确 LRU 需要维护链表，性能开销大

### 7.3 LFU 算法实现（Redis 4.0+）

**原理**：
- 记录访问次数
- 访问次数随时间衰减
- 防止冷数据突然变热

**实现**：
- 8 bit 记录访问频率
- 16 bit 记录最后访问时间
- 访问时，概率增加计数器
- 定期衰减计数器

### 7.4 大 key 问题

**危害**：
- 读取慢
- 内存分配困难
- 阻塞主线程

**解决方案**：
- 拆分大 key
- 使用 hash 结构
- 避免存储大对象

## 八、Redis 事务

### 8.1 事务命令

| 命令 | 说明 |
|------|------|
| MULTI | 开启事务 |
| EXEC | 执行事务 |
| DISCARD | 取消事务 |
| WATCH | 监控 key |

### 8.2 事务特性

- **原子性**：EXEC 后的所有命令要么都执行，要么都不执行
- **隔离性**：事务执行期间不会被其他命令打断
- **无回滚**：命令执行失败不会回滚（与 ACID 不同）

### 8.3 乐观锁

```bash
WATCH balance
balance = GET balance
balance = balance - 100
MULTI
SET balance balance
EXEC
# 如果 balance 被其他客户端修改，EXEC 返回 nil
```

## 九、Redis 性能优化

### 9.1 连接池

使用连接池避免频繁创建/销毁连接。

### 9.2 Pipeline

批量执行命令，减少网络往返。

```java
List<Object> results = redisTemplate.executePipelined(new RedisCallback<Object>() {
    @Override
    public Object doInRedis(RedisConnection connection) {
        for (int i = 0; i < 1000; i++) {
            connection.set(("key:" + i).getBytes(), ("value:" + i).getBytes());
        }
        return null;
    }
});
```

### 9.3 Lua 脚本

原子性执行多个命令，减少网络往返。

```lua
-- 原子性操作
local current = redis.call('GET', KEYS[1])
if current == false then
    redis.call('SET', KEYS[1], ARGV[1])
    return 1
else
    return 0
end
```
