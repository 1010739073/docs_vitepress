# 分布式缓存

## 一、缓存基础

### 1.1 为什么使用缓存

**目的**：
- 减少数据库访问压力
- 提高系统响应速度
- 支持高并发场景

**适用场景**：
- 读多写少的数据
- 计算成本高的数据
- 热点数据

### 1.2 缓存位置

| 位置 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 浏览器缓存 | HTTP Cache-Control | 减少网络请求 | 数据更新不及时 |
| CDN 缓存 | 边缘节点缓存 | 就近访问 | 成本高 |
| 应用本地缓存 | Guava/Caffeine | 速度最快 | 数据不一致 |
| 分布式缓存 | Redis/Memcached | 多节点共享 | 网络开销 |

### 1.3 缓存读写策略

#### 1. Cache-Aside（旁路缓存）

**读操作**：
```
1. 先读缓存
2. 缓存命中，直接返回
3. 缓存未命中，读数据库
4. 将数据写入缓存
5. 返回数据
```

**写操作**：
```
1. 先更新数据库
2. 删除缓存
```

**为什么删除缓存而不是更新缓存？**
- 删除操作更快
- 避免并发写入导致的数据不一致
- 延迟加载，减少无效数据

#### 2. Write-Through（写穿透）

写操作同时更新缓存和数据库，保证一致性。

**缺点**：
- 写入性能降低
- 缓存中存储了可能不常用的数据

#### 3. Write-Behind（写回）

写操作只更新缓存，异步批量更新数据库。

**优点**：
- 写入性能高
- 减少数据库 I/O

**缺点**：
- 数据可能丢失
- 一致性差

## 二、缓存一致性问题

### 2.1 数据不一致的原因

1. 并发读写导致
2. 缓存过期时间设置不当
3. 主从延迟
4. 缓存和数据库操作顺序问题

### 2.2 如何保证缓存一致性

#### 方案一：先更新数据库，再删除缓存（推荐）

```java
public void updateUser(User user) {
    // 1. 更新数据库
    userDao.update(user);

    // 2. 删除缓存
    redisTemplate.delete("user:" + user.getId());
}
```

**存在的问题**：
极端情况下，缓存删除失败，导致数据不一致。

**解决方案**：
- 添加重试机制
- 使用消息队列保证最终一致性

#### 方案二：延迟双删

```java
public void updateUser(User user) {
    // 1. 删除缓存
    redisTemplate.delete("user:" + user.getId());

    // 2. 更新数据库
    userDao.update(user);

    // 3. 延迟删除缓存（延迟时间 > 数据库主从同步时间）
    Thread.sleep(500);
    redisTemplate.delete("user:" + user.getId());
}
```

#### 方案三：消息队列保证最终一致性

```java
public void updateUser(User user) {
    // 1. 更新数据库
    userDao.update(user);

    // 2. 发送消息到 MQ
    mqService.send("cache-invalidate", "user:" + user.getId());
}

// 消费者消费消息，删除缓存
@Consumer(topic = "cache-invalidate")
public void invalidateCache(String key) {
    redisTemplate.delete(key);
}
```

### 2.3 主从延迟导致的不一致

**问题**：
```
T1: 更新主库，删除缓存
T2: 读请求，缓存未命中，读从库，读到旧数据
T3: 将旧数据写入缓存
```

**解决方案**：
- 强制读主库（写操作后的短时间内）
- 使用 Canal 监听 binlog，异步更新缓存

## 三、缓存穿透

### 3.1 什么是缓存穿透

缓存穿透是指查询一个不存在的数据，由于缓存没有命中，每次请求都会查询数据库，可能导致数据库压力过大。

**场景**：
- 恶意攻击：请求大量不存在的 key
- 业务逻辑缺陷：查询不存在的数据

### 3.2 解决方案

#### 方案一：缓存空值

```java
public User getUserById(Long id) {
    String key = "user:" + id;
    User user = redisTemplate.get(key);

    if (user != null) {
        if (user == NULL_USER) {
            return null; // 缓存的是空值
        }
        return user;
    }

    // 查询数据库
    user = userDao.findById(id);
    if (user == null) {
        // 缓存空值，设置较短的过期时间
        redisTemplate.set(key, NULL_USER, 5, TimeUnit.MINUTES);
    } else {
        redisTemplate.set(key, user);
    }
    return user;
}
```

**注意**：
- 空值过期时间不宜过长（5-10 分钟）
- 防止恶意攻击大量 key 占用内存

#### 方案二：布隆过滤器

**原理**：
布隆过滤器是一种空间效率很高的概率型数据结构，用于判断一个元素是否在集合中。

**特性**：
- 不存在的一定不存在
- 存在的可能不存在（有一定误判率）

**实现**：
```java
// 初始化布隆过滤器
BloomFilter<Long> filter = BloomFilter.create(
    Funnels.longFunnel(),
    1000000,  // 预期元素数量
    0.01      // 误判率
);

// 加载所有存在的 ID 到布隆过滤器
List<Long> allIds = userDao.findAllIds();
for (Long id : allIds) {
    filter.put(id);
}

// 查询前先判断
public User getUserById(Long id) {
    if (!filter.mightContain(id)) {
        return null; // 肯定不存在
    }

    // 查询缓存和数据库
    // ...
}
```

**布隆过滤器问题**：
- 不支持删除操作
- 误判率随时间增加

**改进**：使用布隆过滤器 + Redis bitmap 实现支持删除的布隆过滤器（Cuckoo Filter）

#### 方案三：参数校验

在接口层进行参数校验，过滤非法请求。

## 四、缓存击穿

### 4.1 什么是缓存击穿

缓存击穿是指某个热点 key 在缓存过期的瞬间，大量请求同时访问，导致所有请求都打到数据库。

**场景**：
- 热点商品秒杀
- 热点新闻

### 4.2 解决方案

#### 方案一：互斥锁

```java
public User getUserById(Long id) {
    String key = "user:" + id;
    User user = redisTemplate.get(key);

    if (user != null) {
        return user;
    }

    String lockKey = "lock:" + key;
    try {
        // 获取分布式锁
        boolean locked = redisTemplate.setNX(lockKey, "1", 10, TimeUnit.SECONDS);
        if (locked) {
            // 双重检查，防止其他线程已经重建缓存
            user = redisTemplate.get(key);
            if (user != null) {
                return user;
            }

            // 查询数据库
            user = userDao.findById(id);
            redisTemplate.set(key, user, 30, TimeUnit.MINUTES);
            return user;
        } else {
            // 等待并重试
            Thread.sleep(100);
            return getUserById(id);
        }
    } finally {
        // 释放锁
        redisTemplate.delete(lockKey);
    }
}
```

#### 方案二：逻辑过期

**原理**：缓存中不设置真正的过期时间，而是存储一个逻辑过期时间，后台异步更新。

```java
public class CacheValue<T> {
    private T data;
    private long expireTime; // 逻辑过期时间
}

public User getUserById(Long id) {
    String key = "user:" + id;
    CacheValue<User> cacheValue = redisTemplate.get(key);

    if (cacheValue != null) {
        if (cacheValue.getExpireTime() > System.currentTimeMillis()) {
            return cacheValue.getData(); // 未过期
        }

        // 已过期，返回旧数据，并异步更新
        executorService.submit(() -> rebuildCache(id));
        return cacheValue.getData();
    }

    // 首次加载
    return loadAndCache(id);
}
```

**优点**：
- 不会产生缓存击穿
- 用户体验好

**缺点**：
- 数据有一定延迟
- 需要额外的异步更新机制

## 五、缓存雪崩

### 5.1 什么是缓存雪崩

缓存雪崩是指大量缓存同时失效或缓存服务器宕机，导致所有请求都打到数据库。

**场景**：
- 大量 key 设置了相同的过期时间
- 缓存服务器故障

### 5.2 解决方案

#### 方案一：随机过期时间

```java
public void setCache(String key, Object value) {
    // 基础过期时间 30 分钟
    int baseExpire = 30 * 60;
    // 随机偏移 0-5 分钟
    int randomOffset = new Random().nextInt(5 * 60);
    int expire = baseExpire + randomOffset;

    redisTemplate.set(key, value, expire, TimeUnit.SECONDS);
}
```

#### 方案二：缓存预热

系统启动时或定时任务，提前加载热点数据到缓存。

```java
@Scheduled(cron = "0 0 */1 * * ?") // 每小时执行一次
public void preHeatCache() {
    List<Long> hotIds = getHotUserIds();
    for (Long id : hotIds) {
        User user = userDao.findById(id);
        redisTemplate.set("user:" + id, user, 1, TimeUnit.HOURS);
    }
}
```

#### 方案三：高可用缓存架构

使用 Redis 哨兵或 Redis Cluster 保证缓存服务高可用。

```
            客户端
               |
        +------+------+
        |             |
     主 Redis       从 Redis
        |             |
        +------+------+
               |
           Sentinel（监控和自动故障转移）
```

#### 方案四：限流降级

当缓存雪崩发生时，对请求进行限流或降级处理。

```java
@RateLimiter(value = 1000) // 限制每秒 1000 次请求
public User getUserById(Long id) {
    // ...
}

@CircuitBreaker(failureRate = 0.5) // 失败率超过 50% 时熔断
public User getUserById(Long id) {
    // ...
}
```

## 六、热点数据问题

### 6.1 什么是热点数据

热点数据是指被大量并发访问的数据。

### 6.2 解决方案

#### 方案一：本地缓存

```java
// 使用 Caffeine 作为本地缓存
LoadingCache<Long, User> localCache = Caffeine.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(1, TimeUnit.MINUTES)
    .build(id -> loadFromRedis(id));

public User getUserById(Long id) {
    // 先查本地缓存
    return localCache.get(id);
}

private User loadFromRedis(Long id) {
    // 从 Redis 加载
    return redisTemplate.get("user:" + id);
}
```

#### 方案二：多级缓存

```
请求 → 浏览器缓存 → CDN 缓存 → Nginx 缓存 → 本地缓存 → Redis 缓存 → 数据库
```

#### 方案三：热点 key 分片

将热点 key 分散到多个缓存服务器。

```java
// key 分片算法
public String getShardKey(String key, int shardCount) {
    int hash = key.hashCode();
    int shard = Math.abs(hash) % shardCount;
    return key + ":" + shard;
}

// 写入时写入多个分片
public void setHotKey(String key, Object value) {
    for (int i = 0; i < 3; i++) {
        redisTemplate.set(key + ":" + i, value, 30, TimeUnit.MINUTES);
    }
}

// 读取时随机读取一个分片
public Object getHotKey(String key) {
    int shard = new Random().nextInt(3);
    return redisTemplate.get(key + ":" + shard);
}
```

## 七、缓存最佳实践

### 7.1 过期时间设置

| 数据类型 | 建议过期时间 | 说明 |
|---------|------------|------|
| 热点数据 | 5-30 分钟 | 避免雪崩 |
| 配置数据 | 1-24 小时 | 变化频率低 |
| 临时数据 | 1-5 分钟 | 有效期短 |
| 会话数据 | 30 分钟-2 小时 | 根据业务需求 |

### 7.2 缓存粒度

- **粗粒度**：缓存整个对象，减少 key 数量
- **细粒度**：缓存对象的字段，减少数据量

**选择建议**：
- 部分字段更新频繁 → 细粒度
- 整体更新 → 粗粒度
- 数据量大 → 细粒度

### 7.3 缓存更新策略

| 场景 | 策略 |
|------|------|
| 写多读少 | Write-Through |
| 读多写少 | Cache-Aside |
| 写性能要求高 | Write-Behind |
| 数据一致性要求高 | 先更新数据库，再删除缓存 |

### 7.4 监控指标

- **命中率**：cache_hits / (cache_hits + cache_misses)
- **响应时间**：缓存读写耗时
- **内存使用率**：避免内存溢出
- **淘汰数**：了解缓存容量是否充足
- **大 key 数量**：识别性能瓶颈
