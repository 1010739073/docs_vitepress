# 8. 分布式设计模式

## 一、设计模式概述

在分布式系统中，设计模式是解决常见问题的通用解决方案。

**分布式系统的核心挑战**：
- 网络延迟和不可靠性
- 数据一致性
- 故障处理
- 可扩展性

## 二、常见分布式设计模式

### 2.1 代理模式（Proxy Pattern）

**意图**：为其他对象提供一个代理以控制对这个对象的访问。

**适用场景**：
- 远程服务调用
- 权限控制
- 缓存代理

**实现示例**：

```java
// 服务接口
public interface UserService {
    User getUserById(Long id);
}

// 真实服务
public class UserServiceImpl implements UserService {
    @Override
    public User getUserById(Long id) {
        // 实际查询数据库
        return userDao.findById(id);
    }
}

// 代理类
public class UserServiceProxy implements UserService {
    private UserService realService;
    private Cache cache;

    public UserServiceProxy(UserService realService, Cache cache) {
        this.realService = realService;
        this.cache = cache;
    }

    @Override
    public User getUserById(Long id) {
        String key = "user:" + id;
        User user = cache.get(key);
        if (user != null) {
            return user;
        }
        user = realService.getUserById(id);
        cache.set(key, user);
        return user;
    }
}
```

### 2.2 门面模式（Facade Pattern）

**意图**：为子系统中的一组接口提供一个统一的高层接口，使子系统更容易使用。

**适用场景**：
- 简化复杂系统的接口
- 服务聚合

**实现示例**：

```java
public class OrderFacade {
    private UserService userService;
    private InventoryService inventoryService;
    private PaymentService paymentService;

    public void createOrder(Long userId, Long productId, Integer quantity) {
        // 1. 检查用户
        User user = userService.getUserById(userId);
        
        // 2. 检查库存
        inventoryService.checkStock(productId, quantity);
        
        // 3. 扣减库存
        inventoryService.deductStock(productId, quantity);
        
        // 4. 处理支付
        paymentService.processPayment(user.getId(), calculatePrice(productId, quantity));
        
        // 5. 创建订单
        orderRepository.save(new Order(userId, productId, quantity));
    }
}
```

### 2.3 观察者模式（Observer Pattern）

**意图**：定义对象间的一对多依赖关系，当一个对象状态改变时，所有依赖者都会收到通知。

**适用场景**：
- 事件通知
- 消息发布订阅

**实现示例**：

```java
// 主题接口
public interface Subject {
    void registerObserver(Observer observer);
    void removeObserver(Observer observer);
    void notifyObservers(Object data);
}

// 观察者接口
public interface Observer {
    void update(Object data);
}

// 具体主题
public class OrderSubject implements Subject {
    private List<Observer> observers = new ArrayList<>();

    @Override
    public void registerObserver(Observer observer) {
        observers.add(observer);
    }

    @Override
    public void removeObserver(Observer observer) {
        observers.remove(observer);
    }

    @Override
    public void notifyObservers(Object data) {
        for (Observer observer : observers) {
            observer.update(data);
        }
    }

    public void orderCreated(Order order) {
        notifyObservers(order);
    }
}

// 具体观察者
public class InventoryObserver implements Observer {
    @Override
    public void update(Object data) {
        Order order = (Order) data;
        inventoryService.deductStock(order.getProductId(), order.getQuantity());
    }
}

public class NotificationObserver implements Observer {
    @Override
    public void update(Object data) {
        Order order = (Order) data;
        notificationService.sendEmail(order.getUserId(), "订单已创建");
    }
}
```

### 2.4 策略模式（Strategy Pattern）

**意图**：定义一系列算法，把它们封装起来，并使它们可以相互替换。

**适用场景**：
- 多种算法需要切换
- 算法需要动态选择

**实现示例**：

```java
// 策略接口
public interface LoadBalanceStrategy {
    Server select(List<Server> servers);
}

// 具体策略
public class RandomStrategy implements LoadBalanceStrategy {
    @Override
    public Server select(List<Server> servers) {
        return servers.get(new Random().nextInt(servers.size()));
    }
}

public class RoundRobinStrategy implements LoadBalanceStrategy {
    private AtomicInteger counter = new AtomicInteger(0);

    @Override
    public Server select(List<Server> servers) {
        int index = counter.getAndIncrement() % servers.size();
        return servers.get(index);
    }
}

public class LeastConnectionStrategy implements LoadBalanceStrategy {
    @Override
    public Server select(List<Server> servers) {
        return servers.stream()
            .min(Comparator.comparingInt(Server::getConnectionCount))
            .orElse(null);
    }
}

// 上下文
public class LoadBalancer {
    private LoadBalanceStrategy strategy;

    public LoadBalancer(LoadBalanceStrategy strategy) {
        this.strategy = strategy;
    }

    public Server selectServer(List<Server> servers) {
        return strategy.select(servers);
    }

    public void setStrategy(LoadBalanceStrategy strategy) {
        this.strategy = strategy;
    }
}
```

### 2.5 模板方法模式（Template Method Pattern）

**意图**：定义一个算法骨架，将某些步骤延迟到子类中实现。

**适用场景**：
- 多个算法有共同的步骤
- 需要扩展算法的某些步骤

**实现示例**：

```java
// 抽象类
public abstract class DataProcessor {
    // 模板方法
    public final void process() {
        validate();
        parse();
        transform();
        store();
    }

    protected abstract void validate();
    protected abstract void parse();
    protected abstract void transform();
    protected abstract void store();
}

// 具体实现
public class CsvProcessor extends DataProcessor {
    @Override
    protected void validate() { /* CSV 验证 */ }

    @Override
    protected void parse() { /* CSV 解析 */ }

    @Override
    protected void transform() { /* 数据转换 */ }

    @Override
    protected void store() { /* 存储到数据库 */ }
}

public class JsonProcessor extends DataProcessor {
    @Override
    protected void validate() { /* JSON 验证 */ }

    @Override
    protected void parse() { /* JSON 解析 */ }

    @Override
    protected void transform() { /* 数据转换 */ }

    @Override
    protected void store() { /* 存储到数据库 */ }
}
```

## 三、分布式特有模式

### 3.1 Saga 模式

**意图**：将长事务拆分为多个本地事务，通过补偿操作保证最终一致性。

**适用场景**：
- 跨多个服务的事务
- 长事务场景

**实现示例**：

```java
public class OrderSaga {
    private List<SagaStep> steps = new ArrayList<>();

    public void addStep(SagaStep step) {
        steps.add(step);
    }

    public void execute() {
        List<SagaStep> completedSteps = new ArrayList<>();
        try {
            for (SagaStep step : steps) {
                step.execute();
                completedSteps.add(step);
            }
        } catch (Exception e) {
            // 回滚已完成的步骤
            Collections.reverse(completedSteps);
            for (SagaStep step : completedSteps) {
                step.compensate();
            }
            throw e;
        }
    }
}

public interface SagaStep {
    void execute();
    void compensate();
}

public class InventoryStep implements SagaStep {
    @Override
    public void execute() { /* 扣减库存 */ }

    @Override
    public void compensate() { /* 恢复库存 */ }
}

public class PaymentStep implements SagaStep {
    @Override
    public void execute() { /* 扣款 */ }

    @Override
    public void compensate() { /* 退款 */ }
}
```

### 3.2 事件溯源模式（Event Sourcing）

**意图**：将状态变化记录为一系列事件，通过重放事件来恢复状态。

**适用场景**：
- 需要审计日志
- 需要状态回溯
- 高并发写入场景

**实现示例**：

```java
public class OrderAggregate {
    private OrderState state;
    private List<DomainEvent> events = new ArrayList<>();

    public void apply(DomainEvent event) {
        // 根据事件更新状态
        switch (event.getType()) {
            case ORDER_CREATED:
                state = OrderState.CREATED;
                break;
            case ORDER_PAID:
                state = OrderState.PAID;
                break;
            case ORDER_SHIPPED:
                state = OrderState.SHIPPED;
                break;
        }
        events.add(event);
    }

    public List<DomainEvent> getUncommittedEvents() {
        return events;
    }

    public void markEventsAsCommitted() {
        events.clear();
    }
}

public class EventStore {
    public void append(String aggregateId, List<DomainEvent> events) {
        // 写入事件存储
    }

    public List<DomainEvent> load(String aggregateId) {
        // 加载所有事件
        return new ArrayList<>();
    }
}

// 重建状态
public OrderAggregate reconstruct(String aggregateId) {
    OrderAggregate aggregate = new OrderAggregate();
    List<DomainEvent> events = eventStore.load(aggregateId);
    for (DomainEvent event : events) {
        aggregate.apply(event);
    }
    return aggregate;
}
```

### 3.3 CQRS 模式（Command Query Responsibility Segregation）

**意图**：将命令（写操作）和查询（读操作）分离到不同的模型。

**适用场景**：
- 读写分离
- 复杂查询场景

**实现示例**：

```java
// 命令模型
public class OrderCommandHandler {
    private OrderRepository orderRepository;
    private EventStore eventStore;

    public void handle(CreateOrderCommand command) {
        Order order = new Order(command.getUserId(), command.getProductId(), command.getQuantity());
        orderRepository.save(order);
        eventStore.append(order.getId(), order.getUncommittedEvents());
    }
}

// 查询模型
public class OrderQueryHandler {
    private OrderReadModelRepository readModelRepository;

    public OrderSummaryDto getOrderSummary(Long userId) {
        return readModelRepository.findByUserId(userId);
    }

    public List<OrderListDto> listOrders(Long userId) {
        return readModelRepository.listByUserId(userId);
    }
}

// 事件处理器，更新读模型
public class OrderEventHandler {
    private OrderReadModelRepository readModelRepository;

    @EventHandler
    public void handle(OrderCreatedEvent event) {
        OrderReadModel model = new OrderReadModel();
        model.setOrderId(event.getOrderId());
        model.setUserId(event.getUserId());
        model.setStatus("CREATED");
        readModelRepository.save(model);
    }
}
```

### 3.4 限流模式（Rate Limiting Pattern）

**意图**：限制对资源的访问速率，防止系统过载。

**适用场景**：
- API 限流
- 防止恶意攻击
- 资源保护

**实现示例**：

```java
public class RateLimiter {
    private final int maxRequests;
    private final long windowMs;
    private final Map<String, Window> windows = new ConcurrentHashMap<>();

    public RateLimiter(int maxRequests, long windowMs) {
        this.maxRequests = maxRequests;
        this.windowMs = windowMs;
    }

    public boolean tryAcquire(String key) {
        long now = System.currentTimeMillis();
        Window window = windows.computeIfAbsent(key, k -> new Window(now));

        synchronized (window) {
            if (now - window.getStartTime() > windowMs) {
                // 新窗口
                window.setStartTime(now);
                window.setCount(1);
                return true;
            }
            if (window.getCount() < maxRequests) {
                window.incrementCount();
                return true;
            }
            return false;
        }
    }

    private static class Window {
        private long startTime;
        private int count;

        public Window(long startTime) {
            this.startTime = startTime;
            this.count = 0;
        }
        // getters and setters
    }
}

// 使用
RateLimiter limiter = new RateLimiter(100, 1000); // 每秒最多 100 次请求
if (!limiter.tryAcquire(userId)) {
    throw new RateLimitException("请求过于频繁");
}
```

### 3.5 断路器模式（Circuit Breaker Pattern）

**意图**：当服务调用失败率达到阈值时，自动切断调用，防止级联故障。

**适用场景**：
- 服务容错
- 防止雪崩

**实现示例**：

```java
public class CircuitBreaker {
    private enum State { CLOSED, OPEN, HALF_OPEN }

    private State state = State.CLOSED;
    private int failureCount = 0;
    private int failureThreshold = 5;
    private long resetTimeout = 30000; // 30 秒
    private long lastFailureTime = 0;

    public synchronized boolean allowRequest() {
        switch (state) {
            case CLOSED:
                return true;
            case OPEN:
                if (System.currentTimeMillis() - lastFailureTime > resetTimeout) {
                    state = State.HALF_OPEN;
                    return true;
                }
                return false;
            case HALF_OPEN:
                return true;
            default:
                return true;
        }
    }

    public synchronized void recordSuccess() {
        failureCount = 0;
        state = State.CLOSED;
    }

    public synchronized void recordFailure() {
        failureCount++;
        lastFailureTime = System.currentTimeMillis();
        if (failureCount >= failureThreshold) {
            state = State.OPEN;
        }
    }
}

// 使用
CircuitBreaker breaker = new CircuitBreaker();
if (!breaker.allowRequest()) {
    throw new ServiceUnavailableException("服务不可用");
}
try {
    result = remoteService.call();
    breaker.recordSuccess();
} catch (Exception e) {
    breaker.recordFailure();
    throw e;
}
```

### 3.6 重试模式（Retry Pattern）

**意图**：当操作失败时，自动重试多次。

**适用场景**：
- 网络不稳定
- 暂时性故障

**实现示例**：

```java
public class RetryTemplate {
    private int maxRetries = 3;
    private long delayMs = 1000;
    private List<Class<? extends Exception>> retryableExceptions;

    public <T> T execute(Callable<T> task) throws Exception {
        int attempts = 0;
        Exception lastException = null;

        while (attempts < maxRetries) {
            try {
                return task.call();
            } catch (Exception e) {
                lastException = e;
                if (!isRetryable(e)) {
                    throw e;
                }
                attempts++;
                if (attempts < maxRetries) {
                    Thread.sleep(delayMs * attempts);
                }
            }
        }
        throw lastException;
    }

    private boolean isRetryable(Exception e) {
        return retryableExceptions.stream()
            .anyMatch(clazz -> clazz.isInstance(e));
    }
}

// 使用
RetryTemplate retryTemplate = new RetryTemplate();
retryTemplate.setRetryableExceptions(Arrays.asList(
    NetworkException.class,
    TimeoutException.class
));
User user = retryTemplate.execute(() -> userService.getUserById(id));
```

## 四、模式组合应用

### 4.1 电商订单场景

```
客户端 → API网关 → 订单服务 → 库存服务
                              ↓
                        消息队列 → 物流服务
                                 ↓
                             通知服务
```

**使用的模式**：
- **门面模式**：API 网关聚合多个服务
- **观察者模式**：订单创建后通知多个下游服务
- **Saga 模式**：跨服务事务
- **重试模式**：网络请求重试
- **断路器模式**：服务容错

### 4.2 数据处理管道

```
数据采集 → 数据验证 → 数据转换 → 数据存储 → 数据索引
```

**使用的模式**：
- **模板方法模式**：统一处理流程
- **策略模式**：不同数据源使用不同解析策略
- **观察者模式**：处理完成后通知下游

## 五、选择模式的原则

1. **问题匹配**：根据问题类型选择合适的模式
2. **简单优先**：优先选择简单的解决方案
3. **可扩展性**：考虑未来的扩展需求
4. **性能考虑**：模式带来的性能开销
5. **团队熟悉度**：选择团队熟悉的模式
