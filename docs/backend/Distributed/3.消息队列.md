# 消息队列

## 一、消息队列基础

### 1.1 什么是消息队列

消息队列（Message Queue，MQ）是一种应用间的通信方式，消息发送后可以立即返回，由消息队列保证消息的可靠传递。

### 1.2 为什么使用消息队列

**核心价值**：

| 价值 | 说明 | 示例 |
|------|------|------|
| **解耦** | 降低系统间的耦合度 | 订单服务 → MQ → 仓储服务/积分服务 |
| **异步** | 提高系统响应速度 | 用户下单后立即返回，后台异步处理 |
| **削峰** | 应对流量高峰 | 秒杀场景，MQ 缓冲请求 |
| **可靠** | 保证消息不丢失 | 消息持久化、重试机制 |

### 1.3 基本概念

```
┌─────────┐         ┌─────────┐         ┌─────────┐
│ Producer│ ──────► │   MQ    │ ◄────── │ Consumer│
│ 生产者   │  发送   │ 消息队列 │  消费   │ 消费者   │
└─────────┘         └─────────┘         └─────────┘
                            │
                       ┌────┴────┐
                       │ Topic   │ 主题
                       │ Queue   │ 队列
                       │ Message │ 消息
                       └─────────┘
```

**核心概念**：
- **Producer（生产者）**：发送消息的应用
- **Consumer（消费者）**：接收消息的应用
- **Broker（代理）**：消息队列服务端
- **Topic（主题）**：消息的分类
- **Queue（队列）**：消息的存储容器
- **Message（消息）**：传递的数据单元

### 2.1 RabbitMQ

**特点**：
- 基于 AMQP 协议
- 支持多种消息模式（直连、主题、扇出、头部）
- 延迟低，适合实时性要求高的场景
- 运维相对简单

**适用场景**：
- 传统企业应用
- 实时性要求高的场景
- 需要复杂路由的场景

### 2.2 RocketMQ

**特点**：
- 阿里开源，专为电商场景设计
- 支持事务消息、顺序消息、延迟消息
- 吞吐量高，支持海量消息堆积
- 支持消息回溯和死信队列

**适用场景**：
- 电商、金融等业务场景
- 需要事务消息的场景
- 需要消息堆积能力的场景

### 2.3 Kafka

**特点**：
- 高吞吐、低延迟
- 分布式、可扩展
- 基于日志追加，适合流式处理
- 生态系统完善（Kafka Streams、Flink）

**适用场景**：
- 日志收集
- 流式数据处理
- 用户行为追踪

### 2.4 Pulsar

**特点**：
- 云原生架构（计算存储分离）
- 支持 IO 隔离
- 多协议兼容
- 无缝扩缩容

**适用场景**：
- 云原生环境
- 需要多租户隔离
- 需要灵活扩缩容

## 三、消息模型

### 3.1 点对点模型（P2P）

```
Producer 1     Producer 2
    │              │
    ▼              ▼
┌─────────────────────────┐
│         Queue           │
└───────────┬─────────────┘
            │
    ┌───────┴───────┐
    │               │
Consumer 1      Consumer 2
（竞争消费）
```

**特点**：
- 每条消息只被一个消费者消费
- 消息被消费后从队列中删除
- 适合任务分发场景

### 3.2 发布订阅模型（Pub/Sub）

```
Producer 1     Producer 2
    │              │
    ▼              ▼
┌─────────────────────────┐
│        Topic            │
└───────────┬─────────────┘
            │
    ┌───────┼───────┐
    │       │       │
Consumer 1 Consumer 2 Consumer 3
（各自消费）
```

**特点**：
- 每条消息被所有订阅的消费者消费
- 消息持久化，新消费者可以消费历史消息
- 适合广播场景

### 3.3 集群消费 vs 广播消费

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 集群消费 | 消息分发给集群中的一个消费者 | 任务处理、订单处理 |
| 广播消费 | 消息分发给集群中的所有消费者 | 配置更新、通知 |

## 四、消息可靠性

### 4.1 消息发送可靠性

**三种发送方式**：

#### 1. 同步发送

```java
// 同步发送，等待返回结果
SendResult result = producer.send(msg);
if (result.getSendStatus() == SendStatus.SEND_OK) {
    // 发送成功
}
```

**优点**：可靠性高
**缺点**：性能低

#### 2. 异步发送

```java
// 异步发送，不阻塞
producer.send(msg, new SendCallback() {
    @Override
    public void onSuccess(SendResult result) {
        // 发送成功
    }

    @Override
    public void onException(Throwable e) {
        // 发送失败，处理重试
    }
});
```

**优点**：性能高
**缺点**：可靠性需要额外保证

#### 3. 单向发送（Oneway）

```java
// 不关心结果
producer.sendOneway(msg);
```

**适用场景**：对可靠性要求不高的场景（如日志收集）

### 4.2 消息存储可靠性

**持久化机制**：

| MQ | 持久化机制 |
|----|-----------|
| RabbitMQ | 消息写入磁盘后才返回 ACK |
| RocketMQ | 同步刷盘/异步刷盘 |
| Kafka | 页缓存 + 周期性刷盘 + 副本同步 |

**RocketMQ 刷盘策略**：

```xml
<!-- 同步刷盘：消息写入磁盘后才返回成功 -->
<flushDiskType>SYNC_FLUSH</flushDiskType>

<!-- 异步刷盘：消息写入内存后返回成功，后台异步刷盘 -->
<flushDiskType>ASYNC_FLUSH</flushDiskType>
```

### 4.3 消息消费可靠性

**ACK 机制**：

#### 自动 ACK

```java
// 消息被消费后自动 ACK，不管是否成功
@RabbitListener(queues = "queue")
public void handle(String message) {
    // 如果抛出异常，消息已丢失
}
```

#### 手动 ACK

```java
@RabbitListener(queues = "queue")
public void handle(String message, Channel channel, @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) {
    try {
        // 处理消息
        process(message);

        // 手动 ACK
        channel.basicAck(deliveryTag, false);
    } catch (Exception e) {
        // 拒绝消息，重新入队
        channel.basicNack(deliveryTag, false, true);
    }
}
```

### 4.4 消息丢失场景与解决方案

| 场景 | 原因 | 解决方案 |
|------|------|----------|
| 发送阶段丢失 | 网络问题、MQ 故障 | 同步发送 + 本地消息表 |
| 存储阶段丢失 | 刷盘失败、磁盘故障 | 多副本 + 刷盘策略 |
| 消费阶段丢失 | 消费者异常 | 手动 ACK + 重试机制 |

## 五、消息重复消费

### 5.1 产生原因

1. 生产者发送重复（网络抖动）
2. MQ 重复投递（ACK 超时）
3. 消费者重复消费（ACK 失败）

### 5.2 解决方案：幂等性

#### 方案一：数据库唯一索引

```java
public void processOrder(Order order) {
    try {
        orderDao.insert(order); // order_id 有唯一索引
    } catch (DuplicateKeyException e) {
        // 重复消息，忽略
        log.warn("Duplicate order: {}", order.getId());
    }
}
```

#### 方案二：分布式锁

```java
public void processOrder(Order order) {
    String lockKey = "order:" + order.getId();
    RLock lock = redisson.getLock(lockKey);

    if (lock.tryLock(10, TimeUnit.SECONDS)) {
        try {
            // 检查是否已处理
            if (isProcessed(order.getId())) {
                return;
            }
            // 处理订单
            doProcess(order);
            // 标记已处理
            markProcessed(order.getId());
        } finally {
            lock.unlock();
        }
    }
}
```

#### 方案三：状态机

```java
public enum OrderStatus {
    CREATED,      // 已创建
    PROCESSING,   // 处理中
    COMPLETED,    // 已完成
    CANCELED      // 已取消
}

public void processOrder(Order order) {
    // 更新状态，利用数据库行锁
    int updated = orderDao.updateStatus(
        order.getId(),
        OrderStatus.CREATED,
        OrderStatus.PROCESSING
    );

    if (updated == 0) {
        // 状态不是 CREATED，说明已处理或正在处理
        return;
    }

    try {
        // 处理订单
        doProcess(order);
        // 更新状态为完成
        orderDao.updateStatus(order.getId(), OrderStatus.COMPLETED);
    } catch (Exception e) {
        // 更新状态为创建，允许重试
        orderDao.updateStatus(order.getId(), OrderStatus.CREATED);
        throw e;
    }
}
```

## 六、消息顺序性

### 6.1 顺序性需求

**场景**：
- 订单状态变更：创建 → 支付 → 发货 → 完成
- 股票交易：买/卖指令必须按顺序执行

### 6.2 如何保证顺序

#### 方案一：单队列单消费者

```
所有消息发送到同一个队列，由同一个消费者消费
```

**优点**：实现简单
**缺点**：性能差，无法水平扩展

#### 方案二：分区有序（Kafka）

```java
// 根据 key 分区，相同 key 的消息发送到同一个分区
ProducerRecord<String, String> record =
    new ProducerRecord<>("topic", orderId, message);
producer.send(record);
```

**原理**：
- 相同 partition 的消息有序
- 不同 partition 无序

#### 方案三：RocketMQ 顺序消息

```java
// 发送顺序消息
Message msg = new Message("OrderTopic", "TagA", orderId.toString(), body);
// 使用同一个 MessageQueueSelector 保证同一订单的消息发送到同一个队列
SendResult result = producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        Long orderId = (Long) arg;
        int index = (int) (orderId % mqs.size());
        return mqs.get(index);
    }
}, orderId);

// 消费顺序消息
consumer.registerMessageListener(new MessageListenerOrderly() {
    @Override
    public ConsumeOrderlyStatus consumeMessage(
        List<MessageExt> msgs,
        ConsumeOrderlyContext context) {
        // 单线程消费，保证顺序
        for (MessageExt msg : msgs) {
            process(msg);
        }
        return ConsumeOrderlyStatus.SUCCESS;
    }
});
```

### 7.1 积压原因分析

1. 生产者发送速度 > 消费者消费速度
2. 消费者处理逻辑变慢
3. 消费者数量不足
4. 消费者故障

### 7.2 解决方案

#### 方案一：增加消费者

```java
// 增加 Consumer 数量，注意不能超过分区数
// 如果是 RabbitMQ，可以增加 Queue 数量
```

#### 方案二：临时扩容

```
1. 将积压的消息转发到一个新的 Topic（分区数扩大 10 倍）
2. 部署 10 倍的消费者消费新 Topic
3. 消费完成后恢复原配置
```

#### 方案三：丢弃不重要的消息

```java
// 对于日志、统计类消息，可以直接丢弃部分
// 或者只消费最新的消息，跳过积压的部分
```

#### 方案四：异步批量处理

```java

    // 批量处理，提高吞吐量

```

## 八、延迟消息

### 8.1 应用场景

- 订单超时取消（30 分钟未支付）
- 定时提醒
- 延迟重试

### 8.2 实现方案

#### 方案一：RabbitMQ 延迟插件

```java
// 使用 rabbitmq_delayed_message_exchange 插件
@Bean
public CustomExchange delayedExchange() {
    Map<String, Object> args = new HashMap<>();
    args.put("x-delayed-type", "direct");
    return new CustomExchange("delayed.exchange",
        "x-delayed-message", true, false, args);
}

// 发送延迟消息
rabbitTemplate.convertAndSend("delayed.exchange", "delayed.key",
    message, msg -> {
        msg.getMessageProperties().setDelay(30000); // 延迟 30 秒
        return msg;
    });
```

#### 方案二：RocketMQ 延迟消息

```java
// RocketMQ 支持 18 个延迟级别
// 1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h
Message msg = new Message("DelayTopic", "TagA", body);
msg.setDelayTimeLevel(4); // 30 秒
producer.send(msg);
```

#### 方案三：Redis 过期通知

```java
// 设置 key 过期
redisTemplate.opsForValue().set("delay:order:" + orderId,
    orderId, 30, TimeUnit.MINUTES);

// 监听过期事件
@RedisKeyExpirationEventMessageListener
public void onKeyExpired(String expiredKey) {
    String orderId = expiredKey.split(":")[2];
    cancelOrder(orderId);
}
```

**注意**：Redis 过期通知不是实时，有一定延迟

#### 方案四：时间轮（HashedWheelTimer）

```java
HashedWheelTimer timer = new HashedWheelTimer();
timer.newTimeout(timeout -> {
    // 延迟执行的任务
    cancelOrder(orderId);
}, 30, TimeUnit.MINUTES);
```

**适用场景**：单机应用

## 十、最佳实践

### 10.1 消息大小控制

| MQ | 建议消息大小 | 最大消息大小 |
|----|------------|------------|
| RabbitMQ | < 10KB | 128MB |
| RocketMQ | < 256KB | 4MB |
| Kafka | < 1MB | 10MB |

**建议**：
- 消息体尽量小
- 大文件使用文件服务 + 消息传递 URL

### 10.2 消息数量控制

- 批量发送提高吞吐量
- 批量消费提高处理效率

```java
// 批量发送
List<Message> messages = ...;
producer.send(messages);

// 批量消费
@RocketMQMessageListener(
    topic = "topic",
    consumeMode = ConsumeMode.CONCURRENTLY,
    consumeThreadMax = 20,
    consumeThreadNumber = 10
)
public class BatchConsumer implements RocketMQListener<List<MessageExt>> {
    @Override
    public void onMessage(List<MessageExt> messages) {
        batchProcess(messages);
    }
}
```

### 10.3 消费者组配置

- 同一个消费者组内的消费者会竞争消费
- 不同消费者组各自消费

```
消费者组 A (3 个消费者) ──┐
                          ├── Topic
消费者组 B (2 个消费者) ──┘
```

### 10.4 死信队列

处理无法正常消费的消息。

```java
// RabbitMQ 死信队列配置
@Bean
public Queue deadLetterQueue() {
    return QueueBuilder.durable("dead.letter.queue")
        .withArgument("x-dead-letter-exchange", "dlx.exchange")
        .withArgument("x-dead-letter-routing-key", "dlx.key")
        .build();
}

// 消费失败后消息进入死信队列
@RabbitListener(queues = "queue")
public void handle(String message) {
    throw new RuntimeException("处理失败"); // 消息进入死信队列
}
```

### 10.5 监控指标

- **消息积压量**：队列中的消息数量
- **生产速率**：每秒发送的消息数
- **消费速率**：每秒消费的消息数
- **消费延迟**：消息从发送到消费的延迟
- **失败率**：消费失败的消息比例
