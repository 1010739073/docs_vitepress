# PostgreSQL 主从高可用面试题

## 一、基础概念

### Q1：什么是 WAL？它在主从复制中起什么作用？

**答**：WAL（Write-Ahead Log，预写日志）是 PostgreSQL 的"操作日记"。在真正修改数据文件之前，会先把操作记录到 WAL 中。

在主从复制中，WAL 是"搬运的内容"——主库产生 WAL 记录，通过 WAL Sender 进程推送给从库，从库的 WAL Receiver 接收后，在从库上重放（replay）这些 WAL 记录，从而实现数据同步。

> 简单说：WAL 就是主从之间传递的"变化清单"，从库照着清单做一遍，就和主库一致了。

### Q2：流复制和逻辑复制有什么区别？

| 对比维度 | 流复制 | 逻辑复制 |
|---------|-------|---------|
| 复制粒度 | 整个实例（所有数据库） | 按表级别 |
| 原理 | 物理复制（复制 WAL 文件） | 逻辑解码（复制 SQL 变更） |
| 跨版本 | 不支持（主从必须同大版本） | **支持** |
| 选择性 | 全有或全无 | 可以只复制指定表 |
| 从库写入 | 从库只读 | 从库可以写入 |
| 使用场景 | 高可用、读写分离 | 版本升级、数据迁移、多活 |

**加分回答**：流复制基于 WAL 位点，从库直接 apply 物理页变更；逻辑复制使用 pgoutput 插件，将 WAL 解码为逻辑变更（INSERT/UPDATE/DELETE），通过发布-订阅模式传输。

### Q3：为什么需要复制槽？不用的风险是什么？

**答**：复制槽告诉主库"这个从库还需要 WAL 位点 X 之后的数据，别删"。

**不用复制槽的风险**：
- 从库宕机一段时间后，主库的 WAL 已经被清理
- 从库重启发现 WAL 断档，无法继续追赶
- 只能 `pg_basebackup` 全量重建，数据量大时耗时很长

**用复制槽的风险**：
- 从库长时间不连接，WAL 在主库上无限堆积，直到撑爆磁盘
- 必须监控复制槽的 `retained_wal` 大小

### Q4：同步复制和异步复制的区别？什么时候用哪个？

| 维度 | 异步复制 | 同步复制 |
|-----|---------|---------|
| 主库写入流程 | 写入 WAL 就返回，不等从库 | 必须等从库确认收到 WAL 才返回 |
| 性能 | 高 | 主库写入延迟增加 |
| 数据安全 | 主库宕机可能丢数据 | 主库宕机不丢数据 |
| 可用性 | 从库全挂不影响主库写入 | 从库全挂主库无法写入 |

**选择建议**：
- 金融、交易等强一致性场景 → 同步复制
- 日志、报表、数据分析 → 异步复制
- 一般业务 → `FIRST 1 (standby1, standby2)` 折中方案

### Q5：LSN 是什么？为什么主从复制需要它？

**答**：LSN（Log Sequence Number）是 WAL 日志中的字节偏移量，可以理解为数据的"版本号"。主库和从库通过比较 LSN 来判断同步进度。

```sql
主库: pg_current_wal_lsn()        → 0/5000000
从库: pg_last_wal_replay_lsn()    → 0/4FFFFFF

差值 = 主库 LSN - 从库 LSN = 复制延迟量
```

**核心作用**：
- 计算复制延迟：`pg_wal_lsn_diff(主库lsn, 从库lsn)`
- 从库恢复时确定起点：从哪个 LSN 开始 apply WAL
- 故障切换时判断哪个节点的数据更新

## 二、实战操作

### Q6：主库宕机了，怎么手动把从库提升为主库？

```bash
# 1. 确认主库确实宕机，无法短时间内恢复
# 2. 在从库上执行
sudo -u postgres pg_ctl promote -D /var/lib/postgresql/16/main
# 或
sudo -u postgres psql -c "SELECT pg_promote();"

# 3. 验证
psql -c "SELECT pg_is_in_recovery();"
# 返回 false → 升级成功，现在是主库

# 4. 修改应用连接指向新主库
# 5. 原主库恢复后，作为从库重新加入
```

**加分回答**：生产环境不应该手动 promote，应该用 Patroni 的 `patronictl failover` 或 repmgr 的 `standby promote`。手动 promote 可能遗漏步骤（比如忘了改连接配置），导致问题。

### Q7：如何快速搭建一个流复制环境？

**关键步骤**：
```bash
# 主库
# 1. 创建复制用户
# 2. wal_level = replica, 配置 pg_hba.conf
# 3. 创建复制槽

# 从库
# 1. pg_basebackup -h 主库 -U replicator -D $PGDATA -Fp -Xs -P -R
# 2. 启动 PostgreSQL
# 3. 验证：SELECT pg_is_in_recovery();
```

**-R 参数做了什么**？
- 自动创建 `standby.signal` 文件（告诉 PostgreSQL 我是从库）
- 自动写入 `primary_conninfo` 到 `postgresql.auto.conf`

### Q8：备库为什么不能写入？如果强行写入会发生什么？

**答**：备库处于 recovery 模式，所有写操作都会被拒绝：

```sql
-- 在从库上执行
INSERT INTO users (name) VALUES ('test');
-- ERROR:  cannot execute INSERT in a read-only transaction
```

**原理**：从库的 `pg_is_in_recovery()` 返回 true，此时 PostgreSQL 将所有用户事务设为只读。这是为了防止数据分叉——如果两个节点都能写入，数据就会不一致。

**例外**：逻辑复制的订阅端可以写入，因为逻辑复制是按表粒度同步，不会产生冲突。

### Q9：pg_rewind 是做什么用的？什么场景下使用？

**答**：`pg_rewind` 用于让一个"曾经是主库"的节点，快速变回从库。

**场景**：主库宕机 → 从库 promote 成新主库 → 原主库恢复 → 需要作为从库加入新主库。

**问题**：原主库和新主库的时间线（timeline）已经分叉了，直接 `pg_basebackup` 是全量重建，耗时很长。

**pg_rewind 的魔法**：只同步两个节点之间不同的数据块，而不是整个数据库。如果数据量 500GB 但只差了 100MB，pg_rewind 只需要几分钟。

```bash
# 前提：wal_log_hints = on 或 开启了 checksum
sudo -u postgres pg_rewind \
    --target-pgdata=/var/lib/postgresql/16/main \
    --source-server='host=new_primary dbname=postgres user=replicator'
```

## 三、架构设计

### Q10：一个高可用 PostgreSQL 集群应该包含哪些组件？

```
应用层
  ↓
VIP / HAProxy / Pgpool-II     ← 连接路由，自动切换
  ↓
Patroni 节点1  Patroni 节点2  Patroni 节点3  ← 自动故障转移
  + PG(主库)    + PG(从库)    + PG(从库)
  ↓
etcd 集群 (3节点)             ← 分布式共识，选主
  ↓
WAL 归档 + 备份               ← 灾备
```

**必备组件**：
1. **PostgreSQL 流复制**：至少 1 主 + 1 从
2. **Patroni**：自动故障检测和切换
3. **etcd/Consul/ZooKeeper**：分布式共识，选主
4. **HAProxy/Pgpool-II**：连接路由，自动发现主库
5. **监控告警**：Prometheus + Grafana
6. **备份**：WAL 归档 + pg_basebackup

### Q11：脑裂是怎么发生的？怎么预防？

**脑裂**：两个节点都认为自己是主库，同时接受写入。

**典型场景**：
- 网络分区：主库和从库之间网络断了，但各自都能访问
- etcd 集群出问题：Patroni 无法正确判断谁持有 leader 锁
- 手动误操作：在不同节点上各自执行了 promote

**预防措施**：
1. **etcd 集群奇数节点**（3 或 5 个），确保网络分区时能达成多数派
2. **设置 Patroni ttl 合理**（如 30 秒），主库宕机后快速过期
3. **使用 watchdog 设备**：主库宕机时操作系统自动重启
4. **监控告警**：检测到多个节点 claim 为 Leader 时立刻告警
5. **HAProxy 健康检查**：只把流量发给 Patroni 确认为 Leader 的节点

**如果已经脑裂了**：
```bash
# 1. 立刻停止所有 PostgreSQL（两个节点都停）
# 2. 比较 pg_controldata 的 Latest checkpoint location
# 3. 选择 LSN 更大的节点作为主库
# 4. 另一个节点用 pg_rewind 或全量重建，变成从库
```

### Q12：读写分离怎么实现？有哪些坑？

**实现方式**：

| 方式 | 原理 | 优点 | 缺点 |
|-----|------|-----|------|
| **应用层** | 代码里配置两个数据源 | 简单 | 业务侵入 |
| **HAProxy** | Patroni REST API 检测主库 | 成熟稳定 | 多一层代理 |
| **Pgpool-II** | 自己解析 SQL 判断读写 | 功能全 | 配置复杂 |
| **JDBC 驱动** | PostgreSQL JDBC 支持读写分离 | 无需额外组件 | 仅 Java |

**常见的坑**：

1. **写后立刻读，从库还来不及同步**
   ```
   用户注册 → 写入主库 → 立刻跳转查询页 → 查询落在从库 → 查不到刚注册的用户
   ```
   **解决**：关键操作强制读主库，或使用 `synchronous_commit = remote_apply`

2. **序列（sequence）在主从之间不一致**
   ```
   从库上执行 SELECT nextval() → 返回的值和主库不同
   ```
   **解决**：从库只读，不要在从库上调用序列函数

3. **长事务在从库上阻塞 WAL replay**
   ```
   从库上一个复杂查询跑了 10 分钟，期间 WAL replay 被阻塞
   ```
   **解决**：设置 `max_standby_streaming_delay`，超时自动取消阻塞查询

### Q13：如何设计一个可以跨机房容灾的 PostgreSQL 架构？

```
  机房 A（主）                    机房 B（备）
  ┌──────────────┐             ┌──────────────┐
  │ Patroni 主库  │             │ Patroni 从库  │
  │ (同步复制)    │──── WAL ──→│ (延迟 1h)    │
  │              │             │              │
  │ Patroni 从库  │             │ Patroni 从库  │
  │ (异步复制)    │             │ (异步复制)    │
  └──────────────┘             └──────────────┘
         │                            │
         └──── etcd 跨机房集群 ────────┘
              (节点分布在两个机房)
```

**关键设计点**：
1. **同机房同步复制**：保证主库宕机不丢数据
2. **跨机房异步复制**：避免网络延迟拖慢主库
3. **延迟从库**：备机房设置 `recovery_min_apply_delay = 1h`，防误删
4. **etcd 跨机房**：奇数节点分布，防止单机房故障导致无法选主
5. **自动切换策略**：主库机房故障时，考虑数据丢失量，谨慎决定是否自动切换

## 四、性能与优化

### Q14：主从复制延迟过大怎么办？

**排查思路**：
1. 确定延迟在哪里：网络传输？还是 replay 慢？
2. 网络延迟 → 增加带宽、压缩传输
3. replay 慢 → 检查从库是否在跑大查询、磁盘 IO 是否瓶颈
4. 没有复制槽 → 创建复制槽，防止 WAL 被清理后重来

**优化手段**：
```sql
-- 从库上加大 WAL replay 并行度
-- postgresql.conf
max_worker_processes = 16
max_parallel_workers = 8

-- 从库上监控阻塞 replay 的查询
SELECT pid, now() - query_start AS duration, query
FROM pg_stat_activity
WHERE wait_event_type = 'Recovery' AND state = 'active';
```

### Q15：同步复制会拖慢主库吗？怎么量化影响？

**答**：会。同步复制下，主库的每个事务提交都必须等待从库的确认。

**量化方法**：
```sql
-- 异步复制下的写入延迟
\set aid random(1, 100000)
BEGIN;
UPDATE pgbench_accounts SET abalance = abalance + 1 WHERE aid = :aid;
COMMIT;
-- latency average = 2.5 ms

-- 切换到同步复制后
-- latency average = 8.2 ms  ← 增加了网络往返时间
```

**影响因素**：
- 主从之间的网络延迟（RTT）+ 从库磁盘写入延迟
- 同机房：增加 2-5ms
- 跨机房：增加 20-100ms
- 跨地域：增加 100ms+

### Q16：为什么推荐 Patroni 而不是自己写脚本做故障切换？

**答**：自己写脚本的问题：

1. **脑裂**：脚本无法判断"主库真的挂了吗还是网络问题"
2. **选主**：多个从库，选哪个 promote？
3. **时间线分叉**：promote 后原主库恢复，数据怎么对齐？
4. **连接切换**：应用怎么自动连到新主库？
5. **状态持久化**：脚本挂了，集群状态丢失

Patroni 解决这些问题的方式：
- 脑裂 → etcd 分布式锁，只有一个节点能拿到 Leader
- 选主 → 自动选择延迟最小的从库
- 时间线 → 集成 pg_rewind
- 连接 → 提供 REST API，HAProxy 自动发现主库
- 状态 → etcd 持久化存储

## 五、进阶问题

### Q17：简述 PostgreSQL 复制冲突及其处理机制

**答**：复制冲突发生在从库上——当从库正在重放 WAL 时，遇到与用户查询冲突的情况。

**常见冲突**：
- 从库上有个查询正在读某行数据，WAL 说要把这行删了
- 从库上有个查询持有锁，WAL 说需要这个锁

**处理机制**：
```ini
# 等待一定时间，如果查询还没结束，强制取消它
max_standby_streaming_delay = 30s
# 等待归档重放时
max_standby_archive_delay = 30s

# 设置为 -1，一直等待（可能导致延迟无限增长）
max_standby_streaming_delay = -1
```

**最佳实践**：设置合理的超时时间（如 30s），同时限制从库上的长查询。

### Q18：如果在主库上执行了 DROP TABLE，从库上能恢复吗？

**答**：这在技术上是可以的，因为 DROP TABLE 不会立即删除数据文件，而是等到事务提交且没有活跃查询时才会清理。而且从库可以使用延迟重放。

**恢复方法**：
1. 如果从库设置了 `recovery_min_apply_delay`，从库上数据还没被删
2. 暂停从库的 WAL replay：`SELECT pg_wal_replay_pause();`
3. 从从库导出数据：`pg_dump -h standby -t 被删的表`
4. 恢复到主库

**根本方案**：
- 设置 `recovery_min_apply_delay = 1h`，给误删留出恢复窗口
- 使用 PITR（时间点恢复）
- 定期备份验证

### Q19：如何实现 PostgreSQL 的零停机大版本升级？

**答**：使用逻辑复制实现零停机升级。

```
步骤：
1. 在新服务器上安装 PostgreSQL 16（目标版本）
2. 在旧服务器（PG13）上创建逻辑复制发布
   CREATE PUBLICATION upgrade_pub FOR ALL TABLES;
3. 在新服务器上创建逻辑复制订阅
   CREATE SUBSCRIPTION upgrade_sub
   CONNECTION 'host=old_server dbname=mydb'
   PUBLICATION upgrade_pub;
4. 等待数据完全同步（监控 pg_stat_subscription）
5. 停应用 → 切换连接到新库 → 启动应用
6. 删除订阅和发布
```

**注意**：
- 序列（sequence）需要单独同步
- 大表（DDL）变更在升级期间要暂停
- 建议先在测试环境演练

### Q20：Patroni 集群中，为什么 etcd 需要奇数节点？

**答**：这是分布式共识算法的核心要求。

**Raft 协议要求**：任何决策（如选主）需要**多数派**同意才能生效。

- 3 节点：允许 1 个节点故障（2 > 3/2）
- 5 节点：允许 2 个节点故障（3 > 5/2）
- 2 节点：任何 1 个节点故障，无法形成多数派（1 < 2/2），集群不可用

**为什么不能是 2 节点**？因为网络分区时，两个节点各自独立，都能形成多数派（1 > 1/2），导致两个 Leader（脑裂）。

**为什么不能是 4 节点**？4 节点允许 1 个故障（3 > 2），但和 3 节点一样，而 3 节点更便宜。所以 3 或 5 是最优选择。
