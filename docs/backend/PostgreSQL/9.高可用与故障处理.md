# PostgreSQL 高可用与故障处理

## 一、先搞明白：高可用要解决什么问题？

流复制已经让我们有了主从架构。但还缺几样东西：

| 问题 | 流复制做到了吗？ |
|------|:---:|
| 主库宕机，从库能自动接管？ | 不能，需要手动 promote |
| 应用的连接能自动切到新主库？ | 不能，应用不知道主库换了 |
| 主库恢复了，能自动变回从库？ | 不能 |
| 怎么防止脑裂（两个库都以为自己是主库）？ | 没有机制 |

**高可用 = 流复制 + 自动故障检测 + 自动切换 + 连接路由**。

## 二、高可用方案全景

### 2.1 方案对比

| 方案 | 复杂度 | 故障切换 | 脑裂防护 | 适合场景 |
|-----|-------|---------|---------|---------|
| **手动切换** | 低 | 人工 | 人工 | 开发/测试环境 |
| **repmgr** | 中 | 半自动 | 一般 | 小团队 |
| **Patroni + etcd** | 高 | 全自动 | 强 | **生产环境首选** |
| **Patroni + Consul** | 高 | 全自动 | 强 | 已有 Consul 的团队 |
| **Patroni + ZooKeeper** | 高 | 全自动 | 强 | 已有 ZK 的团队 |
| **云厂商托管（RDS/云数据库）** | 低 | 自动 | 自动 | 不想运维的团队 |
| **Stolon** | 高 | 全自动 | 强 | K8s 原生 |
| **Pgpool-II** | 中 | 半自动 | 一般 | 自带连接池和读写分离 |

### 2.2 为什么推荐 Patroni？

```
                    ┌─────────────┐
                    │  etcd 集群   │  ← 分布式共识，选主、存配置
                    │ (最少3节点)  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
        ┌─────────┐  ┌─────────┐  ┌─────────┐
        │ Patroni │  │ Patroni │  │ Patroni │
        │   +     │  │   +     │  │   +     │
        │   PG    │  │   PG    │  │   PG    │
        │ (Leader)│  │(Replica)│  │(Replica)│
        └────┬────┘  └────┬────┘  └────┬────┘
             │            │            │
             └────────────┼────────────┘
                          ↓
                   ┌─────────────┐
                   │   HAProxy   │  ← 根据 etcd 信息，自动把流量发给 Leader
                   │  (读写分离)  │
                   └──────┬──────┘
                          ↓
                    应用连接 haproxy
                    (端口 5000→写, 5001→读)
```

Patroni 的工作方式：
1. 每个 Patroni 节点向 etcd 争抢一个"leader 锁"
2. 拿到锁的节点就是主库，其他是从库
3. Patroni 持续监控 PostgreSQL 进程
4. 主库挂了 → leader 锁过期 → 另一个节点抢到锁 → 自动 promote 成为新主库

## 三、Patroni + etcd + HAProxy 实战部署

### 3.1 架构规划

| 节点 | IP | 角色 |
|-----|-----|------|
| node1 | 192.168.1.11 | etcd + Patroni + PostgreSQL |
| node2 | 192.168.1.12 | etcd + Patroni + PostgreSQL |
| node3 | 192.168.1.13 | etcd + Patroni + PostgreSQL |
| haproxy | 192.168.1.20 | HAProxy（也可部署在应用服务器上） |

### 3.2 安装 etcd 集群

```bash
# 在所有 3 个节点上安装
sudo apt-get install -y etcd

# 每个节点的 /etc/default/etcd 配置不同
# === node1 ===
ETCD_NAME="node1"
ETCD_INITIAL_CLUSTER="node1=http://192.168.1.11:2380,node2=http://192.168.1.12:2380,node3=http://192.168.1.13:2380"
ETCD_INITIAL_CLUSTER_TOKEN="pg-cluster-token"
ETCD_INITIAL_CLUSTER_STATE="new"
ETCD_DATA_DIR="/var/lib/etcd"

# 启动
sudo systemctl start etcd
sudo systemctl enable etcd

# 验证集群
etcdctl member list
etcdctl cluster-health
```

### 3.3 安装 Patroni

```bash
# 安装
sudo apt-get install -y python3-pip
sudo pip3 install patroni[etcd] psycopg2-binary

# 配置 /etc/patroni/postgres.yml（每个节点不同部分：name, node_ip）
```

```yaml
scope: pg_cluster
name: node1  # 每个节点不同

restapi:
  listen: 0.0.0.0:8008
  connect_address: 192.168.1.11:8008

etcd:
  hosts: 192.168.1.11:2379,192.168.1.12:2379,192.168.1.13:2379

bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576  # 1MB，落后超过这个值不切换
    postgresql:
      use_pg_rewind: true
      parameters:
        wal_level: replica
        hot_standby: "on"
        wal_keep_size: 1024
        max_wal_senders: 10
        max_replication_slots: 10
        wal_log_hints: "on"

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 192.168.1.11:5432
  data_dir: /var/lib/postgresql/16/main
  bin_dir: /usr/lib/postgresql/16/bin
  pgpass: /tmp/pgpass

  authentication:
    replication:
      username: replicator
      password: Replicator@123
    superuser:
      username: postgres
      password: Postgres@123

watchdog:
  mode: automatic
  device: /dev/watchdog
```

```bash

sudo systemctl enable patroni

# | node3  | 192.168.1.13 | Replica | running| 0.0       |

```

### 3.4 配置 HAProxy 读写分离

```bash
# 安装在 haproxy 节点
sudo apt-get install -y haproxy

# 配置 /etc/haproxy/haproxy.cfg
```

```ini
global
    maxconn 4096
    daemon

defaults
    mode tcp
    timeout connect 5s
    timeout client 30m
    timeout server 30m

# 主库检测：读头请求
listen pg_master_check
    bind *:8008
    option httpchk GET /master
    http-check expect status 200
    default-server inter 3s fall 3 rise 2
    server pg1 192.168.1.11:8008 check
    server pg2 192.168.1.12:8008 check
    server pg3 192.168.1.13:8008 check

# 写端口 → 只路由到主库
frontend pg_write
    bind *:5000
    default_backend pg_write_backend

backend pg_write_backend
    option httpchk GET /master
    http-check expect status 200
    default-server inter 3s fall 3 rise 2
    server pg1 192.168.1.11:5432 check port 8008
    server pg2 192.168.1.12:5432 check port 8008
    server pg3 192.168.1.13:5432 check port 8008

# 读端口 → 路由到从库（轮询）
frontend pg_read
    bind *:5001
    default_backend pg_read_backend

backend pg_read_backend
    option httpchk GET /replica
    http-check expect status 200
    default-server inter 3s fall 3 rise 2
    server pg1 192.168.1.11:5432 check port 8008
    server pg2 192.168.1.12:5432 check port 8008
    server pg3 192.168.1.13:5432 check port 8008
```

```bash
# 应用连接方式：
# 写操作 → haproxy:5000
# 读操作 → haproxy:5001
```

### 3.5 Patroni 常用运维命令

```bash
# 查看集群状态
patronictl -c /etc/patroni/postgres.yml list

# 手动切换（正常切换，不丢数据）
patronictl -c /etc/patroni/postgres.yml switchover
# 交互式选择目标节点

# 强制切换（原主库已宕机时使用）
patronictl -c /etc/patroni/postgres.yml failover

# 重新初始化从库
patronictl -c /etc/patroni/postgres.yml reinit pg_cluster node3

# 重启集群节点（滚动重启）
patronictl -c /etc/patroni/postgres.yml restart pg_cluster

# 查看历史切换记录
patronictl -c /etc/patroni/postgres.yml history
```

## 四、业务场景故障处理

### 场景 1：主库突然宕机

**现象**：
- 应用写操作报错：`connection refused` 或 `no route to host`
- 监控告警：主库不可达

**自动处理（Patroni 环境下）**：
```
时间线：
T+0s   主库宕机
T+10s  etcd 检测到 leader 锁过期（ttl=30，loop_wait=10）
T+20s  另一个 Patroni 节点抢到 leader 锁
T+21s  新主库 promote 完成
T+25s  HAProxy 健康检查发现新主库，流量切换
T+25s  应用恢复可用（在此期间写入会报错）

总影响时间：约 20~30 秒
```

**人工排查步骤（如果没有自动切换）**：
```bash
# 1. 确认主库状态
ssh primary_host
sudo systemctl status postgresql
tail -100 /var/log/postgresql/postgresql-16-main.log

# 2. 在从库上查看延迟
psql -c "SELECT pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn(), now() - pg_last_xact_replay_timestamp();"

# 3. 如果主库短时间内无法恢复 → 手动 promote
ssh standby_host
sudo -u postgres pg_ctl promote -D /var/lib/postgresql/16/main

# 4. 原主库恢复后 → 作为从库重新加入
# 用 pg_rewind 对齐时间线，或 pg_basebackup 全量重建
```

### 场景 2：从库延迟越来越大

**现象**：
```sql
SELECT pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)) AS lag
FROM pg_stat_replication;
-- 结果：2 GB，还在持续增长！
```

**排查步骤**：
```bash
# 1. 检查从库有没有正在跑的大查询
psql -h standby_host -c "
SELECT pid, now() - query_start AS duration, state, query
FROM pg_stat_activity
WHERE state = 'active'
ORDER BY query_start;
"

# 2. 检查从库磁盘 IO
iostat -x 1 5

# 3. 检查网络带宽
iftop -i eth0

# 4. 检查复制槽
psql -h primary_host -c "
SELECT slot_name, active,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS lag
FROM pg_replication_slots;
"
```

**处理办法**：

| 原因 | 解法 |
|------|------|
| 从库有大查询，CPU 打满 | 杀掉慢查询，限制从库查询并发 |
| 从库磁盘 IO 瓶颈 | 升级 SSD、增加从库 |
| 网络带宽不足 | 压缩 WAL 传输、升级带宽 |
| 大量 DDL 操作 | 错峰执行 DDL，放在低峰期 |
| 没有复制槽导致重来 | 创建复制槽 |

```sql
-- 杀掉阻塞 WAL replay 的查询
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE pid IN (
    SELECT pid FROM pg_locks WHERE locktype = 'relation'
    AND relation IN (
        SELECT relation FROM pg_locks
        WHERE mode = 'AccessExclusiveLock'
    )
);

-- 或直接杀掉所有用户查询
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE pid <> pg_backend_pid()
AND backend_type = 'client backend';
```

### 场景 3：脑裂了（两个库都以为自己是主库）

**现象**：
```sql
-- 两台机器上执行，都是 false
SELECT pg_is_in_recovery();
-- node1: false ← 以为自己是主库！
-- node2: false ← 也以为自己是主库！
```

**这是最危险的场景**——两个节点都接受写入，数据会分叉，再合并几乎不可能。

**原因**：网络分区，主库和从库之间网络断了，但各自都能访问。

**紧急处理**：
```bash
# 1. 立刻关闭两个节点
sudo systemctl stop patroni  # 两台都要停！

# 2. 确定哪个是最新的
# 在 node1 上:
sudo -u postgres pg_controldata /var/lib/postgresql/16/main | grep 'Latest checkpoint location'
# Latest checkpoint location: 0/9000000

# 在 node2 上:
sudo -u postgres pg_controldata /var/lib/postgresql/16/main | grep 'Latest checkpoint location'
# Latest checkpoint location: 0/A000000  ← LSN 更大，选这台做主库

# 3. 以数据最新的节点为主库
# 启动 node2 的 Patroni（它会自动成为主库）
sudo systemctl start patroni

# 4. 原 node1 需要重建
sudo systemctl stop patroni
sudo rm -rf /var/lib/postgresql/16/main/*
sudo -u postgres pg_basebackup -h node2 -U replicator -D /var/lib/postgresql/16/main -Fp -Xs -P -R
sudo systemctl start patroni
```

**预防脑裂**：
- 使用 etcd 奇数节点（3 或 5）
- 设置 `ttl` 足够短，让 Leader 快速过期
- 使用 watchdog 设备：主库宕机时操作系统自动重启

### 场景 4：误删数据，需要从从库恢复

```bash
# 假设用户删掉了用户表，已经过去 1 小时
# 从库还没 replay 完，数据还在！

# 1. 在从库上暂停 WAL replay
psql -h standby_host -c "SELECT pg_wal_replay_pause();"

# 2. 从从库导出丢失的数据
pg_dump -h standby_host -U postgres -d mydb -t users --data-only > users_recovery.sql

# 3. 恢复 WAL replay
psql -h standby_host -c "SELECT pg_wal_replay_resume();"

# 4. 把数据导回主库
psql -h primary_host -U postgres -d mydb < users_recovery.sql

# 5. 如果有冲突，手动清理后重新导入
```

**预防措施**：
```ini
# 在从库 postgresql.conf 中设置 WAL replay 延迟
recovery_min_apply_delay = '1h'  # 从库延迟 1 小时重放

# 这样你永远有 1 小时前的一份"历史快照"
# 误删数据后，从库上数据还在！
```

### 场景 5：WAL 堆积撑爆磁盘

**现象**：`pg_wal/` 目录越来越大，磁盘就要满了。

```sql
-- 检查是什么占用了 WAL
SELECT 
    slot_name,
    active,
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS retained_wal
FROM pg_replication_slots
ORDER BY pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) DESC;
```

**处理**：
```sql
-- 方法 1：删除不活跃的复制槽
SELECT pg_drop_replication_slot('dead_standby_slot');

-- 方法 2：如果还有从库需要用这个槽，手动切换 WAL
SELECT pg_switch_wal();

-- 方法 3：如果实在等不起，强制清理（不推荐）
-- 先备份 pg_wal/，然后手动删除旧的 WAL 文件
-- pg_archivecleanup /var/lib/postgresql/data/pg_wal 0000000100000000000000FF
```

**长期方案**：
```ini
# 增加 WAL 的大小限制
max_wal_size = 8GB
# 设置 WAL 超时，强制切换
archive_timeout = 60
```

## 五、监控与告警

### 5.1 关键监控指标

```sql
-- 综合健康检查脚本
SELECT 
    NOT pg_is_in_recovery() AS is_primary,
    CASE WHEN NOT pg_is_in_recovery() THEN
        (SELECT count(*) FROM pg_stat_replication WHERE state = 'streaming')
    ELSE NULL END AS standby_count,
    CASE WHEN NOT pg_is_in_recovery() THEN
        (SELECT max(pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn))
         FROM pg_stat_replication WHERE state = 'streaming')
    ELSE pg_wal_lsn_diff(pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn())
    END AS replication_lag_bytes,
    (SELECT count(*) FROM pg_replication_slots WHERE NOT active) AS inactive_slots,
    (SELECT count(*) FROM pg_stat_activity WHERE state = 'idle in transaction'
     AND now() - state_change > interval '5 minutes') AS long_idle_txns;
```

### 5.2 Prometheus + Grafana 告警规则

```yaml
# prometheus 告警规则示例
groups:
  - name: postgresql_ha
    rules:
      - alert: ReplicationLag
        expr: pg_replication_lag_bytes > 104857600  # 100MB
        for: 5m
        annotations:
          summary: "从库延迟超过 100MB"

      - alert: NoStandbyServer
        expr: pg_replication_standby_count < 1
        for: 1m
        annotations:
          summary: "没有活跃的从库！"

      - alert: InactiveReplicationSlot
        expr: pg_replication_slots_inactive > 0
        for: 5m
        annotations:
          summary: "存在不活跃的复制槽，WAL 可能堆积"
```

## 六、总结

| 概念 | 关键点 |
|------|-------|
| Patroni | 自动选主、自动故障转移 |
| etcd | 分布式共识，存集群状态 |
| HAProxy | 读写分离路由，自动发现主库 |
| pg_rewind | 主库降级从库的数据同步工具 |
| 复制槽 | 必须监控，不活跃的槽会撑爆磁盘 |
| recovery_min_apply_delay | 延迟重放 WAL，防误删 |
| 脑裂 | 最危险场景，发现后立刻停掉所有节点 |
