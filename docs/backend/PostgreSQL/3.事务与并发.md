# PostgreSQL 事务与并发控制

## 一、事务基础

### 1.1 ACID 特性

| 特性 | 说明 | PostgreSQL 实现 |
|-----|------|----------------|
| **原子性（Atomicity）** | 事务要么全部成功，要么全部失败 | 回滚段 |
| **一致性（Consistency）** | 事务执行前后数据库保持一致性 | 约束、触发器 |
| **隔离性（Isolation）** | 并发事务相互隔离 | MVCC |
| **持久性（Durability）** | 提交的事务永久保存 | WAL（预写日志） |

### 1.2 事务控制

```sql
-- 开始事务
BEGIN;
-- 或
START TRANSACTION;

-- 提交事务
COMMIT;

-- 回滚事务
ROLLBACK;

-- 设置保存点
SAVEPOINT my_savepoint;

-- 回滚到保存点
ROLLBACK TO SAVEPOINT my_savepoint;

-- 释放保存点
RELEASE SAVEPOINT my_savepoint;

-- 设置事务特性
BEGIN ISOLATION LEVEL READ COMMITTED READ WRITE;
BEGIN ISOLATION LEVEL SERIALIZABLE READ ONLY DEFERRABLE;
```

### 1.3 完整事务示例

```sql
-- 带保存点的事务
BEGIN;

-- 插入用户
INSERT INTO users (username, email) VALUES ('张三', 'zhangsan@example.com');

SAVEPOINT after_user_insert;

-- 插入订单
INSERT INTO orders (user_id, total_amount) VALUES (currval('users_id_seq'), 100.00);

-- 如果订单插入失败，回滚到保存点
-- ROLLBACK TO SAVEPOINT after_user_insert;

-- 提交事务
COMMIT;

-- 带错误处理的事务
DO $$
BEGIN
    -- 插入用户
    INSERT INTO users (username, email) VALUES ('李四', 'lisi@example.com');
    
    -- 插入订单
    INSERT INTO orders (user_id, total_amount) VALUES (currval('users_id_seq'), 200.00);
    
    -- 更新统计
    UPDATE user_stats SET order_count = order_count + 1 WHERE user_id = currval('users_id_seq');
    
EXCEPTION WHEN OTHERS THEN
    -- 发生错误时回滚
    RAISE NOTICE 'Error: %', SQLERRM;
    ROLLBACK;
    RETURN;
END;
$$;
```

## 二、隔离级别

### 2.1 隔离级别对比

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|---------|------|-----------|------|---------|
| **READ UNCOMMITTED** | 不允许 | 可能 | 可能 | 同 READ COMMITTED |
| **READ COMMITTED** | 不允许 | 可能 | 可能 | MVCC（默认） |
| **REPEATABLE READ** | 不允许 | 不允许 | 可能（PG 中不允许） | MVCC |
| **SERIALIZABLE** | 不允许 | 不允许 | 不允许 | 可串行化快照 |

### 2.2 设置隔离级别

```sql
-- 设置会话隔离级别
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 设置事务隔离级别
BEGIN ISOLATION LEVEL READ COMMITTED;
BEGIN ISOLATION LEVEL REPEATABLE READ;
BEGIN ISOLATION LEVEL SERIALIZABLE;

-- 查看当前隔离级别
SHOW transaction_isolation;
```

### 2.3 各隔离级别详解

#### READ COMMITTED（读已提交）

```sql
-- 默认隔离级别
-- 事务中的每个查询看到的数据是查询开始时的快照

-- 会话 A
BEGIN;
SELECT * FROM users WHERE id = 1;  -- 看到 age = 25

-- 会话 B
BEGIN;
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;

-- 会话 A（同一事务中再次查询）
SELECT * FROM users WHERE id = 1;  -- 看到 age = 30（不可重复读）
COMMIT;
```

#### REPEATABLE READ（可重复读）

```sql
-- 事务开始时的快照在整个事务期间保持不变

BEGIN ISOLATION LEVEL REPEATABLE READ;

SELECT * FROM users WHERE id = 1;  -- 看到 age = 25

-- 其他事务更新并提交

SELECT * FROM users WHERE id = 1;  -- 仍然看到 age = 25（可重复读）

-- 尝试更新
UPDATE users SET age = 35 WHERE id = 1;  -- 等待或报错（取决于并发情况）

COMMIT;
```

#### SERIALIZABLE（可串行化）

```sql
-- 最强的隔离级别，保证事务串行执行的效果

BEGIN ISOLATION LEVEL SERIALIZABLE;

SELECT * FROM users WHERE age > 25;

-- 其他事务插入 age = 30 的记录

SELECT * FROM users WHERE age > 25;  -- 看不到新记录（幻读防止）

-- 尝试插入
INSERT INTO users (username, age) VALUES ('新用户', 30);
-- 可能报错：could not serialize access due to read/write dependencies

COMMIT;
```

## 三、MVCC（多版本并发控制）

### 3.1 MVCC 原理

PostgreSQL 使用 MVCC 来实现事务隔离，每个事务看到的数据是事务开始时的快照。

**系统列：**
- `xmin`：插入事务的 ID
- `xmax`：删除事务的 ID（0 表示未删除）
- `cmin`/`cmax`：命令 ID

```sql
-- 查看系统列
SELECT xmin, xmax, cmin, cmax, * FROM users WHERE id = 1;

-- 查看事务 ID
SELECT txid_current();

-- 查看快照信息
SELECT txid_current_snapshot();
```

### 3.2 行版本

```sql
-- 更新操作创建新行版本
UPDATE users SET age = 26 WHERE id = 1;
-- 旧行：xmax = 当前事务ID
-- 新行：xmin = 当前事务ID

-- 删除操作标记行版本
DELETE FROM users WHERE id = 1;
-- 行：xmax = 当前事务ID

-- 查询只看到满足以下条件的行：
-- xmin 已提交且 xmax 为 0 或 xmax 未提交/回滚
```

### 3.3 清理（VACUUM）

```sql
-- 手动清理死元组
VACUUM users;

-- 清理并分析
VACUUM ANALYZE users;

-- 完全清理（会锁表，回收空间）
VACUUM FULL users;

-- 查看清理统计
SELECT 
    schemaname,
    tablename,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze,
    vacuum_count,
    autovacuum_count
FROM pg_stat_user_tables
WHERE tablename = 'users';

-- 查看死元组数量
SELECT 
    n_live_tup,
    n_dead_tup,
    round(n_dead_tup::numeric / nullif(n_live_tup, 0) * 100, 2) AS dead_ratio
FROM pg_stat_user_tables
WHERE tablename = 'users';
```

## 四、锁机制

### 4.1 锁类型

| 锁模式 | 说明 |
|-------|------|
| **ACCESS SHARE** | SELECT |
| **ROW SHARE** | SELECT FOR UPDATE/FOR SHARE |
| **ROW EXCLUSIVE** | INSERT, UPDATE, DELETE |
| **SHARE UPDATE EXCLUSIVE** | VACUUM, ANALYZE, CREATE INDEX CONCURRENTLY |
| **SHARE** | CREATE INDEX |
| **SHARE ROW EXCLUSIVE** | 触发器启用/禁用 |
| **EXCLUSIVE** | REFRESH MATERIALIZED VIEW CONCURRENTLY |
| **ACCESS EXCLUSIVE** | DROP, TRUNCATE, VACUUM FULL, LOCK TABLE |

### 4.2 行级锁

```sql
-- FOR UPDATE（排他锁）
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 其他事务无法更新此行，直到当前事务提交

-- FOR SHARE（共享锁）
SELECT * FROM users WHERE id = 1 FOR SHARE;
-- 其他事务可以读取，但不能更新

-- FOR NO KEY UPDATE（无键更新锁）
SELECT * FROM users WHERE id = 1 FOR NO KEY UPDATE;
-- 允许并发更新非键列

-- FOR KEY SHARE（键共享锁）
SELECT * FROM users WHERE id = 1 FOR KEY SHARE;
-- 允许并发更新非键列，但阻止删除和外键更新

COMMIT;
```

### 4.3 表级锁

```sql
-- 显式表锁
BEGIN;
LOCK TABLE users IN ACCESS EXCLUSIVE MODE;
-- 执行操作
COMMIT;

-- 锁模式
-- ACCESS SHARE: 最弱，不阻塞其他读
-- ROW SHARE: 阻塞排他锁
-- ROW EXCLUSIVE: 阻塞共享锁和排他锁
-- SHARE UPDATE EXCLUSIVE: 阻塞其他 SHARE UPDATE EXCLUSIVE
-- SHARE: 阻塞排他更新
-- SHARE ROW EXCLUSIVE: 阻塞 SHARE 和排他更新
-- EXCLUSIVE: 阻塞所有锁
-- ACCESS EXCLUSIVE: 最强，阻塞所有操作
```

### 4.4 查看锁信息

```sql
-- 查看当前锁
SELECT 
    l.locktype,
    l.relation::regclass,
    l.mode,
    l.granted,
    a.pid,
    a.query,
    a.state
FROM pg_locks l
JOIN pg_stat_activity a ON l.pid = a.pid
WHERE NOT l.granted;

-- 查看锁等待
SELECT 
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    blocked_activity.query AS blocked_statement,
    blocking_activity.query AS blocking_statement
FROM pg_locks blocked_locks
JOIN pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_locks blocking_locks ON blocked_locks.locktype = blocking_locks.locktype
    AND blocked_locks.relation = blocking_locks.relation
    AND blocked_locks.pid != blocking_locks.pid
JOIN pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;

-- 终止进程
SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE pid = 12345;
```

## 五、死锁处理

### 5.1 死锁产生

```sql
-- 会话 A
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 会话 B
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 2;
-- 会话 A
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 等待会话 B
-- 会话 B
UPDATE accounts SET balance = balance + 100 WHERE id = 1;  -- 等待会话 A（死锁）
```

### 5.2 死锁检测与处理

```sql
-- PostgreSQL 自动检测死锁并终止其中一个事务
-- 错误信息：ERROR: deadlock detected

-- 查看死锁日志
-- 在 postgresql.conf 中设置：
-- log_lock_waits = on
-- deadlock_timeout = 1s

-- 预防死锁的方法：
-- 1. 按相同顺序访问资源
-- 2. 保持事务简短
-- 3. 使用较低的隔离级别
-- 4. 使用 NOWAIT 立即返回错误

-- 使用 NOWAIT
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;
-- 如果无法立即获得锁，立即报错而不是等待
```

### 5.3 应用程序死锁处理

```python
# Python 示例：死锁重试机制
import psycopg2
from psycopg2 import OperationalError
import time

def execute_with_retry(conn, operation, max_retries=3):
    retry_count = 0
    while True:
        try:
            with conn.cursor() as cur:
                operation(cur)
            conn.commit()
            break
        except OperationalError as e:
            conn.rollback()
            if 'deadlock detected' in str(e) and retry_count < max_retries:
                retry_count += 1
                time.sleep(0.1 * retry_count)  # 指数退避
            else:
                raise

# 使用示例
def transfer_money(cur):
    cur.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1")
    cur.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 2")

conn = psycopg2.connect("dbname=mydb user=myuser")
execute_with_retry(conn, transfer_money)
```

## 六、并发控制策略

### 6.1 乐观并发控制

```sql
-- 使用行版本号
ALTER TABLE users ADD COLUMN version INTEGER DEFAULT 0;

-- 更新时检查版本
UPDATE users 
SET email = 'new@example.com', version = version + 1
WHERE id = 1 AND version = 1;

-- 如果返回 0 行，说明数据已被修改
```

### 6.2 悲观并发控制

```sql
-- 使用 FOR UPDATE 锁定行
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 执行业务逻辑
UPDATE users SET email = 'new@example.com' WHERE id = 1;
COMMIT;
```

### 6.3 咨询锁

```sql
-- 会话级咨询锁
SELECT pg_advisory_lock(123);        -- 获取锁
SELECT pg_advisory_unlock(123);      -- 释放锁
SELECT pg_advisory_unlock_all();     -- 释放所有会话锁

-- 事务级咨询锁
SELECT pg_advisory_xact_lock(123);   -- 事务结束自动释放

-- 尝试获取锁（不阻塞）
SELECT pg_try_advisory_lock(123);    -- 返回 true/false

-- 使用场景：防止并发执行特定操作
BEGIN;
SELECT pg_advisory_xact_lock(hashtext('process_orders'));
-- 执行需要互斥的操作
COMMIT;
```

## 七、高级事务特性

### 7.1 两阶段提交

```sql
-- 准备事务
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
PREPARE TRANSACTION 'transfer_001';

-- 查看准备的事务
SELECT * FROM pg_prepared_xacts;

-- 提交准备的事务
COMMIT PREPARED 'transfer_001';

-- 回滚准备的事务
ROLLBACK PREPARED 'transfer_001';
```

### 7.2 可延迟约束

```sql
-- 创建可延迟约束
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id) DEFERRABLE INITIALLY DEFERRED
);

-- 在事务中延迟检查
BEGIN;
SET CONSTRAINTS ALL DEFERRED;
-- 插入订单（此时 users 表可能还没有对应记录）
INSERT INTO orders (user_id) VALUES (999);
-- 插入用户
INSERT INTO users (id) VALUES (999);
COMMIT;  -- 约束在提交时检查
```

### 7.3 事务超时

```sql
-- 设置语句超时
SET statement_timeout = '30s';

-- 设置锁等待超时
SET lock_timeout = '10s';

-- 设置空闲事务超时
SET idle_in_transaction_session_timeout = '60s';

-- 在事务中设置
BEGIN;
SET LOCAL statement_timeout = '30s';
-- 执行操作
COMMIT;
```

## 八、最佳实践

### 8.1 事务设计原则

1. **保持事务简短**
   ```sql
   -- 不好
   BEGIN;
   -- 大量数据处理
   PERFORM pg_sleep(10);
   COMMIT;
   
   -- 好
   BEGIN;
   -- 最小必要操作
   COMMIT;
   ```

2. **按相同顺序访问资源**
   ```sql
   -- 所有事务都按相同顺序访问表
   BEGIN;
   UPDATE accounts SET balance -= 100 WHERE id = 1;
   UPDATE accounts SET balance += 100 WHERE id = 2;
   COMMIT;
   ```

3. **使用合适的隔离级别**
   ```sql
   -- 默认使用 READ COMMITTED
   -- 需要可重复读时使用 REPEATABLE READ
   -- 需要防止幻读时使用 SERIALIZABLE
   ```

### 8.2 错误处理最佳实践

```sql
-- 使用 PL/pgSQL 处理错误
CREATE OR REPLACE FUNCTION transfer_funds(
    from_account INT,
    to_account INT,
    amount NUMERIC
) RETURNS VOID AS $$
BEGIN
    -- 检查余额
    IF (SELECT balance FROM accounts WHERE id = from_account) < amount THEN
        RAISE EXCEPTION '余额不足';
    END IF;
    
    -- 执行转账
    UPDATE accounts SET balance = balance - amount WHERE id = from_account;
    UPDATE accounts SET balance = balance + amount WHERE id = to_account;
    
    -- 记录日志
    INSERT INTO transaction_log (from_account, to_account, amount)
    VALUES (from_account, to_account, amount);
    
EXCEPTION
    WHEN insufficient_funds THEN
        RAISE NOTICE '余额不足';
        ROLLBACK;
    WHEN OTHERS THEN
        RAISE NOTICE '错误: %', SQLERRM;
        ROLLBACK;
END;
$$ LANGUAGE plpgsql;
```

### 8.3 监控建议

```sql
-- 监控长时间运行的事务
SELECT 
    pid,
    usename,
    application_name,
    state,
    xact_start,
    now() - xact_start AS xact_duration,
    query_start,
    now() - query_start AS query_duration,
    query
FROM pg_stat_activity
WHERE state != 'idle'
AND xact_start IS NOT NULL
ORDER BY xact_start;

-- 监控准备的事务
SELECT 
    transaction,
    gid,
    prepared,
    owner,
    database
FROM pg_prepared_xacts;

-- 监控锁等待
SELECT 
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    blocked_activity.query AS blocked_statement,
    blocking_activity.query AS blocking_statement,
    blocked_activity.application_name AS blocked_application,
    blocking_activity.application_name AS blocking_application
FROM pg_locks blocked_locks
JOIN pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_locks blocking_locks ON blocked_locks.locktype = blocking_locks.locktype
    AND blocked_locks.relation = blocking_locks.relation
    AND blocked_locks.pid != blocking_locks.pid
JOIN pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;
```
