# PostgreSQL 主从复制

## 一、先理解：为什么要主从复制？

### 1.1 单台数据库的困境

假设你只有一台 PostgreSQL 服务器：

| 问题 | 后果 |
|------|------|
| 服务器宕机 | 整个服务不可用，用户骂人 |
| 读请求太多 | 一台机器扛不住，响应越来越慢 |
| 磁盘损坏 | 数据丢失，老板骂人 |
| 做备份时需要停服 | 备份期间用户用不了 |

**主从复制的解决思路**：多搞几台服务器，一台写（主库），其他跟着同步（从库）。主库挂了从库顶上，查询压力分散到从库。

### 1.2 主从复制的核心价值

```
                          ┌─────────────┐
                   读请求 →  从库 1      │
                  ／       └─────────────┘
     ┌──────────┐
     │  主库    │  ─ 复制数据 ─
     │ (读写)   │
     └──────────┘
                  ＼       ┌─────────────┐
                   读请求 →  从库 2      │
                          └─────────────┘

主库负责：所有写操作（INSERT / UPDATE / DELETE）
从库负责：分担读操作（SELECT），主库挂了还能顶上去
```

- **读写分离**：主库写，从库读，查询性能翻倍
- **高可用**：主库挂了，从库立刻升级为主库
- **备份**：直接在从库上做备份，不影响主库
- **灾备**：从库放在不同机房/城市，机房断电都不怕

## 二、WAL：复制的基础

### 2.1 WAL 是什么？

WAL（Write-Ahead Log，预写日志）是 PostgreSQL 的"操作日记"。

**在改数据之前，先把"我要做什么"写进日记。** 就像你每次花钱前，先在记账本上写一笔，花了再更新余额。万一系统崩溃了，重启后翻翻日记就知道哪些操作没做完。

```
用户执行: UPDATE users SET age = 26 WHERE id = 1;

  ↓ 第 1 步：写 WAL（操作日记）
  WAL: "事务 100: 把 users 表 id=1 的 age 改成 26"

  ↓ 第 2 步：改内存中的数据
  内存中的数据页: age 变成 26

  ↓ 第 3 步：定期刷到磁盘（Checkpoint）
  磁盘上的数据文件: age 变成 26
```

### 2.2 WAL 级别

WAL 记录了数据库的所有变更。具体记录多少信息，由 `wal_level` 参数控制：

| wal_level | 记录内容 | 用途 |
|-----------|---------|------|
| **minimal** | 只记录崩溃恢复所需的信息 | 不复制，纯单机 |
| **replica** | 记录所有变更，支持只读从库 | 流复制（默认推荐） |
| **logical** | 记录详细的逻辑变更 | 逻辑复制、CDC |

```sql
-- 查看当前 WAL 级别
SHOW wal_level;

-- 修改 WAL 级别（需要重启）
-- postgresql.conf:
wal_level = replica
```

### 2.3 WAL 文件长什么样？

WAL 日志存储在 `pg_wal/` 目录下，每个文件 16MB：

```bash
ls -lh /var/lib/postgresql/data/pg_wal/
# -rw------- 1 postgres postgres 16M 000000010000000000000001
# -rw------- 1 postgres postgres 16M 000000010000000000000002
# ...

# 文件名含义：
# 00000001  →  时间线 ID（每次恢复会 +1）
# 00000000  →  高位 LSN
# 00000001  →  低位 LSN
```

### 2.4 LSN：数据版本号

LSN（Log Sequence Number，日志序列号）是一个 64 位数字，标记 WAL 中的位置，就像是**数据的变化版本号**。主从复制的本质，就是**把主库的 WAL 搬过去，从库照着执行一遍**。

```sql
-- 查看当前 WAL 位置
SELECT pg_current_wal_lsn();
-- 结果: 0/3000128

-- 人能读懂的格式
SELECT pg_walfile_name(pg_current_wal_lsn());
-- 结果: 000000010000000000000003

-- 计算两个 LSN 之间的差距（字节数）
SELECT pg_size_pretty(
    pg_wal_lsn_diff('0/3000000', '0/4000000')
);
-- 结果: 16 MB

-- 查看 WAL 的 LSN 基础信息
SELECT pg_current_wal_lsn(), pg_current_wal_insert_lsn(), pg_current_wal_flush_lsn();
```

**LSN 在主从复制中的关键角色**：每个从库都记录着自己的 `replay_lsn`（重放到哪了），主库一比较就知道从库落后了多少。

## 三、流复制：主从复制的核心机制

### 3.1 流复制工作原理

流复制（Streaming Replication）是 PostgreSQL 内置的主从复制机制，不需要任何第三方工具。

```
  主库                          从库
  ┌──────────┐                ┌──────────┐
  │ WAL 写入  │  ── 网络传输 ──→ │ WAL 接收  │
  │ (WAL Sender)│              │ (WAL Receiver)│
  │          │                │    ↓      │
  │          │                │ WAL 重放   │
  │          │                │ (Recovery) │
  └──────────┘                └──────────┘

简化理解：
主库每执行一条写操作 → 生成 WAL → WAL Sender 进程发给从库 → 从库 WAL Receiver 接收 → 从库按 WAL 重放一遍
```

**整个过程的核心是**：主库只要 WAL 位点比从库新，就把新产生的 WAL 记录源源不断地推送给从库。这使得从库数据延迟通常控制在**毫秒级**。

### 3.2 流复制模式

| 模式 | 说明 | 数据安全 | 性能 |
|-----|------|---------|------|
| **异步复制** | 主库不管从库有没有收到，直接返回 | 可能丢数据 | 最快 |
| **同步复制** | 主库等从库确认收到才返回 | 不丢数据 | 较慢 |
| **半同步（FIRST/ANY）** | 等 N 个从库中任意 M 个确认 | 折中 | 折中 |

**怎么选？**
- 报表系统、日志系统 → **异步**，丢点数据没关系
- 交易系统、金融系统 → **同步**，丢一笔钱都不行
- 一般业务 → **FIRST 1 (standby1, standby2)**，至少有一台同步就行

### 3.3 流复制角色

```sql
-- 查看当前节点是主还是从
SELECT pg_is_in_recovery();
-- false = 主库
-- true  = 从库

-- 查看复制状态（主库上执行）
SELECT 
    application_name,     -- 从库名称
    client_addr,          -- 从库 IP
    state,                -- streaming = 正常复制中
    sync_state,           -- async/sync = 异步/同步
    sent_lsn,             -- 主库已发送的 WAL 位置
    write_lsn,            -- 从库已写入磁盘的位置
    flush_lsn,            -- 从库已刷盘的位置
    replay_lsn,           -- 从库已重放的位置
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS lag  -- 延迟大小
FROM pg_stat_replication;

-- 查看接收状态（从库上执行）
SELECT 
    status,               -- streaming = 正常
    receive_start_lsn,
    received_lsn,         -- 从主库收到了多少
    last_msg_send_time,   -- 最后一次收到消息的时间
    last_msg_receipt_time
FROM pg_stat_wal_receiver;
```

## 四、手把手搭建流复制

### 4.1 主库配置

```bash
# 1. 创建复制用户
sudo -u postgres psql
```

```sql
CREATE ROLE replicator WITH REPLICATION LOGIN ENCRYPTED PASSWORD 'Replicator@123';
```

```bash
# 2. 修改 postgresql.conf
sudo vim /etc/postgresql/16/main/postgresql.conf
```

```ini
# 主库核心配置
listen_addresses = '*'                    # 允许远程连接
wal_level = replica                       # WAL 级别：replica
max_wal_senders = 10                      # 最多 10 个从库同时连接
wal_keep_size = 1024                      # 保留 1GB WAL，供从库追赶

max_replication_slots = 10                # 最多 10 个复制槽
hot_standby = on                          # 从库允许只读查询

# 可选：启用归档
archive_mode = on
archive_command = 'cp %p /var/lib/postgresql/archive/%f'
```

```bash
# 3. 修改 pg_hba.conf，允许复制连接
sudo vim /etc/postgresql/16/main/pg_hba.conf
```

```ini
# 允许从库连接复制
host    replication     replicator      192.168.1.0/24     scram-sha-256
```

```bash
# 4. 重启主库
sudo systemctl restart postgresql
```

### 4.2 从库搭建

```bash
# 1. 停止从库 PostgreSQL
sudo systemctl stop postgresql

# 2. 清空数据目录
sudo rm -rf /var/lib/postgresql/16/main/*

# 3. 从主库拉取基础备份（关键步骤！）
sudo -u postgres pg_basebackup \
    -h 192.168.1.10 \          # 主库 IP
    -p 5432 \
    -U replicator \            # 复制用户
    -D /var/lib/postgresql/16/main \
    -Fp \                      # 普通格式
    -Xs \                      # 流式获取 WAL
    -P \                       # 显示进度
    -R                         # 自动创建 standby.signal 和连接配置

# -R 参数会自动创建两个文件：
# 1. standby.signal → 告诉 PostgreSQL 我是从库
# 2. postgresql.auto.conf → 自动写入主库连接信息

# 4. 配置从库
sudo vim /etc/postgresql/16/main/postgresql.conf
```

```ini
# 从库配置
hot_standby = on               # 允许只读查询
primary_conninfo = 'host=192.168.1.10 port=5432 user=replicator password=Replicator@123'
primary_slot_name = 'standby1_slot'
```

```bash
# 5. 启动从库
sudo systemctl start postgresql

# 6. 验证
sudo -u postgres psql -c "SELECT pg_is_in_recovery();"
# 返回 t (true) = 从库模式，搭建成功！
```

### 4.3 创建复制槽（重要！）

**为什么需要复制槽？** 假设从库宕机了一天，这期间主库产生的 WAL 可能已经被清理了。从库重启后发现 WAL 没了，就只能重新全量同步。

复制槽的作用：**告诉主库"这个从库还活着，别删它需要的 WAL"**。

```sql
-- 在主库上创建复制槽
SELECT pg_create_physical_replication_slot('standby1_slot');

-- 查看所有复制槽
SELECT 
    slot_name,
    slot_type,
    active,          -- 是否正在使用
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS retained_wal
FROM pg_replication_slots;

-- 删除不再需要的槽（危险！会导致从库需要全量重建）
SELECT pg_drop_replication_slot('old_standby_slot');
```

> **警告**：复制槽如果长时间没有从库连接，WAL 会一直堆积，直到撑爆磁盘！务必监控 `retained_wal` 大小。

## 五、复制延迟问题

### 5.1 延迟是怎么产生的？

主从复制的延迟 = 主库写入时间和从库重放完成时间的差值。可能的原因：

| 原因 | 表现 | 解决 |
|------|------|------|
| 网络带宽不足 | 大量 WAL 积压在主库 | 升级网络 / 压缩传输 |
| 从库性能差 | 从库 replay 速度跟不上 WAL 产生速度 | 升级从库硬件 |
| 从库上跑大查询 | 从库忙于处理查询，没空 replay WAL | 限制从库长查询 |
| 没有复制槽 | WAL 被清理，从库推倒重来 | 创建复制槽 |

### 5.2 延迟监控

```sql
-- 主库：精确的延迟计算
SELECT 
    application_name,
    client_addr,
    state,
    sync_state,
    -- 几个关键 LSN
    pg_wal_lsn_diff(pg_current_wal_lsn(), sent_lsn) AS pending_bytes,       -- 还没发出去的
    pg_wal_lsn_diff(sent_lsn, write_lsn) AS write_lag_bytes,                -- 发送中
    pg_wal_lsn_diff(write_lsn, flush_lsn) AS flush_lag_bytes,               -- 刷盘中
    pg_wal_lsn_diff(flush_lsn, replay_lsn) AS replay_lag_bytes,             -- 等待重放
    -- 总延迟
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)) AS total_lag
FROM pg_stat_replication;

-- 从库：查看自身延迟
SELECT 
    pg_is_in_recovery(),
    pg_last_wal_receive_lsn() AS received_lsn,
    pg_last_wal_replay_lsn() AS replayed_lsn,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn())
    ) AS replay_lag,
    now() - pg_last_xact_replay_timestamp() AS replication_delay
FROM pg_stat_wal_receiver;
```

### 5.3 延迟写在业务中的坑（重要！）

```sql
-- 场景：用户刚注册，立刻跳转到个人中心查询数据
-- 如果读请求落到从库上...

-- 用户注册（写主库）
INSERT INTO users (username, email) VALUES ('张三', 'zhangsan@example.com');

-- 马上查询（读从库）
SELECT * FROM users WHERE username = '张三';
-- 可能查不到！因为从库还在追赶 WAL！

-- 解决办法 1：关键操作强制读主库
SELECT * FROM users WHERE username = '张三' /* 标记：读主库 */;

-- 解决办法 2：写入后延迟一会再读
-- 应用层：注册成功后 sleep(100ms) 再跳转

-- 解决办法 3：同步复制，写主库时等从库确认
-- synchronous_commit = remote_apply
```

## 六、同步复制详解

### 6.1 同步复制的级别

从"主库什么时候告诉客户端成功"的角度，有 4 个级别：

| 级别 | 含义 | 性能 | 安全性 |
|-----|------|------|-------|
| **off** | 不等任何从库 | 最快 | 可能丢数据 |
| **local** | 主库自己写到磁盘就返回 | 快 | 主库崩溃丢数据 |
| **remote_write** | 从库收到并写入 OS 缓存 | 中等 | 从库 OS 崩溃丢数据 |
| **on** (默认) | 从库刷到磁盘 | 慢 | 从库宕机不丢数据 |
| **remote_apply** | 从库重放并可见 | 最慢 | 最安全，读写强一致 |

```sql
-- 设置为 remote_apply（读从库也能读到刚写的数据）
ALTER SYSTEM SET synchronous_commit = 'remote_apply';
SELECT pg_reload_conf();

-- 查看当前设置
SHOW synchronous_commit;
```

### 6.2 多从库的同步策略

```ini
# 至少 1 个从库同步即可（FIRST）
synchronous_standby_names = 'FIRST 1 (standby1, standby2, standby3)'

# 任意 1 个从库同步即可（和 FIRST 1 类似，但不按优先级）
synchronous_standby_names = 'ANY 1 (standby1, standby2, standby3)'

# 必须 2 个从库都同步
synchronous_standby_names = 'FIRST 2 (standby1, standby2)'

# 指定优先级排序
synchronous_standby_names = '2 (standby1, standby2, standby3)'
```

### 6.3 同步复制的风险

**如果所有同步从库都挂了，主库会卡住**——写操作会无限等待，直到至少有一个同步从库恢复连接。

```sql
-- 主库会看到这个错误：
-- ERROR:  canceling statement due to user request
-- 写操作一直等待从库确认，超时后被取消

-- 紧急处理：降级为异步
ALTER SYSTEM SET synchronous_standby_names = '';
SELECT pg_reload_conf();
-- 主库恢复写入，但从库数据可能落后
```

## 七、逻辑复制

### 7.1 逻辑复制 vs 流复制

| 对比维度 | 流复制（物理复制） | 逻辑复制 |
|---------|-----------------|---------|
| 复制粒度 | 整个实例（所有库所有表） | 按表、按库 |
| 版本兼容 | 主从必须同大版本 | **支持跨大版本** |
| 选择性 | 全有或全无 | 可以只复制某几张表 |
| 写入从库 | 从库只读 | 从库也可以写 |
| 使用场景 | 高可用、读写分离 | **版本升级、数据迁移、多活** |

### 7.2 逻辑复制搭建

```sql
-- === 主库（发布端）===

-- 1. 设置 WAL 级别
-- wal_level = logical

-- 2. 创建发布
CREATE PUBLICATION my_pub FOR TABLE users, orders;
-- 或发布所有表
CREATE PUBLICATION my_pub FOR ALL TABLES;

-- 3. 查看发布
SELECT * FROM pg_publication;
\dRp+

-- === 从库（订阅端）===

-- 4. 创建订阅
CREATE SUBSCRIPTION my_sub
CONNECTION 'host=192.168.1.10 port=5432 dbname=mydb user=replicator password=Replicator@123'
PUBLICATION my_pub;

-- 5. 查看订阅状态
SELECT 
    subname,
    pid,
    received_lsn,
    latest_end_lsn,
    pg_size_pretty(pg_wal_lsn_diff(received_lsn, latest_end_lsn)) AS lag
FROM pg_stat_subscription;

-- 6. 管理订阅
ALTER SUBSCRIPTION my_sub ENABLE;          -- 启用
ALTER SUBSCRIPTION my_sub DISABLE;         -- 暂停
ALTER SUBSCRIPTION my_sub REFRESH PUBLICATION;  -- 刷新
DROP SUBSCRIPTION my_sub;                  -- 删除
```

### 7.3 逻辑复制的经典应用场景

**场景 1：大版本升级零停机**

```
旧版本 PG13 主库
    │
    │ 逻辑复制
    ↓
新版本 PG16 从库  ← 验证数据一致后，切换到这个库
```

**场景 2：多活写入**

```
  北京库 ←─── 逻辑复制 ───→ 深圳库
   (写北京用户)              (写深圳用户)
```

两张表互相复制，各自写各自的数据。

## 八、复制监控与告警

### 8.1 监控 SQL 合集

```sql
-- 1. 复制状态总览
SELECT 
    pid,
    usename,
    application_name,
    client_addr,
    state,
    sync_state,
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)) AS lag
FROM pg_stat_replication;

-- 2. 复制槽状态（关注 active = false 的槽）
SELECT 
    slot_name,
    active,
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS retained_wal
FROM pg_replication_slots
WHERE NOT active;

-- 3. WAL 产生速率
SELECT 
    pg_current_wal_lsn(),
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0')
    ) AS total_wal;

-- 4. 从库是否存活（在主库执行，看 state 是否 streaming）
SELECT count(*) > 0 AS has_standby
FROM pg_stat_replication
WHERE state = 'streaming';
```

### 8.2 关键告警指标

| 指标 | 告警阈值 | 说明 |
|-----|---------|------|
| 复制延迟 | > 100MB 或 > 10秒 | 从库严重落后 |
| 复制槽 inactive | > 5分钟 | 从库失联 |
| 复制槽 WAL 堆积 | > 10GB | 可能导致磁盘满 |
| 从库总数 | < 1 | 没有从库可用 |

## 九、总结

| 概念 | 一句话理解 |
|------|----------|
| **WAL** | 数据库的"操作日记"，先写日记再改数据 |
| **LSN** | WAL 里的字节偏移量，相当于数据的版本号 |
| **流复制** | 主库把 WAL 推给从库，从库照着做一遍 |
| **复制槽** | 告诉主库"别删这个从库需要的 WAL" |
| **同步复制** | 主库必须等从库确认收到 WAL 才能提交事务 |
| **逻辑复制** | 按表级别复制，支持跨版本、选择性复制 |
