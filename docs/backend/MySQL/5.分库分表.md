# MySQL分库分表深度解析

## 一、分库分表的定义

### 1. 什么是分库分表
分库分表是一种数据库水平扩展方案，通过将单一数据库拆分为多个数据库（分库）或将单一表拆分为多个表（分表），来解决数据库性能瓶颈和存储容量限制的问题。

### 2. 分库分表的目的
- **解决性能瓶颈**：当单库单表数据量过大时，查询性能会显著下降
- **提高系统可用性**：分散风险，单个库或表故障不会影响整个系统
- **提高并发能力**：分散并发请求到多个数据库实例
- **突破存储限制**：单个数据库的存储容量有限，分库分表可以扩展存储容量

### 3. 分库分表的类型
- **水平分库**：将数据按照一定规则分散到不同的数据库实例中
- **水平分表**：将数据按照一定规则分散到同一个数据库的不同表中
- **垂直分库**：按照业务功能将表分散到不同的数据库实例中
- **垂直分表**：将表的列按照一定规则分散到不同的表中

## 二、分库分表的原理

### 1. 水平分库分表

#### 1.1 分片策略
- **范围分片**：根据数据的范围进行分片，如按时间范围、ID范围等
- **哈希分片**：根据数据的哈希值进行分片，如对ID进行哈希取模
- **列表分片**：根据数据的枚举值进行分片，如按地区、用户类型等
- **复合分片**：结合多种分片策略

#### 1.2 分片键的选择
- **选择原则**：
  - 唯一性：确保数据均匀分布
  - 稳定性：避免数据频繁迁移
  - 业务相关性：与业务查询模式匹配
  - 可扩展性：支持未来数据增长

- **常见分片键**：
  - 用户ID
  - 订单ID
  - 时间戳
  - 地理位置

### 2. 垂直分库分表

#### 2.1 垂直分库
- **原理**：按照业务功能将表分散到不同的数据库实例中
- **适用场景**：不同业务模块的访问频率和数据量差异较大
- **优势**：隔离业务，提高安全性和稳定性

#### 2.2 垂直分表
- **原理**：将表的列按照访问频率和大小分散到不同的表中
- **适用场景**：表的列数过多，或某些列访问频率低
- **优势**：减少数据传输量，提高查询性能

### 3. 分库分表的实现方式

#### 3.1 应用层实现
- **优点**：灵活性高，可定制性强
- **缺点**：开发和维护成本高
- **实现方式**：在应用代码中实现分片逻辑

#### 3.2 中间件实现
- **优点**：透明性高，开发成本低
- **缺点**：引入额外依赖，可能影响性能
- **常见中间件**：
  - Sharding-JDBC
  - MyCAT
  - ProxySQL
  - Vitess

### 4. 数据迁移与扩容

#### 4.1 数据迁移
- **离线迁移**：停止服务进行数据迁移
- **在线迁移**：在不停止服务的情况下进行数据迁移
- **工具**：pt-online-schema-change、Canal等

#### 4.2 扩容策略
- **翻倍扩容**：将分片数量翻倍，如从4个分片扩容到8个分片
- **一致性哈希**：使用一致性哈希算法，减少数据迁移量

## 三、分库分表的优缺点

### 1. 优点
- **提高性能**：分散数据量，减少单库单表的压力
- **提高可用性**：单个库或表故障不会影响整个系统
- **提高并发能力**：分散并发请求到多个数据库实例
- **突破存储限制**：扩展存储容量
- **便于维护**：可以针对不同分片进行独立维护和优化

### 2. 缺点
- **增加系统复杂度**：需要处理分片逻辑、数据迁移、事务等问题
- **跨分片查询困难**：需要处理跨分片的查询和聚合
- **事务处理复杂**：分布式事务处理难度大
- **数据一致性挑战**：需要保证多个分片之间的数据一致性
- **运维成本增加**：需要维护多个数据库实例
- **技术门槛高**：需要掌握分布式系统相关知识

  - 优点：灵活性高，适应业务变化
  - 缺点：实现复杂，需要动态调整路由规则

```java
public void migrateData(int oldShardCount, int newShardCount) {
    // 新分片数量必须是旧分片数量的倍数
    if (newShardCount % oldShardCount != 0) {
        throw new IllegalArgumentException("New shard count must be a multiple of old shard count");
    }
    
    int factor = newShardCount / oldShardCount;
    
    // 迁移数据
    for (int i = 0; i < oldShardCount; i++) {
        String oldTable = "order_" + i;
        // 查询旧分片的数据
        List<Order> orders = jdbcTemplate.query(
            "SELECT * FROM " + oldTable,
            new BeanPropertyRowMapper<>(Order.class)
        );
        
        // 分配到新分片
        for (Order order : orders) {
            int newShardIndex = (int) (order.getId() % newShardCount);
            String newTable = "order_" + newShardIndex;
            // 插入到新分片
            jdbcTemplate.update(
                "INSERT INTO " + newTable + " VALUES (?, ?, ?, ?)",
                order.getId(), order.getUserId(), order.getAmount(), order.getCreateTime()
            );
        }
    }
}
```

### 12. 分库分表后如何处理全局唯一ID？

**详细答案**：
分库分表后生成全局唯一ID的方法：

- **UUID**：全局唯一，但字符串较长，索引效率低
- **雪花算法（Snowflake）**：生成64位数字ID，包含时间戳、机器ID、序列号
- **数据库自增**：使用单独的数据库实例生成自增ID
- **Redis自增**：使用Redis的INCR命令生成自增ID
- **Leaf**：美团开源的分布式ID生成系统
- **UID-Generator**：百度开源的分布式ID生成系统

**代码示例**：
```java
// 雪花算法生成全局唯一ID
public class SnowflakeIdGenerator {
    private final long workerId;
    private final long datacenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;
    
    public SnowflakeIdGenerator(long workerId, long datacenterId) {
        this.workerId = workerId;
        this.datacenterId = datacenterId;
    }
    
    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("Clock moved backwards");
        }
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & 4095;
            if (sequence == 0) {
                timestamp = tilNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        lastTimestamp = timestamp;
        return ((timestamp - 1288834974657L) << 22)
                | (datacenterId << 17)
                | (workerId << 12)
                | sequence;
    }
    
    private long tilNextMillis(long lastTimestamp) {
        long timestamp = System.currentTimeMillis();
        while (timestamp <= lastTimestamp) {
            timestamp = System.currentTimeMillis();
        }
        return timestamp;
    }
}
```
