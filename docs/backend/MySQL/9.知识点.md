# MySQL 高频面试题

> 本章汇总 MySQL 核心知识点与高频面试题，与事务、锁、MVCC、索引、分库分表各章节互补，侧重面试中常问的概念辨析、原理追问及实战问题。

## 一、InnoDB 与 MyISAM 的区别

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务支持 | 支持（ACID） | 不支持 |
| 锁粒度 | 行级锁 + 间隙锁 | 表级锁 |
| 崩溃恢复 | 自动崩溃恢复（redo log） | 需手动修复（myisamchk） |
| 外键 | 支持 | 不支持 |
| 全文索引 | 5.6+ 支持 | 原生支持 |
| 存储结构 | 聚簇索引（数据与主键在一起） | 非聚簇索引（索引与数据分开） |
| count(*) | 全表扫描，较慢 | 保存行数，快 |
| 适用场景 | 高并发、写密集型 | 读密集型、不需要事务的场景 |

**面试追问**：为什么 InnoDB 推荐使用自增主键？
- 答案：自增主键使新记录始终插入到 B+ 树叶子节点的最后位置，避免页分裂，减少索引维护开销。若使用 UUID 或非顺序主键，插入时可能触发页分裂和页合并，影响性能。

## 二、一条 SQL 查询语句是如何执行的？

```
连接器 → 查询缓存 → 分析器 → 优化器 → 执行器 → 存储引擎
```

1. **连接器**：建立 TCP 连接，进行身份验证，查询权限
2. **查询缓存**（MySQL 8.0 已移除）：命中则直接返回结果；命中率低，且更新操作会清空缓存，8.0 后已废弃
3. **分析器**：词法分析 + 语法分析，生成解析树
4. **优化器**：选择最优执行计划（如选择哪个索引、JOIN 顺序）
5. **执行器**：调用存储引擎接口，逐行读取并过滤，返回结果

## 三、一条 SQL 更新语句是如何执行的？

```
连接器 → 分析器 → 优化器 → 执行器 → 存储引擎
                  ↓
             记录 Redo Log（prepare）
                  ↓
             记录 Binlog
                  ↓
          提交事务（commit）
```

**重点：两阶段提交**

| 阶段 | 日志 | 说明 |
|------|------|------|
| 1 | Redo Log prepare | 写入 Redo Log，状态设为 prepare |
| 2 | Binlog 写入 | 写入 Binlog |
| 3 | Redo Log commit | 将 Redo Log 状态设为 commit |

**为什么需要两阶段提交？**
- 如果先写 Redo Log 再写 Binlog：Redo Log 写完但 Binlog 未写时崩溃，恢复时 Redo Log 会将数据恢复，但 Binlog 无记录，主从不一致
- 如果先写 Binlog 再写 Redo Log：Binlog 记录了但 Redo Log 未写，崩溃后数据未恢复，但 Binlog 有记录，同样不一致
- **两阶段提交保证两份日志逻辑一致**，是实现主从一致的基础

**Redo Log 与 BinLog 的区别**：

| | Redo Log | BinLog |
|---|---|---|
| 归属 | InnoDB 存储引擎 | MySQL Server 层 |
| 作用 | 崩溃恢复（物理日志） | 主从复制 + 数据恢复 |
| 写入方式 | 循环写，固定大小 | 追加写，可无限增长 |
| 内容 | 物理页的修改 | SQL 语句或行变化 |

## 四、MySQL 日志体系

### 1. Error Log（错误日志）
- 记录 MySQL 启动、运行、停止时的错误信息
- 默认开启，由 `log_error` 配置

### 2. Slow Query Log（慢查询日志）
- 记录执行时间超过 `long_query_time` 的 SQL（默认 10s）
- 配合 `slow_query_log_file` 使用
- 可用 `mysqldumpslow` 或 `pt-query-digest` 分析

```sql
-- 查看慢查询配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询（临时）
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;
```

### 3. General Query Log（通用查询日志）
- 记录所有连接和 SQL 语句（生产环境谨慎开启）
- 文件位置由 `general_log_file` 指定

### 4. Undo Log（回滚日志）
- 记录数据修改前的值，用于事务回滚
- 是 MVCC 实现的基础（ReadView + undo log 链）

### 5. Relay Log（中继日志）
- 从库读取主库 BinLog 后写入本地 relay log
- 由 SQL 线程重放，执行同步

## 五、EXPLAIN 深度解析

`EXPLAIN` 是分析 SQL 性能的核心工具，常见字段：

### type（访问类型，从优到差）
```
system > const > eq_ref > ref > range > index > ALL
```
| type | 含义 |
|------|------|
| system | 表只有一行（系统表） |
| const | 主键或唯一索引等值查询，只匹配一行 |
| eq_ref | 唯一索引扫描，每行只返回一条 |
| ref | 非唯一索引等值查询 |
| range | 索引范围扫描（> < between in） |
| index | 全索引扫描 |
| ALL | 全表扫描（最差） |

### key 与 possible_keys
- `possible_keys`：可能被使用的索引
- `key`：实际使用的索引

### rows
预估扫描行数，越少越好

### Extra（关键信息）
| 值 | 含义 |
|----|------|
| Using index | **覆盖索引**，无需回表 |
| Using where | 使用 WHERE 过滤 |
| Using temporary | 使用临时表（需优化） |
| Using filesort | 文件排序，**需优化** |
| Using index condition | 索引下推（ICP） |
| Using join buffer | JOIN 时使用 join buffer |

**案例**：
```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 1 ORDER BY create_time;
-- Extra: Using where; Using index  ✓ 良好
-- Extra: Using filesort            ✗ 需要优化（给 create_time 加索引）
```

## 六、SQL 优化方法

### 1. 避免全表扫描
- 给 WHERE、JOIN、ORDER BY、GROUP BY 涉及字段加索引
- 避免在 WHERE 条件中对字段做运算或函数

```sql
-- ❌ 索引失效
WHERE YEAR(create_time) = 2026
WHERE id + 1 = 100

-- ✅ 正确写法
WHERE create_time >= '2026-01-01' AND create_time < '2027-01-01'
WHERE id = 99
```

### 2. 避免 SELECT *
- 只查询需要的列，减少网络传输和回表次数
- 配合覆盖索引效果更佳

### 3. 批量插入 vs 逐条插入
```sql
-- ❌ 逐条插入
INSERT INTO t VALUES (1); INSERT INTO t VALUES (2);

-- ✅ 批量插入
INSERT INTO t VALUES (1), (2), (3), (4), (5);
```

### 4. 大表分页优化
```sql
-- ❌ 深度分页（OFFSET 越大越慢）
SELECT * FROM orders LIMIT 1000000, 10;

-- ✅ 方案1：主键延迟关联
SELECT * FROM orders o
INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 1000000, 10) t
ON o.id = t.id;

-- ✅ 方案2：游标分页
SELECT * FROM orders WHERE id > #{lastId} ORDER BY id LIMIT 10;
```

### 5. JOIN 优化
- 小表驱动大表（MySQL 自动优化，但需注意字段顺序）
- 关联字段加索引
- 避免 JOIN 超过 3 张表

```sql
-- EXPLAIN 查看 join 类型，确保非嵌套循环（nj > 3）
EXPLAIN SELECT * FROM orders o INNER JOIN users u ON o.user_id = u.id;
```

## 七、MySQL 主从复制原理

```
主库：
  事务执行 → 记录 Binlog（dump thread）

从库：
  I/O thread → 接收 Binlog → 写入 Relay Log
  SQL thread → 读取 Relay Log → 重放事件 → 执行 SQL
```

### 三种复制方式

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 异步复制 | 主库提交后不等从库 | 性能高 | 可能丢数据 |
| 半同步复制 | 主库等至少一个从库写入 relay log | 数据安全性提升 | 有一定延迟 |
| 全同步复制 | 主库等所有从库执行完 | 强一致 | 性能差 |

### 主从延迟原因及解决
| 原因 | 解决方案 |
|------|---------|
| 从库 SQL 线程单线程重放 | 升级到并行复制（5.6+ 多线程 SQL thread） |
| 大事务执行时间长 | 拆分为小事务 |
| 网络延迟 | 部署同机房或优化网络 |
| 从库配置低 | 提升从库硬件配置 |

**如何判断主从延迟？**
```sql
SHOW SLAVE STATUS\G
-- Seconds_Behind_Master: 0 表示无延迟，NULL 表示未启动
```

## 八、MySQL 读写分离与负载均衡

### 读写分离架构
```
应用层
    ↓
  Proxy / Sharding-JDBC
    ↓
主库（写）  ←→  多个从库（读）
```

### 常见中间件
- **Sharding-JDBC**：客户端分片，轻量，Java 技术栈首选
- **MyCAT**：服务端代理，功能全
- **ProxySQL**：专注于 MySQL 的代理

### 主从数据一致性问题
- 读写分离后，写操作立即读到旧数据（主从延迟）
- 解决方案：
  - **强制走主库**：对一致性要求高的读请求路由到主库
  - **选择性走主库**：关键业务读主库，普通读从库
  - **延迟感知**：在应用层记录延迟时间，延迟期内走主库

## 九、MySQL 三大范式与反范式设计

### 三大范式
| 范式 | 要求 | 示例 |
|------|------|------|
| 1NF | 字段原子性，不可再分 | 地址字段拆分为省/市/区 |
| 2NF | 满足1NF + 非主键字段完全依赖主键 | 消除部分依赖（复合主键场景） |
| 3NF | 满足2NF + 非主键字段不传递依赖主键 | A→B→C 拆表 |

### 反范式设计
为提升查询性能，故意违反范式，冗余存储数据：

| 场景 | 反范式做法 |
|------|-----------|
| 避免 JOIN | 在订单表冗余用户名，避免查用户表 |
| 热门字段 | 缓存常用统计值（如阅读量） |
| 时间序列查询 | 冗余月/日字段，避免 YEAR() MONTH() 函数 |

**原则**：范式是起点，反范式是优化，根据业务场景权衡。

## 十、COUNT(*) / COUNT(1) / COUNT(列) 区别

| 写法 | 说明 | 性能 |
|------|------|------|
| COUNT(*) | 统计行数，包含 NULL | InnoDB 优化过，最快 |
| COUNT(1) | 统计行数，1 作为表达式 | 与 COUNT(*) 几乎无差别 |
| COUNT(列) | 统计该列非 NULL 行数 | 最慢，需逐行检查 NULL |

**InnoDB 为什么 MyISAM 的 COUNT(*) 更快？**
- MyISAM 把总行数存在元数据中，查询直接返回
- InnoDB 需要实时扫描，MVCC 机制下无法直接返回准确行数

## 十一、一张表中字段数量多少合适？

| 场景 | 建议 |
|------|------|
| 字段数量 | 单表字段数建议不超过 **20-30** 个 |
| 原因 | 数据页 16KB 固定，过多字段导致行过长，页利用率下降，频繁页分裂 |
| 解决方案 | 垂直分表：将大字段（TEXT/VARCHAR(1000+)）或低频字段拆分出去 |

**VARCHAR(1000) vs TEXT 的区别？**
- VARCHAR：行内存储，最大 65535 字节，有长度限制
- TEXT：行外存储，只存指针，不影响行大小，但无法建完整索引（需前缀索引）

## 十二、MySQL 参数调优（高频配置项）

| 参数 | 说明 | 建议值 |
|------|------|--------|
| `innodb_buffer_pool_size` | 缓冲池大小，存数据和索引 | 设置为可用内存的 **60-80%** |
| `innodb_log_file_size` | Redo Log 文件大小 | 较大值（如 1GB）减少日志切换 |
| `max_connections` | 最大连接数 | 根据业务调整，过大耗内存 |
| `innodb_flush_log_at_trx_commit` | Redo Log 刷盘策略 | 1=强一致，2=折中，0=高性能低一致 |
| `sync_binlog` | BinLog 刷盘策略 | 1=每次提交刷盘（安全），n=每n次 |
| `slow_query_log` | 慢查询日志开关 | 生产环境开启 |
| `long_query_time` | 慢查询阈值（秒） | 通常设为 1 |

## 十三、常见问题与解决思路

### Q1：MySQL CPU 飙高怎么处理？
**排查步骤**：
1. `SHOW PROCESSLIST` 查看连接状态
2. `EXPLAIN` 定位慢查询
3. 检查是否有全表扫描
4. 确认索引是否生效
5. 考虑加缓存（Redis）分担读压力
6. 读写分离分散负载

### Q2：MySQL 连接数打满怎么办？
```sql
-- 查看当前连接
SHOW STATUS LIKE 'Threads_connected';
SHOW PROCESSLIST;

-- 临时增加连接数
SET GLOBAL max_connections = 2000;

-- 建议：使用连接池（HikariCP / Druid），避免频繁建立连接
```

### Q3：MySQL 突然卡住怎么办？
1. 检查锁等待：`SELECT * FROM information_schema.INNODB_LOCK_WAITS;`
2. 查看事务：`SELECT * FROM information_schema.INNODB_TRX;`
3. 查看锁：`SHOW ENGINE INNODB STATUS;`
4. 杀掉长时间未提交的事务：`KILL <trx_mysql_thread_id>`

### Q4：如何删除大表中的大量数据？
```sql
-- ❌ DELETE 逐条删除：产生大量 binlog，耗时且阻塞
DELETE FROM orders WHERE create_time < '2024-01-01';

-- ✅ 分批删除：每次删除 1000 条，减小锁范围
DELETE FROM orders WHERE create_time < '2024-01-01' LIMIT 1000;

-- ✅ 最佳方案：新建表 + 迁移 + rename（DDL，非 DML）
CREATE TABLE orders_new LIKE orders;
INSERT INTO orders_new SELECT * FROM orders WHERE create_time >= '2024-01-01';
RENAME TABLE orders TO orders_old, orders_new TO orders;
DROP TABLE orders_old;
```

### Q5：MySQL 如何实现高可用？
| 方案 | 原理 | 适用场景 |
|------|------|---------|
| 主从 + MHA/Orchestrator | 自动切换主库 | 中小型业务 |
| MGR（MySQL Group Replication） | 多主同步， Paxos 协议 | 高可用要求高 |
| MySQL Cluster | 分布式集群 | 超大规模 |
| Galera Cluster | 多主同步，无延迟 | 强一致场景 |
