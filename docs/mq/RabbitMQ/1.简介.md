# RabbitMQ 简介

## 一、什么是 RabbitMQ

RabbitMQ 是一个开源的消息代理软件（也称为消息队列），最初由 Rabbit Technologies Ltd. 开发，后来被 Pivotal Software（现为 VMware 的一部分）收购。

### 核心定义

RabbitMQ 实现了 **AMQP（Advanced Message Queuing Protocol，高级消息队列协议）**，是一个可靠、可扩展、可移植的消息传递系统。

### 历史背景

| 时间 | 事件 |
|------|------|
| 2007年 | Rabbit Technologies Ltd. 发布 RabbitMQ 1.0 |
| 2010年 | 被 SpringSource（现 Pivotal）收购 |
| 2013年 | 发布支持 AMQP 1.0 的版本 |
| 2019年 | 发布 RabbitMQ 3.8，引入 Quorum Queues |
| 2021年 | 发布 RabbitMQ 3.9，引入 Streams |

## 二、RabbitMQ 的特点

### 2.1 核心特性

| 特性 | 说明 |
|------|------|
| **可靠性** | 支持消息持久化、消息确认、事务等机制 |
| **灵活性** | 支持多种消息路由模式 |
| **高可用性** | 支持集群部署、镜像队列 |

| **插件系统** | 丰富的插件生态 |

#### 适用场景

**适合使用 RabbitMQ 的场景：**

1. **业务系统解耦**：订单、支付、物流等业务系统
2. **异步处理**：用户注册后发送欢迎邮件、短信
3. **流量削峰**：秒杀、抢购等高并发场景
4. **广播通知**：配置更新、系统通知
5. **延迟任务**：订单超时取消、定时提醒
6. **实时性要求高**：延迟要求在毫秒级

**不适合使用 RabbitMQ 的场景：**

1. **海量日志收集**：建议使用 Kafka
2. **流式数据处理**：建议使用 Kafka
3. **超大数据量堆积**：Kafka 更适合

### 3.2 核心组件

#### Producer（生产者）

创建消息并发布到 Exchange 的客户端。

```
Producer → 发送消息 → Exchange
```

#### Exchange（交换机）

接收生产者发送的消息，并根据路由规则将消息路由到一个或多个 Queue。

**类型**：
- Direct Exchange（直连交换机）
- Fanout Exchange（扇形交换机）
- Topic Exchange（主题交换机）
- Headers Exchange（头交换机）

#### Queue（队列）

存储消息的容器，消息直到被消费或过期。

```
Queue → 存储消息 → Consumer
```

#### Binding（绑定）

Exchange 和 Queue 之间的关联关系，包含路由键（Routing Key）。

```
Exchange --[Binding: routing-key]--> Queue
```

#### Consumer（消费者）

从 Queue 获取消息并处理的客户端。

#### Connection（连接）

客户端与 RabbitMQ 之间的 TCP 连接。

#### Channel（通道）

在 Connection 内部创建的逻辑连接，用于多路复用。

```
Connection
    ├── Channel 1
    ├── Channel 2
    └── Channel 3
```

#### Virtual Host（虚拟主机）

RabbitMQ 内部的逻辑隔离单位，类似数据库中的数据库概念。

#### Broker（代理）

RabbitMQ 服务器实例，包含 Exchange、Queue、Binding 等组件。

## 四、RabbitMQ 应用场景

### 4.1 解耦

**场景**：用户注册后，需要发送欢迎邮件、发送欢迎短信、赠送优惠券。

**传统方式**：
```
用户注册 → 发送邮件 → 发送短信 → 赠送优惠券
        (耦合度高，任何一个环节失败都影响整体)
```

**使用 RabbitMQ**：
```
用户注册 → 发送消息到 MQ
                ↓
         ┌──────┼──────┐
         ↓      ↓      ↓
      邮件服务 短信服务 优惠券服务
      (独立处理，互不影响)
```

**代码示例**：
```java
// 用户注册服务
public void register(User user) {
    // 1. 保存用户
    userDao.save(user);

    // 2. 发送消息到 MQ
    rabbitTemplate.convertAndSend("user.registered", user);
}

// 邮件服务
@RabbitListener(queues = "email.queue")
public void sendWelcomeEmail(User user) {
    emailService.send(user.getEmail(), "欢迎注册");
}

// 短信服务
@RabbitListener(queues = "sms.queue")
public void sendWelcomeSms(User user) {
    smsService.send(user.getPhone(), "欢迎注册");
}
```

### 4.2 异步

**场景**：用户下单后，需要扣减库存、生成订单、发送通知。

**同步方式**：
```
用户下单 → 扣减库存 → 生成订单 → 发送通知 → 返回结果
(响应时间 = 各环节时间之和，用户体验差)
```

**异步方式**：
```
用户下单 → 返回成功 → 异步处理
                    ↓
         ┌──────────┼──────────┐
         ↓          ↓          ↓
      扣减库存    生成订单    发送通知
(响应时间 = 最短环节时间，用户体验好)
```

**代码示例**：
```java
// 订单服务
@Transactional
public void createOrder(Order order) {
    // 1. 生成订单
    orderDao.save(order);

    // 2. 发送消息到 MQ，异步处理其他业务
    rabbitTemplate.convertAndSend("order.created", order);

    // 3. 立即返回
    return order;
}

// 库存服务
@RabbitListener(queues = "inventory.queue")
public void deductStock(Order order) {
    inventoryService.deduct(order.getProductId(), order.getCount());
}
```

### 4.3 削峰

**场景**：秒杀活动，瞬间流量巨大，可能压垮数据库。

**问题**：
```
正常流量：100 QPS
秒杀流量：10000 QPS
数据库承受能力：500 QPS

直接请求 → 数据库崩溃
```

**解决方案**：
```
用户请求 → MQ (缓冲) → 消费者 (限速) → 数据库
        ↓
    消息队列（可以堆积）
```

**代码示例**：
```java
// 秒杀接口
public void seckill(Long productId, Long userId) {
    // 1. 发送消息到 MQ（快速返回）
    rabbitTemplate.convertAndSend("seckill.queue",
        new SeckillMessage(productId, userId));
}

// 消费者（限速消费）
@RabbitListener(queues = "seckill.queue")
@RabbitListener(concurrency = "10-20") // 限制并发数
public void processSeckill(SeckillMessage message) {
    // 2. 扣减库存
    inventoryService.deduct(message.getProductId(), 1);

    // 3. 创建订单
    orderService.create(message.getUserId(), message.getProductId());
}
```

### 4.4 广播

**场景**：系统配置更新，需要通知所有服务节点。

**代码示例**：
```java
// 配置中心
public void updateConfig(Config config) {
    // 1. 更新数据库
    configDao.update(config);

    // 2. 广播配置更新
    rabbitTemplate.convertAndSend("config.exchange", "",
        new ConfigUpdateEvent(config));
}

// 各服务监听配置更新
@RabbitListener(queues = "config.queue")
public void onConfigUpdate(ConfigUpdateEvent event) {
    // 刷新本地配置缓存
    configCache.refresh(event.getConfig());
}
```

### 4.5 延迟任务

**场景**：订单下单后 30 分钟未支付，自动取消。

**代码示例**：
```java
// 创建订单
public void createOrder(Order order) {
    orderDao.save(order);

    // 发送延迟消息，30 分钟后检查订单状态
    rabbitTemplate.convertAndSend("delayed.exchange", "order.cancel",
        new OrderCancelMessage(order.getId()),
        message -> {
            message.getMessageProperties().setDelay(30 * 60 * 1000);
            return message;
        });
}

// 延迟消息消费者
@RabbitListener(queues = "order.cancel.queue")
public void cancelOrder(OrderCancelMessage message) {
    Order order = orderDao.findById(message.getOrderId());
    if (order.getStatus() == OrderStatus.UNPAID) {
        order.setStatus(OrderStatus.CANCELLED);
        orderDao.update(order);
    }
}
```

### 5.2 Quorum Queues（仲裁队列）

RabbitMQ 3.8 引入的新队列类型，用于替代镜像队列。

**特点**：
- 使用 Raft 共识算法
- 数据一致性更强
- 更好的故障恢复能力
- 支持消息去重

**适用场景**：
- 对数据一致性要求高的场景
- 需要消息可靠投递的场景

### 5.3 Streams（流）

RabbitMQ 3.9 引入的日志存储结构，类似 Kafka。

**特点**：
- 基于日志追加
- 支持消息回溯
- 高吞吐量
- 支持 AMQP 协议

**适用场景**：
- 事件溯源
- 日志收集
- 流式处理

## 六、RabbitMQ 性能指标

### 6.1 性能参考

| 场景 | 吞吐量 | 延迟 |
|------|--------|------|
| 简单发送 | 50,000 msg/s | < 1ms |
| 持久化消息 | 20,000 msg/s | < 5ms |
| 事务消息 | 5,000 msg/s | < 10ms |
| 集群模式 | 40,000 msg/s | < 2ms |

### 6.2 影响性能的因素

1. **消息大小**：消息越大，吞吐量越低
2. **持久化**：开启持久化会降低性能
3. **网络延迟**：网络质量影响消息传输
4. **消费者处理速度**：消费者处理慢会导致消息堆积
5. **硬件配置**：CPU、内存、磁盘 I/O 都会影响性能
