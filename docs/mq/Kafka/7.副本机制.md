# Kafka 副本机制

## 副本概述

### 副本类型

| 类型 | 角色 | 职责 |
|------|------|------|
| Leader | 主副本 | 处理所有读写请求 |
| Follower | 从副本 | 从 Leader 同步数据，参与 Leader 选举 |

### 副本分布策略

副本均匀分布在不同的 Broker 上，以提高容错性。

```
Topic: my-topic (3个分区，3个副本)

Partition 0:
├── Leader: Broker 1
├── Follower: Broker 2
└── Follower: Broker 3

Partition 1:
├── Leader: Broker 2
├── Follower: Broker 3
└── Follower: Broker 1

Partition 2:
├── Leader: Broker 3
├── Follower: Broker 1
└── Follower: Broker 2
```

## ISR（In-Sync Replica）

### ISR 概念

ISR 是与 Leader 保持同步的副本集合，只有 ISR 中的副本才有资格成为新的 Leader。

### ISR 判断标准

```properties
# Follower 落后 Leader 的最大时间
replica.lag.time.max.ms=30000     # 默认 30 秒
```

如果一个 Follower 在 `replica.lag.time.max.ms` 时间内没有向 Leader 发送拉取请求，或者虽然发送了请求但落后 Leader 太多，就会被移出 ISR。

### ISR 动态变化

```
ISR = [Leader, Follower1, Follower2]

↓ Follower2 故障，落后超过 30 秒

ISR = [Leader, Follower1]

↓ Follower2 恢复，追上 Leader

ISR = [Leader, Follower1, Follower2]
```

### 查看 ISR

```bash
# 查看 Topic 详细信息
bin/kafka-topics.sh --describe \n  --bootstrap-server localhost:9092 \n  --topic my-topic

# 输出示例
Topic: my-topic   PartitionCount: 3   ReplicationFactor: 3
Topic: my-topic   Partition: 0    Leader: 1   Replicas: 1,2,3   Isr: 1,2,3
Topic: my-topic   Partition: 1    Leader: 2   Replicas: 2,3,1   Isr: 2,3
Topic: my-topic   Partition: 2    Leader: 3   Replicas: 3,1,2   Isr: 3,1,2
```

## 副本同步机制

### 同步流程

```
Follower                        Leader
   │                              │
   │──── Fetch Request ──────────>│
   │                              │
   │<──── Fetch Response ─────────│
   │    (messages + HW + LEO)     │
   │                              │
   │──── Append to Log            │
   │                              │
   │──── Update LEO               │
   │                              │
```

### 关键概念

#### LEO（Log End Offset）

日志末端偏移量，表示下一条待写入消息的 offset。

```
Log: [0, 1, 2, 3, 4, 5] (6条消息)
LEO = 6 (下一条消息的 offset)
```

#### HW（High Watermark）

高水位，表示 ISR 中所有副本都已同步的消息 offset。

```
Leader LEO = 10
Follower1 LEO = 8
Follower2 LEO = 9

HW = min(10, 8, 9) = 8

消费者只能消费到 HW 之前的消息（offset < 8）
```

### HW 更新机制

```
1. Follower 从 Leader 拉取数据
2. Leader 更新 Follower 的 LEO
3. Leader 计算 HW = min(所有 ISR 副本的 LEO)
4. Leader 将 HW 返回给 Follower
5. Follower 更新自己的 HW
```

## Leader 选举

### 选举触发条件

1. Leader 所在 Broker 故障
2. 手动触发重选举
3. 优先副本选举

### 选举流程

```
1. Controller 检测到 Leader 故障
2. Controller 从 ISR 中选择一个新的 Leader
3. Controller 更新 ZooKeeper 中的 Leader 信息
4. Controller 通知所有 Broker 更新元数据
5. 新 Leader 开始处理请求
```

### 选举策略

```properties
# 不干净的 Leader 选举（允许非 ISR 副本成为 Leader）
unclean.leader.election.enable=false    # 默认禁用
```

- **true**：允许任何副本成为 Leader（可能丢失数据）
- **false**：只允许 ISR 中的副本成为 Leader（保证数据不丢失，但可能不可用）

### 优先副本选举

优先副本是 AR（Assigned Replica）中的第一个副本。

```properties
# 自动触发优先副本选举的时间间隔
auto.leader.rebalance.enable=true
leader.imbalance.check.interval.seconds=300
leader.imbalance.per.broker.percentage=10
```

```bash
# 手动触发优先副本选举
bin/kafka-leader-election.sh \n  --bootstrap-server localhost:9092 \n  --election-type preferred \n  --topic my-topic
```

## 副本故障恢复

### Follower 故障恢复

```
1. Follower 故障，被移出 ISR
2. Follower 恢复
3. Follower 读取本地 HW
4. Follower 将日志截断到 HW 位置
5. Follower 从 Leader 拉取数据
6. Follower 追上 Leader 后，重新加入 ISR
```

### Leader 故障恢复

```
1. Leader 故障
2. Controller 从 ISR 中选举新 Leader
3. 新 Leader 更新 HW = 新 Leader 的 LEO
4. Follower 从新 Leader 拉取数据
5. Follower 截断日志到新 HW 位置
6. 同步完成后，ISR 恢复
```

## Epoch 机制

### Controller Epoch

标识 Controller 的代际，每次 Controller 选举后递增。

### Leader Epoch

标识 Partition Leader 的代际，每次 Leader 选举后递增。

```
Leader Epoch = (epoch, startOffset)

作用：
1. 防止 Follower 从旧 Leader 拉取数据
2. 确定 Follower 需要截断的位置
3. 帮助 Follower 恢复正确状态
```

### Leader Epoch 文件

每个 Partition 有一个 `leader-epoch-checkpoint` 文件：

```
# 格式：epoch startOffset
0 0
1 100
2 500
```

## 副本配置详解

```properties
# 默认副本数
default.replication.factor=3

# 最小同步副本数
min.insync.replicas=2

# Follower 落后 Leader 的最大时间
replica.lag.time.max.ms=30000

# 副本拉取间隔
replica.fetch.backoff.ms=1000
replica.fetch.min.bytes=1
replica.fetch.max.bytes=1048576
replica.fetch.wait.max.ms=500

# 副本数量限制
num.replica.fetchers=1

# 不干净的 Leader 选举
unclean.leader.election.enable=false

# 优先副本选举
auto.leader.rebalance.enable=true
leader.imbalance.check.interval.seconds=300
leader.imbalance.per.broker.percentage=10
```

## 副本监控指标

| 指标 | 描述 | 建议阈值 |
|------|------|----------|
| UnderReplicatedPartitions | 副本数不足的 Partition 数量 | 0 |
| OfflinePartitionsCount | 离线的 Partition 数量 | 0 |
| UnderMinIsrPartitionCount | ISR 副本数小于 min.insync.replicas 的 Partition 数量 | 0 |
| LeaderCount | Leader 副本数量 | - |
| PartitionCount | Partition 数量 | - |
| IsrCount | ISR 平均大小 | - |

## 最佳实践

### 1. 副本数量

- **生产环境**：至少 3 个副本
- **测试环境**：1-2 个副本

### 2. min.insync.replicas

```properties
min.insync.replicas=2

# 配合生产者 acks=all 使用
# 保证至少 2 个副本写入成功
```

### 3. 副本分布

- 跨机架分布，避免机架故障
- 跨可用区分布，提高容灾能力

### 4. 监控告警

```
- 监控 UnderReplicatedPartitions，及时处理副本同步问题
- 监控 IsrShrinks，及时处理 ISR 收缩
- 监控 Leader 均衡度，避免单个 Broker 负载过高
```
