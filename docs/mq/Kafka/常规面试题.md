# Kafka 常规面试题

## 一、基础概念

### 1. Kafka 是什么？有哪些特点？

**答案**：

Apache Kafka 是一个分布式流处理平台，具有以下特点：

1. **高吞吐量**：单机每秒可处理百万级消息
2. **低延迟**：消息延迟可控制在毫秒级
3. **高可用性**：支持副本机制，依赖 ZooKeeper/KRaft 实现集群协调
4. **可扩展性**：支持在线扩容，数据分片存储
5. **持久化**：消息持久化到磁盘，支持消息回溯
6. **支持多种客户端**：提供 Java、Python、Go、C++ 等多种语言客户端

### 2. Kafka 与传统消息队列的区别是什么？

**答案**：

| 特性 | Kafka | RabbitMQ | ActiveMQ |
|------|-------|----------|----------|
| 吞吐量 | 极高 | 中等 | 低 |
| 延迟 | 低 | 低 | 低 |
| 消息回溯 | 支持 | 不支持 | 不支持 |
| 消息堆积 | 支持 | 受内存限制 | 受内存限制 |
| 协议 | 自定义 TCP | AMQP | JMS |
| 消息确认 | Ack | Ack | Ack |

Kafka 的核心区别：
- 基于日志的消息存储，天然支持消息回溯
- 分布式架构，水平扩展能力强
- 批量发送，吞吐量极高

### 3. Kafka 的核心组件有哪些？

**答案**：

1. **Broker**：Kafka 集群中的服务器节点
2. **Topic**：消息的逻辑分类
3. **Partition**：Topic 的物理分片
4. **Producer**：消息生产者
5. **Consumer**：消息消费者
6. **Consumer Group**：消费者组，实现负载均衡
7. **ZooKeeper/KRaft**：集群协调服务

### 4. 什么是 Topic 和 Partition？它们的关系是什么？

**答案**：

**Topic**：
- Kafka 中消息的逻辑分类
- 类似于数据库中的表
- 可以有多个 Partition

**Partition**：
- Topic 的物理分片
- 每个 Partition 是一个有序的、不可变的消息序列
- 每个 Partition 都有唯一的 offset
- 支持消息的并行处理和负载均衡

**关系**：
- 一个 Topic 可以有多个 Partition
- 消息根据分区策略分配到不同的 Partition
- 每个 Partition 可以有多个副本

### 5. 什么是 Offset？有哪些类型？

**答案**：

**Offset** 是 Partition 中每条消息的唯一标识，从 0 开始递增。

**Offset 类型**：

1. **Log End Offset (LEO)**：日志末端偏移量，表示下一条待写入消息的 offset
2. **High Watermark (HW)**：高水位，表示 ISR 中所有副本都已同步的消息 offset
3. **Consumer Offset**：消费者消费到的 offset

**消费范围**：
- `auto.offset.reset=latest`：从 LEO 开始消费
- `auto.offset.reset=earliest`：从 0 开始消费
- `auto.offset.reset=none`：抛出异常

### 6. 什么是 Consumer Group？有什么作用？

**答案**：

**Consumer Group** 是一组消费者的集合，用于实现负载均衡。

**作用**：
1. **负载均衡**：均匀分配 Partition 给组内的消费者
2. **消息不重复**：同一条消息只会被组内的一个消费者消费
3. **消费独立**：不同消费者组可以独立消费相同的消息
4. **容错性**：消费者故障时，Partition 重新分配

**规则**：
- 组内消费者数量不能超过 Partition 数量
- 超过 Partition 数量的消费者会空闲
- 消费者加入或离开组时，触发 Rebalance

### 7. 什么是 Rebalance？什么时候触发？

**答案**：

**Rebalance** 是指消费者组中的消费者数量变化时，Partition 重新分配的过程。

**触发条件**：
1. 消费者加入或离开组
2. 消费者会话超时
3. 订阅的 Topic 或 Partition 数量变化
4. 手动触发 Rebalance

**影响**：
- Rebalance 期间，消费者停止消费
- 可能导致消费延迟
- 频繁 Rebalance 会影响性能

### 8. Kafka 的副本机制是怎样的？

**答案**：

Kafka 通过副本机制保证数据可靠性和高可用性。

**副本类型**：
- **Leader**：主副本，处理所有读写请求
- **Follower**：从副本，从 Leader 同步数据

**ISR (In-Sync Replica)**：
- 与 Leader 保持同步的副本集合
- 只有 ISR 中的副本才有资格成为新的 Leader
- ISR 动态变化，落后太多的 Follower 会被移出 ISR

**副本同步**：
- Follower 主动从 Leader 拉取数据
- Leader 维护 Follower 的 LEO 和 HW
- 消费者只能消费到 HW 之前的消息

**答案**：

```java

for (ConsumerRecord<String, String> record : records) {
    process(record);

consumer.commitSync();
```

### 10. Kafka 如何保证消息不重复？

**答案**：

**生产端 - 幂等性**：
```java
// 启用幂等性，保证同一消息不会重复写入
props.put("enable.idempotence", "true");
```

**消费端 - 手动处理**：
1. 使用唯一 ID 去重
2. 使用数据库唯一约束
3. 使用 Redis Set 记录已处理的 ID

```java
// 使用 Redis 去重
public void processMessage(String messageId, String content) {
    // 检查是否已处理
    if (redis.sismember("processed_messages", messageId)) {
        return;
    }

    // 处理消息
    handleMessage(content);

    // 标记已处理
    redis.sadd("processed_messages", messageId);
}
```

### 11. Kafka 的消息顺序性如何保证？

**答案**：

**Partition 内有序**：
- 同一个 Partition 内的消息按顺序存储
- 消费者按顺序消费

**Partition 间无序**：
- 不同 Partition 之间的消息没有顺序关系

**保证业务顺序性的方法**：

```java
// 1. 使用相同的 Key，保证消息发送到同一个 Partition
producer.send(new ProducerRecord<>("topic", "user-123", "message-1"));
producer.send(new ProducerRecord<>("topic", "user-123", "message-2"));
producer.send(new ProducerRecord<>("topic", "user-123", "message-3"));

// 2. 自定义分区器
public class OrderPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        // 根据业务 Key 选择 Partition
        return Math.abs(key.hashCode()) % numPartitions;
    }
}
```

### 12. Kafka 的消息压缩机制是怎样的？

**答案**：

Kafka 支持多种压缩算法：

| 压缩类型 | 压缩比 | 压缩速度 | 适用场景 |
|---------|--------|----------|----------|
| none | 1:1 | 最快 | 网络带宽充足 |
| gzip | 最高 | 慢 | 存储/带宽受限 |
| snappy | 中等 | 快 | 实时性要求高 |
| lz4 | 中等 | 很快 | 实时性要求高 |
| zstd | 高 | 中等 | 平衡性能和压缩比 |

**配置**：
```java
// 启用压缩
props.put("compression.type", "lz4");
```

**压缩时机**：
- 生产端：批量压缩
- Broker 端：根据配置决定是否压缩存储
- 消费端：自动解压

### 14. Kafka 如何实现高性能？

**答案**：

1. **顺序写入**：消息追加到日志末尾，顺序写入速度可达 600MB/s
2. **零拷贝**：使用 `sendfile` 系统调用，减少数据拷贝次数
3. **批量发送**：生产者批量发送消息，Broker 批量拉取消息
4. **内存池**：使用 ByteBuffer 池，避免频繁的内存分配和 GC
5. **压缩**：减少网络传输和磁盘存储开销
6. **多线程**：网络线程和 IO 线程分离
7. **页缓存**：依赖操作系统的页缓存，减少内存使用

### 15. Kafka 的日志段（Log Segment）是如何工作的？

**答案**：

**日志段组成**：
- `.log` 文件：实际存储消息的文件
- `.index` 文件：偏移量索引文件
- `.timeindex` 文件：时间戳索引文件

**日志段命名**：
- 以起始 offset 命名
- 例如：`00000000000000000000.log` 表示从 offset=0 开始

**日志段滚动**：
```properties
# 基于大小滚动
log.segment.bytes=1073741824   # 1GB

# 基于时间滚动
log.roll.hours=168             # 7天
```

**索引查找流程**：
1. 根据 offset 二分查找 .index 文件
2. 找到最近的索引项
3. 从索引项记录的 position 开始顺序扫描 .log 文件
4. 找到目标 offset 的消息

### 16. Kafka 的 ISR 是什么？有什么作用？

**答案**：

**ISR (In-Sync Replica)**：
- 与 Leader 保持同步的副本集合
- 包含 Leader 和所有同步的 Follower

**作用**：
1. **数据一致性**：ISR 中的副本数据一致
2. **Leader 选举**：只有 ISR 中的副本才有资格成为新的 Leader
3. **消息确认**：acks=all 时，等待 ISR 中所有副本确认

**ISR 判断标准**：
```properties
# Follower 落后 Leader 的最大时间
replica.lag.time.max.ms=30000     # 默认 30 秒
```

如果一个 Follower 在 30 秒内没有向 Leader 发送拉取请求，就会被移出 ISR。

### 17. Kafka 的 HW 和 LEO 是什么？

**答案**：

**LEO (Log End Offset)**：
- 日志末端偏移量
- 表示下一条待写入消息的 offset
- 每个 Partition 的每个副本都有自己的 LEO

**HW (High Watermark)**：
- 高水位
- 表示 ISR 中所有副本都已同步的消息 offset
- HW = min(所有 ISR 副本的 LEO)
- 消费者只能消费到 HW 之前的消息

**示例**：
```
Leader LEO = 10
Follower1 LEO = 8
Follower2 LEO = 9

HW = min(10, 8, 9) = 8

消费者只能消费到 offset < 8 的消息
```

### 18. Kafka 的 Leader 选举是怎样的？

**答案**：

**选举触发条件**：
1. Leader 所在 Broker 故障
2. 手动触发重选举
3. 优先副本选举

**选举流程**：
1. Controller 检测到 Leader 故障
2. Controller 从 ISR 中选择一个新的 Leader
3. Controller 更新元数据
4. Controller 通知所有 Broker 更新元数据
5. 新 Leader 开始处理请求

**选举策略**：
```properties
# 不干净的 Leader 选举（允许非 ISR 副本成为 Leader）
unclean.leader.election.enable=false    # 默认禁用
```

- `false`：只允许 ISR 中的副本成为 Leader（保证数据不丢失）
- `true`：允许任何副本成为 Leader（可能丢失数据，但可用性更高）

### 19. Kafka 如何实现 Exactly-Once 语义？

### 20. Kafka 的 Zero Copy 是什么？

**答案**：

**Zero Copy** 是指数据直接在内核空间传输，避免在用户空间和内核空间之间拷贝。

**传统方式（4次拷贝，4次上下文切换）**：
```
磁盘 → 内核缓冲区 → 用户缓冲区 → 内核Socket缓冲区 → 网卡
```

**Zero Copy（2次拷贝，2次上下文切换）**：
```
磁盘 → 内核缓冲区 → 网卡
```

**Kafka 实现**：
```java
// 使用 FileChannel.transferTo 实现零拷贝
FileChannel fileChannel = new RandomAccessFile(file, "r").getChannel();
long transferred = fileChannel.transferTo(position, count, socketChannel);
```

**优势**：
- 减少数据拷贝次数
- 减少上下文切换
- 提高 CPU 利用率
- 降低延迟

### 21. Kafka 如何处理消息积压？

**答案**：

**排查原因**：
1. 消费者处理速度慢
2. 消费者故障
3. 网络问题
4. Partition 分布不均

**解决方案**：

1. **增加消费者数量**：
```java
// 增加消费者，但不超过 Partition 数量
ExecutorService executor = Executors.newFixedThreadPool(10);

for (int i = 0; i < 10; i++) {
    executor.submit(() -> {
        KafkaConsumer<String, String> consumer = createConsumer();
        consumer.subscribe(Arrays.asList("my-topic"));
        while (true) {
            ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
            processRecords(records);
            consumer.commitAsync();
        }
    });
}
```

2. **增加消费者线程**：
```java
// 单消费者多线程处理
KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
ExecutorService executor = Executors.newFixedThreadPool(10);

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        executor.submit(() -> process(record));
    }
    consumer.commitAsync();
}
```

3. **增加 Partition 数量**：
```bash
bin/kafka-topics.sh --alter \
  --bootstrap-server localhost:9092 \
  --topic my-topic \
  --partitions 12
```

4. **优化消费者处理逻辑**：
- 减少数据库操作
- 使用批量处理
- 异步处理

### 24. Kafka 如何进行容量规划？

**答案**：

**计算公式**：

1. **消息数量**：
```
每日消息量 = 每秒消息量 × 86400
```

2. **消息大小**：
```
每日消息大小 = 每日消息量 × 平均消息大小
```

3. **磁盘空间**：
```
磁盘空间 = 每日消息大小 × 保留天数 × 副本数 / 压缩比
```

4. **Partition 数量**：
```
Partition 数量 = 消费者数量 × 单个消费者处理能力
```

**示例**：
```
假设：
- 每秒消息量：10,000 条
- 平均消息大小：1 KB
- 保留天数：7 天
- 副本数：3
- 压缩比：0.5

计算：
- 每日消息量 = 10,000 × 86400 = 864,000,000 条
- 每日消息大小 = 864,000,000 × 1 KB = 864 GB
- 磁盘空间 = 864 × 7 × 3 / 0.5 = 3.6 TB
```
