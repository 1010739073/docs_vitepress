# Kafka 存储机制

## 存储架构

Kafka 的消息存储采用**分区日志**结构，每个 Partition 对应一个目录，包含多个日志段文件。

```
kafka-logs/
├── topic-0/
│   ├── 00000000000000000000.log
│   ├── 00000000000000000000.index
│   ├── 00000000000000000000.timeindex
│   ├── 00000000000000000001.log
│   ├── 00000000000000000001.index
│   └── 00000000000000000001.timeindex
└── topic-1/
    ├── 00000000000000000000.log
    ├── 00000000000000000000.index
    └── 00000000000000000000.timeindex
```

## 日志段（Log Segment）

### 日志段概念

每个 Partition 被划分为多个日志段，每个日志段由以下文件组成：

1. **.log 文件** - 实际存储消息的文件
2. **.index 文件** - 偏移量索引文件
3. **.timeindex 文件** - 时间戳索引文件

### 日志段命名规则

日志段文件以**起始偏移量**命名，例如：
- `00000000000000000000.log` - 从 offset=0 开始
- `00000000000000000001.log` - 从 offset=1 开始
- `00000000000000005000.log` - 从 offset=5000 开始

### 活跃日志段

只有一个日志段是活跃的（正在写入），其他日志段是已关闭的（只读）。

## 消息格式

### 消息结构

```
┌─────────────────────────────────────────────────────────────┐
│                    Record Batch                              │
├─────────────────────────────────────────────────────────────┤
│ base offset (8B) │ length (4B) │ partition leader epoch (4B) │
├─────────────────────────────────────────────────────────────┤
│ magic (1B) │ crc (4B) │ attributes (2B) │ last offset delta (4B) │
├─────────────────────────────────────────────────────────────┤
│ base timestamp (8B) │ max timestamp (8B) │ producer id (8B) │
├─────────────────────────────────────────────────────────────┤
│ producer epoch (2B) │ base sequence (4B) │ records count (4B) │
├─────────────────────────────────────────────────────────────┤
│                      Record 1                                │
│  length │ attributes │ timestamp delta │ offset delta │
│  key length │ key │ value length │ value │ headers    │
├─────────────────────────────────────────────────────────────┤
│                      Record 2                                │
│  ...                                                         │
└─────────────────────────────────────────────────────────────┘
```

### 消息压缩

Kafka 支持多种压缩算法：

| 压缩类型 | 压缩比 | 压缩速度 | 适用场景 |
|---------|--------|----------|----------|
| none | 1:1 | 最快 | 网络带宽充足 |
| gzip | 最高 | 慢 | 存储/带宽受限 |
| snappy | 中等 | 快 | 实时性要求高 |
| lz4 | 中等 | 很快 | 实时性要求高 |
| zstd | 高 | 中等 | 平衡性能和压缩比 |

## 索引机制

### 偏移量索引（.index）

偏移量索引将**相对偏移量**映射到**物理位置**。

```
索引文件结构：
┌───────────────┬───────────────┐
│ relative offset │  position    │
├───────────────┼───────────────┤
│      0         │      0       │
│      20        │     1024     │
│      40        │     2048     │
│      60        │     3072     │
│      ...       │     ...      │
└───────────────┴───────────────┘
```

**稀疏索引**：并非每条消息都有索引项，而是每隔一定字节数（`log.index.interval.bytes`）创建一个索引项。

### 时间戳索引（.timeindex）

时间戳索引将**时间戳**映射到**相对偏移量**。

```
时间戳索引文件结构：
┌───────────────┬───────────────┐
│  timestamp    │ relative offset │
├───────────────┼───────────────┤
│  1609459200000 │      0        │
│  1609459201000 │     20        │
│  1609459202000 │     40        │
│  ...          │     ...        │
└───────────────┴───────────────┘
```

### 索引查找流程

```
1. 根据 offset 二分查找 .index 文件，找到最近的索引项
2. 从索引项记录的 position 开始顺序扫描 .log 文件
3. 找到目标 offset 的消息
```

## 日志清理策略

### 1. 基于时间清理（log.retention.hours）

```properties
# 消息保留时间
log.retention.hours=168        # 7天
log.retention.minutes=10080
log.retention.ms=604800000
```

### 2. 基于大小清理（log.retention.bytes）

```properties
# 每个日志段的最大大小
log.segment.bytes=1073741824   # 1GB

# 每个 Partition 的最大大小
log.retention.bytes=-1         # 无限制
```

### 3. 基于日志压缩（log.cleanup.policy=compact）

保留每个 Key 的最新值，适用于**日志压缩**场景。

```properties
# 启用日志压缩
log.cleanup.policy=compact

# 压缩检查间隔
log.cleaner.enable=true
log.cleaner.dedupe.buffer.size=134217728
log.cleaner.threads=1
log.cleaner.min.cleanable.ratio=0.5
```

## 日志压缩原理

日志压缩保留每个 Key 的最新值：

```
压缩前：
Key1: Value1 (offset=0)
Key2: Value2 (offset=1)
Key1: Value3 (offset=2)  ← Key1 的更新
Key3: Value4 (offset=3)
Key2: Value5 (offset=4)  ← Key2 的更新

压缩后：
Key1: Value3 (offset=2)
Key3: Value4 (offset=3)
Key2: Value5 (offset=4)
```

### 日志压缩流程

```
1. Cleaner 线程选择需要清理的日志段
2. 读取日志段，构建 Key → 最新 Offset 的映射
3. 遍历日志段，保留每个 Key 的最新值
4. 将清理后的数据写入新日志段
5. 原子性替换旧日志段
```

## 文件系统优化

### 1. 顺序写入

Kafka 采用**追加写入**方式，将消息追加到日志末尾。

**性能优势**：
- 顺序写入速度可达 600MB/s（机械硬盘）
- 随机写入速度仅 100KB/s
- 差距约 6000 倍

### 2. 零拷贝（Zero Copy）

Kafka 使用 `sendfile` 系统调用实现零拷贝传输：

```
传统方式（4次拷贝，4次上下文切换）：
磁盘 → 内核缓冲区 → 用户缓冲区 → 内核socket缓冲区 → 网卡

零拷贝（2次拷贝，2次上下文切换）：
磁盘 → 内核缓冲区 → 网卡
```

```java
// Kafka 使用 FileChannel.transferTo 实现零拷贝
FileChannel fileChannel = new RandomAccessFile(file, "r").getChannel();
long transferred = fileChannel.transferTo(position, count, socketChannel);
```

### 3. 页缓存（Page Cache）

Kafka 不在 JVM 堆中缓存消息，而是依赖操作系统的页缓存：

```properties
# 日志刷盘策略
log.flush.interval.messages=10000   # 每多少条消息刷盘
log.flush.interval.ms=1000          # 每多少毫秒刷盘
```

## 存储配置详解

```properties
# 日志目录
log.dirs=/tmp/kafka-logs

# 日志段大小
log.segment.bytes=1073741824       # 1GB

# 日志段滚动时间
log.roll.hours=168                 # 7天

# 索引间隔
log.index.interval.bytes=4096      # 4KB

# 索引最大大小
log.index.size.max.bytes=10485760  # 10MB

# 消息保留时间
log.retention.hours=168            # 7天

# 消息保留大小
log.retention.bytes=-1             # 无限制

# 日志清理策略
log.cleanup.policy=delete          # delete 或 compact

# 日志压缩配置
log.cleaner.enable=true
log.cleaner.dedupe.buffer.size=134217728
log.cleaner.threads=1
log.cleaner.min.cleanable.ratio=0.5

# 文件描述符配置
# 每个日志段至少需要2个文件描述符（log + index）
# 建议设置：max open files = partitions * (segments per partition) * 2 + 1000
```

## 存储监控指标

| 指标 | 描述 | 建议阈值 |
|------|------|----------|
| LogEndOffset | 日志末端偏移量 | - |
| LogStartOffset | 日志起始偏移量 | - |
| Size | Partition 大小 | < 磁盘容量的 80% |
| LogFlushRateAndTimeMs | 刷盘速率和时间 | < 100ms |
| LogCleanerBytes | 清理的字节数 | - |
| LogCleanerThreads | 清理线程数 | - |
