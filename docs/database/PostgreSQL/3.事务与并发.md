# PostgreSQL 事务与并发控制

## 一、事务基础

### 1.1 ACID 特性

| 特性 | 说明 | PostgreSQL 实现 |
|-----|------|----------------|
| **原子性（Atomicity）** | 事务要么全部成功，要么全部失败 | 回滚段 |
| **一致性（Consistency）** | 事务执行前后数据库保持一致性 | 约束、触发器 |
| **隔离性（Isolation）** | 并发事务相互隔离 | MVCC |
| **持久性（Durability）** | 提交的事务永久保存 | WAL（预写日志） |

### 1.2 事务控制

```sql
-- 开始事务
BEGIN;
-- 或
START TRANSACTION;

-- 提交事务
COMMIT;

-- 回滚事务
ROLLBACK;

-- 设置保存点
SAVEPOINT my_savepoint;

-- 回滚到保存点
ROLLBACK TO SAVEPOINT my_savepoint;

-- 释放保存点
RELEASE SAVEPOINT my_savepoint;

-- 设置事务特性
BEGIN ISOLATION LEVEL READ COMMITTED READ WRITE;
BEGIN ISOLATION LEVEL SERIALIZABLE READ ONLY DEFERRABLE;
```

### 1.3 完整事务示例

```sql
-- 带保存点的事务
BEGIN;

-- 插入用户
INSERT INTO users (username, email) VALUES ('张三', 'zhangsan@example.com');

SAVEPOINT after_user_insert;

-- 插入订单
INSERT INTO orders (user_id, total_amount) VALUES (currval('users_id_seq'), 100.00);

-- 如果订单插入失败，回滚到保存点
-- ROLLBACK TO SAVEPOINT after_user_insert;

-- 提交事务
COMMIT;

-- 带错误处理的事务
DO $$
BEGIN
    -- 插入用户
    INSERT INTO users (username, email) VALUES ('李四', 'lisi@example.com');
    
    -- 插入订单
    INSERT INTO orders (user_id, total_amount) VALUES (currval('users_id_seq'), 200.00);
    
    -- 更新统计
    UPDATE user_stats SET order_count = order_count + 1 WHERE user_id = currval('users_id_seq');
    
EXCEPTION WHEN OTHERS THEN
    -- 发生错误时回滚
    RAISE NOTICE 'Error: %', SQLERRM;
    ROLLBACK;
    RETURN;
END;
$$;
```

## 二、隔离级别

### 2.1 隔离级别对比

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|---------|------|-----------|------|---------|
| **READ UNCOMMITTED** | 不允许 | 可能 | 可能 | 同 READ COMMITTED |
| **READ COMMITTED** | 不允许 | 可能 | 可能 | MVCC（默认） |
| **REPEATABLE READ** | 不允许 | 不允许 | 可能（PG 中不允许） | MVCC |
| **SERIALIZABLE** | 不允许 | 不允许 | 不允许 | 可串行化快照 |

### 2.2 设置隔离级别

```sql
-- 设置会话隔离级别
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 设置事务隔离级别
BEGIN ISOLATION LEVEL READ COMMITTED;
BEGIN ISOLATION LEVEL REPEATABLE READ;
BEGIN ISOLATION LEVEL SERIALIZABLE;

-- 查看当前隔离级别
SHOW transaction_isolation;
```

### 2.3 各隔离级别详解

#### READ COMMITTED（读已提交）

```sql
-- 默认隔离级别
-- 事务中的每个查询看到的数据是查询开始时的快照

-- 会话 A
BEGIN;
SELECT * FROM users WHERE id = 1;  -- 看到 age = 25

-- 会话 B
BEGIN;
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;

-- 会话 A（同一事务中再次查询）
SELECT * FROM users WHERE id = 1;  -- 看到 age = 30（不可重复读）
COMMIT;
```

#### REPEATABLE READ（可重复读）

```sql
-- 事务开始时的快照在整个事务期间保持不变

BEGIN ISOLATION LEVEL REPEATABLE READ;

SELECT * FROM users WHERE id = 1;  -- 看到 age = 25

-- 其他事务更新并提交

SELECT * FROM users WHERE id = 1;  -- 仍然看到 age = 25（可重复读）

-- 尝试更新
UPDATE users SET age = 35 WHERE id = 1;  -- 等待或报错（取决于并发情况）

COMMIT;
```

#### SERIALIZABLE（可串行化）

```sql
-- 最强的隔离级别，保证事务串行执行的效果

BEGIN ISOLATION LEVEL SERIALIZABLE;

SELECT * FROM users WHERE age > 25;

-- 其他事务插入 age = 30 的记录

SELECT * FROM users WHERE age > 25;  -- 看不到新记录（幻读防止）

-- 尝试插入
INSERT INTO users (username, age) VALUES ('新用户', 30);
-- 可能报错：could not serialize access due to read/write dependencies

COMMIT;
```

## 三、MVCC（多版本并发控制）

### 3.1 先理解问题：为什么需要 MVCC？

假设你正在写一本小说，同时有多个读者在读你的书。你还没有写完，但读者们已经能读到最新的章节了。

**问题来了**：如果你在读者 A 读第 5 章的时候，修改了第 5 章的内容，读者 A 应该看到修改前的还是修改后的？如果读者 A 看到一半突然变了，体验会很差。

数据库也面临同样的问题。当用户 A 正在查询数据时，用户 B 同时修改了同一条数据——用户 A 应该看到修改前还是修改后的？

**MVCC 的答案**：谁也别等谁，各看各的！它给每个用户一个「数据快照」，就像拍照一样，拍完之后你改你的，我看我的，互不干扰。

### 3.2 核心思想：一行数据，多个版本

普通数据库里，一条记录只有一份，更新时直接覆盖。但 PostgreSQL 不一样——它**从不原地修改数据**。每次 UPDATE，它会在表里插入一条**新版本**的数据，同时把旧版本标记为"过期"。

```
一条数据的生命线：

  INSERT                      UPDATE                      DELETE
    ↓                           ↓                           ↓
  [版本1]  ────────→  [版本1(过期)] + [版本2]  ────────→  [版本1(过期)] + [版本2(过期)]
    ↑                           ↑                           ↑
  当前有效                   当前有效                     全部过期，等待清理
```

**关键好处**：
- 读操作不用等写操作（读取旧版本即可）
- 写操作不用等读操作（写入新版本即可）
- 读写互不阻塞，并发性能极高

### 3.3 PostgreSQL 怎么做到的？——两个隐藏字段

PostgreSQL 的每一行数据，除了你定义的字段外，还藏着两个重要字段：

| 字段 | 含义 | 通俗理解 |
|-----|------|---------|
| **xmin** | 这行数据是哪个事务创建的 | "出生证明"——记录这条数据"出生"时的事务编号 |
| **xmax** | 这行数据是哪个事务删除的 | "死亡证明"——记录这条数据"死亡"时的事务编号（0 表示还活着） |

> 提醒：这两个字段叫"系统列"，默认查询 `SELECT *` 不会显示，要显式指定才能看到。

```sql
-- 查看一行数据的 xmin 和 xmax
SELECT xmin, xmax, * FROM users WHERE id = 1;
--  xmin | xmax | id |  name  | age
-- ------+------+----+--------+-----
--   100 |    0 |  1 | 张三   |  25
--  ↑        ↑
-- 出生在    还没死
-- 事务100
```

### 3.4 一步步看 MVCC 怎么工作

我们来模拟一个完整的场景，从无到有，一步步看数据怎么变化。

#### 场景：用户表 `users`，初始为空

```sql
CREATE TABLE users (id SERIAL PRIMARY KEY, name TEXT, age INT);
```

#### 第 1 步：INSERT——让数据"出生"

```
事务 100：INSERT INTO users (name, age) VALUES ('张三', 25);
```

执行后，表里多了一行：

```
| xmin | xmax | id | name | age |
|------|------|----|------|-----|
| 100  | 0    | 1  | 张三 | 25  |
```

**解读**：这条数据"出生"在事务 100，还没有"死亡"（xmax=0），所以它是**活着的**。

---

#### 第 2 步：UPDATE——旧版本"死亡"，新版本"出生"

```
事务 101：UPDATE users SET age = 26 WHERE name = '张三';
```

> 注意：PostgreSQL 不会把 age=25 改成 age=26。它做的是：
> 1. 把旧版本标记为"死亡"（xmax = 101）
> 2. 插入一条新版本（xmin = 101）

执行后，表里变成了**两行**：

```
| xmin | xmax | id | name | age |  ← 旧版本
|------|------|----|------|-----|
| 100  | 101  | 1  | 张三 | 25  |  ← 在事务 101 里"死亡"了

| xmin | xmax | id | name | age |  ← 新版本
|------|------|----|------|-----|
| 101  | 0    | 1  | 张三 | 26  |  ← 在事务 101 里"出生"，还活着
```

**关键理解**：同一行数据在表里存在两个版本！旧版本没有被删除，只是被标记了。

---

#### 第 3 步：又有事务在查询——它看到什么？

```
事务 102：SELECT * FROM users WHERE name = '张三';
```

事务 102 是怎么判断该看哪个版本的？规则很简单：

**一条数据对事务 T 可见，当且仅当：**
- 它的 xmin 是事务 T 开始前**已经提交的**事务，并且
- 它的 xmax 是 0，或者 xmax 对应的事务在 T 开始时**还没提交**（或已回滚）

换句话说：**"出生时我还在，死亡时我还没走"**。

对于事务 102：
- 旧版本（xmin=100, xmax=101）：事务 100 和 101 都在 102 之前提交了，所以 xmax=101 已提交 → 旧版本对 102 **不可见**（因为它已经"死"了）
- 新版本（xmin=101, xmax=0）：事务 101 已提交，xmax=0 → 新版本对 102 **可见**

所以事务 102 看到的是 `张三, age=26`。

---

#### 第 4 步：DELETE——只是标记"死亡"

```
事务 103：DELETE FROM users WHERE name = '张三';
```

执行后：

```
| xmin | xmax | id | name | age |
|------|------|----|------|-----|
| 100  | 101  | 1  | 张三 | 25  |  ← 早已死亡
| 101  | 103  | 1  | 张三 | 26  |  ← 刚被事务 103 标记为死亡
```

DELETE 没有真正删除数据，只是把 xmax 填上了事务 103 的编号。

---

#### 第 5 步：这些"尸体"怎么办？——VACUUM

随着时间推移，表里堆积了大量"死亡"数据（也叫**死元组**）。虽然它们对任何事务都不可见，但占着磁盘空间，拖慢查询。

这时候就需要 VACUUM 来"收尸"：

```sql
-- 手动清理
VACUUM users;
```

VACUUM 做的事：
1. 找到所有 xmax 已提交且对任何活跃事务都不可见的死元组
2. 回收它们占用的空间，标记为"可重用"
3. 更新统计信息，帮助查询优化器做更好的决策

> **重要**：普通 VACUUM 只是把空间标记为可重用，**不会把磁盘空间还给操作系统**。如果表膨胀严重，需要 `VACUUM FULL`（但会锁表！）。

### 3.5 并发场景实战：三个事务同时操作

这才是 MVCC 真正发光的地方。假设有三个事务**同时**操作同一条数据：

```
时间线：  t1 → → → → → t2 → → → → → t3 → → → → → t4
         │              │              │              │
事务 A   │ BEGIN        │ SELECT age   │              │ COMMIT
         │              │ 看到 age=25  │              │
事务 B   │              │ BEGIN        │ UPDATE       │
         │              │              │ age=26       │ COMMIT
事务 C   │              │              │              │ BEGIN
         │              │              │              │ SELECT age
         │              │              │              │ 看到 age=26
```

| 时刻 | 发生了什么 |
|------|-----------|
| t1 | 事务 A 开始，拍了一张"快照" |
| t2 | 事务 A 查询 age=25（这是它快照里的数据） |
| t2 | 事务 B 开始 |
| t3 | 事务 B 把 age 改成 26，提交 |
| t4 | 事务 A 再次查询，**仍然看到 age=25**（因为它用的是 t1 的快照） |
| t4 | 事务 C 开始，查询 age=26（事务 B 已提交，C 的快照包含它） |

**关键点**：事务 A 和事务 C 同时看到**不同的数据版本**，但它们都没有等待对方，也没有读到脏数据。这就是 MVCC 的威力。

### 3.6 事务 ID 环绕问题

PostgreSQL 用 32 位整数作为事务 ID，大约 40 亿个。用完了怎么办？它会**回绕**（wraparound），就像汽车里程表到了 999999 又回到 000000。

但这就引出一个问题：如果一个事务 ID 回绕了，谁算"老"事务、谁算"新"事务？PostgreSQL 用了一个巧妙的"循环比较"算法来解决它，但前提是**必须定期 VACUUM**，否则回绕时会把旧数据当成新数据，导致数据丢失。

这也是为什么 PostgreSQL 的 autovacuum（自动清理）是**强制开启**且不能长期关闭的。

### 3.7 VACUUM 详解

```sql
-- 普通 VACUUM：标记死元组空间可重用，不锁表
VACUUM users;

-- VACUUM ANALYZE：清理 + 更新统计信息（推荐日常使用）
VACUUM ANALYZE users;

-- VACUUM FULL：物理回收空间，会锁表！谨慎使用
VACUUM FULL users;

-- 查看表里有多少"尸体"
SELECT 
    schemaname,
    tablename,
    n_live_tup   AS 活着的行数,
    n_dead_tup   AS 死了的行数,
    round(n_dead_tup::numeric / nullif(n_live_tup + n_dead_tup, 0) * 100, 2) AS 死元组比例
FROM pg_stat_user_tables
WHERE tablename = 'users';
-- 如果死元组比例超过 20%，说明该表需要 VACUUM

-- 查看 VACUUM 历史
SELECT 
    schemaname,
    tablename,
    last_vacuum       AS 上次手动清理,
    last_autovacuum   AS 上次自动清理,
    last_analyze      AS 上次手动分析,
    last_autoanalyze  AS 上次自动分析,
    vacuum_count      AS 手动清理次数,
    autovacuum_count  AS 自动清理次数
FROM pg_stat_user_tables
WHERE tablename = 'users';
```

### 3.8 查看系统列

```sql
-- 查看隐藏字段（xmin/xmax 是每行都有的）
SELECT xmin, xmax, * FROM users WHERE id = 1;

-- 查看当前事务的 ID
SELECT txid_current();

-- 查看当前快照信息（可见事务范围）
SELECT txid_current_snapshot();
-- 结果类似：100:100:  表示 xmin=100, xmax=100, 后面跟着活跃事务列表
```

### 3.9 MVCC 一句话总结

> **PostgreSQL 从不原地修改数据，每次 UPDATE 都是"插入新版本 + 标记旧版本"。每个事务基于自己的快照决定看到哪个版本，让读写互不阻塞。VACUUM 负责定期清理那些没人看的旧版本。**

## 四、锁机制

### 4.1 锁的分类：两大维度

PostgreSQL 的锁可以从两个维度来理解：

| 维度 | 锁的对象 | 怎么加锁 | 数量 |
|-----|---------|---------|------|
| **表级锁** | 整张表 | 自动由 SQL 语句触发，或手动 `LOCK TABLE` | 8 种模式 |
| **行级锁** | 表中的某一行 | 在 `SELECT ... FOR UPDATE/SHARE` 时加锁 | 4 种模式 |

> **核心原则**：PostgreSQL 的锁是**自动加、自动放**的。你执行一条 SQL，它自动获取需要的锁，事务结束自动释放。你不需要手动管理锁，但需要理解每种操作会拿什么锁，才能避免锁冲突。

### 4.2 表级锁：8 种模式，从弱到强

PostgreSQL 的表级锁有 8 个级别，按照**互斥程度**从弱到强排列。理解它们的核心是：**强锁会阻塞弱锁，但弱锁之间可以共存**。

#### 4.2.1 锁级别一览

| 级别 | 锁模式 | 触发场景 | 通俗理解 |
|-----|-------|---------|---------|
| 1 | **ACCESS SHARE** | 普通 `SELECT` | "我就看看，不干啥" |
| 2 | **ROW SHARE** | `SELECT FOR UPDATE / FOR SHARE` | "我打算改这几行，先占个位" |
| 3 | **ROW EXCLUSIVE** | `INSERT / UPDATE / DELETE` | "我在改数据，别来改表结构" |
| 4 | **SHARE UPDATE EXCLUSIVE** | `VACUUM / ANALYZE / CREATE INDEX CONCURRENTLY` | "我在维护，别同时做同样的维护" |
| 5 | **SHARE** | `CREATE INDEX`（非 CONCURRENTLY） | "我在建索引，别写数据" |
| 6 | **SHARE ROW EXCLUSIVE** | `CREATE TRIGGER / 触发器操作` | "我在改触发器，别改数据也别建索引" |
| 7 | **EXCLUSIVE** | `REFRESH MATERIALIZED VIEW CONCURRENTLY` | "我在刷新物化视图，别写数据" |
| 8 | **ACCESS EXCLUSIVE** | `DROP TABLE / ALTER TABLE / TRUNCATE / VACUUM FULL` | "我要拆房子了，所有人都出去！" |

#### 4.2.2 锁冲突矩阵

下表是**已有锁（列）对新来锁（行）的阻塞关系**。`✗` = 冲突（等待），`✓` = 不冲突（可以同时执行）。

```
现有锁 →     AS    RS    RE   SUE    S   SRE    E   AE
新请求 ↓
ACCESS SHARE        ✓     ✓     ✓     ✓     ✓     ✓     ✓     ✗
ROW SHARE           ✓     ✓     ✓     ✓     ✓     ✓     ✗     ✗
ROW EXCLUSIVE       ✓     ✓     ✓     ✓     ✗     ✗     ✗     ✗
SHARE UPDATE EXCL   ✓     ✓     ✓     ✗     ✗     ✗     ✗     ✗
SHARE               ✓     ✓     ✗     ✗     ✓     ✗     ✗     ✗
SHARE ROW EXCLUSIVE ✓     ✓     ✗     ✗     ✗     ✗     ✗     ✗
EXCLUSIVE           ✓     ✗     ✗     ✗     ✗     ✗     ✗     ✗
ACCESS EXCLUSIVE    ✗     ✗     ✗     ✗     ✗     ✗     ✗     ✗
```

**从矩阵中能看出什么？**

- **ACCESS SHARE（SELECT）几乎不冲突任何东西**，只有 ACCESS EXCLUSIVE 会阻塞它。这就是为什么 `SELECT` 几乎不会被阻塞——除非有人在 `DROP TABLE`。
- **ACCESS EXCLUSIVE 冲突一切**。`ALTER TABLE`、`DROP TABLE`、`TRUNCATE` 会等所有操作结束才能执行。
- **ROW EXCLUSIVE（INSERT/UPDATE/DELETE）之间不冲突**。多人同时写数据不会被彼此阻塞（这要归功于 MVCC）。

#### 4.2.3 常见 SQL 对应的锁级别

```sql
-- 来看看你日常写的 SQL 到底拿了什么锁：

-- SELECT → 拿 ACCESS SHARE 锁（级别 1，最弱）
SELECT * FROM users;

-- INSERT / UPDATE / DELETE → 拿 ROW EXCLUSIVE 锁（级别 3）
INSERT INTO users VALUES (...);
UPDATE users SET age = 26 WHERE id = 1;
DELETE FROM users WHERE id = 1;

-- SELECT FOR UPDATE → 拿 ROW SHARE 锁（级别 2）
SELECT * FROM users WHERE id = 1 FOR UPDATE;

-- CREATE INDEX → 拿 SHARE 锁（级别 5），会阻塞写入
CREATE INDEX idx_name ON users(name);

-- CREATE INDEX CONCURRENTLY → 拿 SHARE UPDATE EXCLUSIVE（级别 4），不阻塞写入！
CREATE INDEX CONCURRENTLY idx_name ON users(name);

-- VACUUM → 拿 SHARE UPDATE EXCLUSIVE（级别 4）
VACUUM users;

-- DROP TABLE / ALTER TABLE / TRUNCATE → 拿 ACCESS EXCLUSIVE（级别 8，最强）
DROP TABLE users;
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
TRUNCATE TABLE users;
```

> **面试陷阱**：为什么 `CREATE INDEX CONCURRENTLY` 比普通 `CREATE INDEX` 好？因为普通建索引会拿 SHARE 锁（级别 5），阻塞 INSERT/UPDATE/DELETE；而 CONCURRENTLY 只拿 SHARE UPDATE EXCLUSIVE 锁（级别 4），和写入不冲突。代价是建索引更慢，且可能失败需要重试。

#### 4.2.4 手动加表锁

```sql
-- 显式锁表
BEGIN;
LOCK TABLE users IN ACCESS EXCLUSIVE MODE;  -- 最强锁，等所有操作结束
-- 做一些需要独占表的操作...
COMMIT;  -- 锁自动释放

-- 生产环境慎用！手动锁表会阻塞所有对该表的操作
```

### 4.3 行级锁：4 种模式，锁定特定行

行级锁比表级锁精细得多，只锁住某几行，不影响同一张表其他行的读写。

| 模式 | 通俗理解 | 谁会被阻塞 |
|-----|---------|-----------|
| **FOR UPDATE** | "这行我要改了，谁也别动" | 其他 `FOR UPDATE`、`FOR NO KEY UPDATE`、`FOR SHARE`、`FOR KEY SHARE` |
| **FOR NO KEY UPDATE** | "这行我要改，但我不动主键/唯一键" | 其他 `FOR UPDATE`、`FOR NO KEY UPDATE`（但不阻塞 `FOR KEY SHARE`） |
| **FOR SHARE** | "这行我要读，别改它" | 其他 `FOR UPDATE`、`FOR NO KEY UPDATE`（但允许其他 `FOR SHARE`、`FOR KEY SHARE`） |
| **FOR KEY SHARE** | "我依赖这行的主键，别删也别改键" | 其他 `FOR UPDATE`、`FOR NO KEY UPDATE`（允许其他 `FOR SHARE`、`FOR KEY SHARE`） |

#### 行级锁冲突矩阵

```
现有锁 →     FOR UPDATE   FOR NO KEY UPDATE   FOR SHARE   FOR KEY SHARE
新请求 ↓
FOR UPDATE           ✗              ✗               ✗              ✗
FOR NO KEY UPDATE    ✗              ✗               ✗              ✗
FOR SHARE            ✗              ✗               ✓              ✓
FOR KEY SHARE        ✗              ✓               ✓              ✓
```

**关键规律**：
- `FOR UPDATE` 是行级锁里最强的，跟谁都冲突
- `FOR SHARE` 和 `FOR KEY SHARE` 之间不冲突（共享锁之间兼容）
- `FOR KEY SHARE` 和 `FOR NO KEY UPDATE` 不冲突——这是 PostgreSQL 的精妙设计：**外键检查用 FOR KEY SHARE，普通更新用 FOR NO KEY UPDATE，两者互不阻塞**

#### 实际使用示例

```sql
-- 场景 1：先读后写，防止读到后数据被改
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 锁定这一行
-- 这里做业务判断...
UPDATE users SET balance = balance - 100 WHERE id = 1;  -- 安全更新
COMMIT;

-- 场景 2：转账——锁定两行，避免死锁（按相同顺序锁）
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 先锁 id 小的
SELECT * FROM accounts WHERE id = 2 FOR UPDATE;  -- 再锁 id 大的
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- 场景 3：不想等待，拿不到锁马上报错
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE NOWAIT;
-- 如果别人锁着这行，立刻报错，不等待

-- 场景 4：等待，但最多等 5 秒
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE SKIP LOCKED;
-- 跳过已被锁定的行，只处理能立即锁定的行（适合任务队列）
```

### 4.4 锁问题的排查

```sql
-- 查看当前锁
SELECT 
    l.locktype,
    l.relation::regclass,
    l.mode,
    l.granted,
    a.pid,
    a.query,
    a.state
FROM pg_locks l
JOIN pg_stat_activity a ON l.pid = a.pid
WHERE NOT l.granted;

-- 查看锁等待
SELECT 
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    blocked_activity.query AS blocked_statement,
    blocking_activity.query AS blocking_statement
FROM pg_locks blocked_locks
JOIN pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_locks blocking_locks ON blocked_locks.locktype = blocking_locks.locktype
    AND blocked_locks.relation = blocking_locks.relation
    AND blocked_locks.pid != blocking_locks.pid
JOIN pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;

-- 终止进程
SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE pid = 12345;
```

## 五、死锁处理

### 5.1 死锁产生

```sql
-- 会话 A
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 会话 B
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 2;
-- 会话 A
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 等待会话 B
-- 会话 B
UPDATE accounts SET balance = balance + 100 WHERE id = 1;  -- 等待会话 A（死锁）
```

### 5.2 死锁检测与处理

```sql
-- PostgreSQL 自动检测死锁并终止其中一个事务
-- 错误信息：ERROR: deadlock detected

-- 查看死锁日志
-- 在 postgresql.conf 中设置：
-- log_lock_waits = on
-- deadlock_timeout = 1s

-- 预防死锁的方法：
-- 1. 按相同顺序访问资源
-- 2. 保持事务简短
-- 3. 使用较低的隔离级别
-- 4. 使用 NOWAIT 立即返回错误

-- 使用 NOWAIT
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;
-- 如果无法立即获得锁，立即报错而不是等待
```

### 5.3 应用程序死锁处理

```python
# Python 示例：死锁重试机制
import psycopg2
from psycopg2 import OperationalError
import time

def execute_with_retry(conn, operation, max_retries=3):
    retry_count = 0
    while True:
        try:
            with conn.cursor() as cur:
                operation(cur)
            conn.commit()
            break
        except OperationalError as e:
            conn.rollback()
            if 'deadlock detected' in str(e) and retry_count < max_retries:
                retry_count += 1
                time.sleep(0.1 * retry_count)  # 指数退避
            else:
                raise

# 使用示例
def transfer_money(cur):
    cur.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1")
    cur.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 2")

conn = psycopg2.connect("dbname=mydb user=myuser")
execute_with_retry(conn, transfer_money)
```

## 八、最佳实践

### 8.1 事务设计原则

1. **保持事务简短**
   ```sql
   -- 不好
   BEGIN;
   -- 大量数据处理
   PERFORM pg_sleep(10);
   COMMIT;
   
   -- 好
   BEGIN;
   -- 最小必要操作
   COMMIT;
   ```

2. **按相同顺序访问资源**
   ```sql
   -- 所有事务都按相同顺序访问表
   BEGIN;
   UPDATE accounts SET balance -= 100 WHERE id = 1;
   UPDATE accounts SET balance += 100 WHERE id = 2;
   COMMIT;
   ```

3. **使用合适的隔离级别**
   ```sql
   -- 默认使用 READ COMMITTED
   -- 需要可重复读时使用 REPEATABLE READ
   -- 需要防止幻读时使用 SERIALIZABLE
   ```

### 8.2 错误处理最佳实践

```sql
-- 使用 PL/pgSQL 处理错误
CREATE OR REPLACE FUNCTION transfer_funds(
    from_account INT,
    to_account INT,
    amount NUMERIC
) RETURNS VOID AS $$
BEGIN
    -- 检查余额
    IF (SELECT balance FROM accounts WHERE id = from_account) < amount THEN
        RAISE EXCEPTION '余额不足';
    END IF;
    
    -- 执行转账
    UPDATE accounts SET balance = balance - amount WHERE id = from_account;
    UPDATE accounts SET balance = balance + amount WHERE id = to_account;
    
    -- 记录日志
    INSERT INTO transaction_log (from_account, to_account, amount)
    VALUES (from_account, to_account, amount);
    
EXCEPTION
    WHEN insufficient_funds THEN
        RAISE NOTICE '余额不足';
        ROLLBACK;
    WHEN OTHERS THEN
        RAISE NOTICE '错误: %', SQLERRM;
        ROLLBACK;
END;
$$ LANGUAGE plpgsql;
```
