# PostgreSQL 索引与性能优化

## 一、索引基础

### 1.1 什么是索引

索引是数据库中用于加速数据检索的数据结构。PostgreSQL 使用 B-Tree 作为默认索引类型，同时支持多种其他索引类型。

### 1.2 索引类型

| 索引类型 | 说明 | 适用场景 |
|---------|------|---------|
| **B-tree** | 默认类型，适合等值和范围查询 | 大多数场景 |
| **Hash** | 仅支持等值查询 | 等值查询 |
| **GiST** | 通用搜索树 | 几何、范围类型 |
| **SP-GiST** | 空间分区 GiST | 高维数据 |
| **GIN** | 通用倒排索引 | 数组、JSONB、全文搜索 |
| **BRIN** | 块范围索引 | 大表顺序数据 |

### 1.3 创建索引

```sql
-- 创建 B-tree 索引（默认）
CREATE INDEX idx_users_username ON users(username);

-- 创建唯一索引
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- 创建复合索引
CREATE INDEX idx_orders_user_date ON orders(user_id, order_date);

-- 创建部分索引
CREATE INDEX idx_active_users ON users(email) WHERE is_active = TRUE;

-- 创建函数索引
CREATE INDEX idx_users_lower_username ON users(LOWER(username));

-- 创建表达式索引
CREATE INDEX idx_users_age_group ON users((age / 10));

-- 创建 GIN 索引（JSONB）
CREATE INDEX idx_data_gin ON table_name USING GIN(data);

-- 创建 GIN 索引（数组）
CREATE INDEX idx_tags_gin ON articles USING GIN(tags);

-- 创建 GiST 索引（几何数据）
CREATE INDEX idx_location_gist ON places USING GiST(location);

-- 创建 BRIN 索引（大表）
CREATE INDEX idx_logs_brin ON logs USING BRIN(created_at);

-- 创建覆盖索引（INCLUDE，PostgreSQL 11+）
CREATE INDEX idx_users_email_cover ON users(email) INCLUDE (username, created_at);
```

### 1.4 管理索引

```sql
-- 查看表上的索引
\di users
-- 或
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'users';

-- 查看索引大小
SELECT 
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_indexes
JOIN pg_stat_user_indexes USING (indexrelname)
WHERE tablename = 'users';

-- 重建索引
REINDEX INDEX idx_users_username;
REINDEX TABLE users;
REINDEX DATABASE mydb;

-- 删除索引
DROP INDEX idx_users_username;
DROP INDEX IF EXISTS idx_users_username;

-- 重命名索引
ALTER INDEX idx_users_username RENAME TO idx_users_name;

-- 禁用索引（PostgreSQL 不支持直接禁用，可以标记为无效）
UPDATE pg_index SET indisvalid = FALSE WHERE indexrelname = 'idx_users_username';

-- 分析表（更新统计信息）
ANALYZE users;
ANALYZE VERBOSE users;
```

## 二、索引原理

### 2.1 B-Tree 结构

PostgreSQL 使用 Lehman-Yao 高并发 B-Tree 实现：

```
                    [Root Node]
                   /     |     \
            [Internal] [Internal] [Internal]
             /    |    \      /    |    \      /    |    \
        [Leaf] [Leaf] [Leaf] [Leaf] [Leaf] [Leaf] [Leaf] [Leaf] [Leaf]
        / | \  / | \  / | \  / | \  / | \  / | \  / | \  / | \  / | \
       数据项 数据项 数据项 数据项 数据项 数据项 数据项 数据项 数据项
```

**特点：**
- 平衡树，所有叶子节点在同一层
- 支持高并发读写
- 支持向前和向后扫描
- 自动维护平衡

### 2.2 索引扫描类型

```sql
-- 索引扫描（Index Scan）
-- 使用索引找到行，然后访问表获取数据
EXPLAIN SELECT * FROM users WHERE id = 1;

-- 仅索引扫描（Index Only Scan）
-- 所有需要的数据都在索引中，无需访问表
EXPLAIN SELECT username FROM users WHERE username = '张三';

-- 位图索引扫描（Bitmap Index Scan）
-- 使用位图合并多个索引条件
EXPLAIN SELECT * FROM users WHERE age > 25 AND age < 35;

-- 顺序扫描（Seq Scan）
-- 全表扫描
EXPLAIN SELECT * FROM users WHERE email LIKE '%@example.com';
```

### 2.3 索引选择原则

```sql
-- 1. 高选择性列适合建索引
-- 选择性 = 不同值数量 / 总行数
SELECT 
    COUNT(DISTINCT email) * 1.0 / COUNT(*) AS email_selectivity,
    COUNT(DISTINCT is_active) * 1.0 / COUNT(*) AS active_selectivity
FROM users;

-- 2. 小表不需要索引
-- 3. 频繁更新的列谨慎建索引
-- 4. 外键列应该建索引
-- 5. 经常用于 WHERE、JOIN、ORDER BY、GROUP BY 的列

-- 查看查询是否使用索引
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM users WHERE username = '张三';
```

## 三、查询优化

### 3.1 执行计划分析

```sql
-- 基础 EXPLAIN
EXPLAIN SELECT * FROM users WHERE age > 25;

-- 详细执行计划（实际执行）
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM users WHERE age > 25;

-- JSON 格式
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT * FROM users WHERE age > 25;

-- 查看执行计划节点类型
EXPLAIN SELECT * FROM users;
-- Seq Scan on users

EXPLAIN SELECT * FROM users WHERE id = 1;
-- Index Scan using users_pkey on users

EXPLAIN SELECT username FROM users WHERE username = '张三';
-- Index Only Scan using idx_users_username on users
```

### 3.2 执行计划节点

| 节点类型 | 说明 | 性能 |
|---------|------|------|
| **Seq Scan** | 顺序扫描 | 大表慢 |
| **Index Scan** | 索引扫描 | 通常快 |
| **Index Only Scan** | 仅索引扫描 | 最快 |
| **Bitmap Index Scan** | 位图索引扫描 | 多条件时快 |
| **Nested Loop** | 嵌套循环连接 | 小数据集快 |
| **Hash Join** | 哈希连接 | 大数据集快 |
| **Merge Join** | 合并连接 | 有序数据快 |
| **Sort** | 排序 | 有开销 |
| **Aggregate** | 聚合 | 视数据量 |

### 3.3 查询重写优化

```sql
-- 1. 避免在索引列上使用函数
-- 低效
SELECT * FROM users WHERE EXTRACT(YEAR FROM created_at) = 2023;

-- 高效
SELECT * FROM users 
WHERE created_at >= '2023-01-01' AND created_at < '2024-01-01';

-- 或者使用函数索引
CREATE INDEX idx_users_created_year ON users(EXTRACT(YEAR FROM created_at));

-- 2. 使用 UNION ALL 代替 UNION（如果不需要去重）
-- 低效
SELECT username FROM users WHERE age < 20
UNION
SELECT username FROM users WHERE age > 60;

-- 高效
SELECT username FROM users WHERE age < 20
UNION ALL
SELECT username FROM users WHERE age > 60;

-- 3. 使用 EXISTS 代替 IN（子查询）
-- 低效
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders);

-- 高效
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- 4. 避免 SELECT *
-- 低效
SELECT * FROM users WHERE id = 1;

-- 高效
SELECT id, username, email FROM users WHERE id = 1;

-- 5. 使用 LIMIT 代替 OFFSET（大数据量时）
-- 低效
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 100000;

-- 高效（键集分页）
SELECT * FROM users 
WHERE id > (SELECT id FROM users ORDER BY id OFFSET 100000 LIMIT 1)
ORDER BY id LIMIT 10;

-- 6. 使用 JOIN 代替子查询
-- 低效
SELECT u.*, (SELECT COUNT(*) FROM orders WHERE user_id = u.id) AS order_count
FROM users u;

-- 高效
SELECT u.*, COALESCE(o.order_count, 0) AS order_count
FROM users u
LEFT JOIN (SELECT user_id, COUNT(*) AS order_count FROM orders GROUP BY user_id) o
ON u.id = o.user_id;
```

## 四、统计信息

### 4.1 统计信息作用

PostgreSQL 查询优化器使用统计信息来估计行数和选择最优执行计划。

```sql
-- 查看表统计信息
SELECT 
    tablename,
    attname AS column_name,
    n_distinct,
    most_common_vals,
    most_common_freqs,
    histogram_bounds,
    correlation
FROM pg_stats
WHERE tablename = 'users';

-- 查看表行数估计
SELECT 
    relname,
    reltuples::bigint AS estimated_rows,
    pg_size_pretty(pg_relation_size(oid)) AS table_size
FROM pg_class
WHERE relname = 'users';

-- 手动更新统计信息
ANALYZE users;
ANALYZE users(username, email);  -- 指定列

-- 查看最后分析时间
SELECT 
    schemaname,
    tablename,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables
WHERE tablename = 'users';
```

### 4.2 自动统计信息收集

```sql
-- 查看 autovacuum 设置
SHOW autovacuum;
SHOW autovacuum_analyze_threshold;
SHOW autovacuum_analyze_scale_factor;

-- 修改表级别的统计信息收集
ALTER TABLE users SET (autovacuum_analyze_scale_factor = 0.05);

-- 禁用自动分析（不推荐）
ALTER TABLE users SET (autovacuum_enabled = FALSE);
```

## 五、表分区

### 5.1 声明式分区（PostgreSQL 10+）

```sql
-- 创建分区表
CREATE TABLE measurements (
    id SERIAL,
    city_id INT NOT NULL,
    logdate DATE NOT NULL,
    peaktemp INT,
    unitsales INT
) PARTITION BY RANGE (logdate);

-- 创建分区
CREATE TABLE measurements_y2024m01 PARTITION OF measurements
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE measurements_y2024m02 PARTITION OF measurements
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

CREATE TABLE measurements_y2024m03 PARTITION OF measurements
    FOR VALUES FROM ('2024-03-01') TO ('2024-04-01');

-- 默认分区
CREATE TABLE measurements_default PARTITION OF measurements DEFAULT;

-- 查看分区信息
SELECT 
    parent.relname AS parent_table,
    child.relname AS partition_name,
    pg_get_expr(child.relpartbound, child.oid) AS partition_range
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
WHERE parent.relname = 'measurements';

-- 分区维护
-- 分离分区
ALTER TABLE measurements DETACH PARTITION measurements_y2024m01;

-- 附加分区
ALTER TABLE measurements ATTACH PARTITION measurements_y2024m01
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

-- 删除分区
DROP TABLE measurements_y2024m01;
```

### 5.2 列表分区

```sql
-- 按列表分区
CREATE TABLE orders (
    id SERIAL,
    status VARCHAR(20),
    amount NUMERIC(10,2),
    created_at TIMESTAMP
) PARTITION BY LIST (status);

CREATE TABLE orders_pending PARTITION OF orders FOR VALUES IN ('pending');
CREATE TABLE orders_completed PARTITION OF orders FOR VALUES IN ('completed');
CREATE TABLE orders_cancelled PARTITION OF orders FOR VALUES IN ('cancelled');
```

### 5.3 哈希分区

```sql
-- 按哈希分区
CREATE TABLE events (
    id SERIAL,
    event_type VARCHAR(50),
    event_data JSONB,
    created_at TIMESTAMP
) PARTITION BY HASH (id);

CREATE TABLE events_p0 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE events_p1 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE events_p2 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE events_p3 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

## 七、性能监控

### 7.1 pg_stat_statements

```sql
-- 启用 pg_stat_statements（postgresql.conf）
shared_preload_libraries = 'pg_stat_statements'
pg_stat_statements.track = all

-- 查看最耗时的查询
SELECT 
    query,
    calls,
    total_time,
    mean_time,
    rows,
    100.0 * shared_blks_hit / nullif(shared_blks_hit + shared_blks_read, 0) AS hit_percent
FROM pg_stat_statements
ORDER BY mean_time DESC
LIMIT 10;

-- 重置统计
SELECT pg_stat_statements_reset();
```

### 7.2 系统统计视图

```sql
-- 表统计
SELECT 
    schemaname,
    tablename,
    seq_scan,
    seq_tup_read,
    idx_scan,
    idx_tup_fetch,
    n_tup_ins,
    n_tup_upd,
    n_tup_del,
    n_live_tup,
    n_dead_tup,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables
ORDER BY seq_scan DESC;

-- 索引统计
SELECT 
    schemaname,
    tablename,
    indexrelname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
ORDER BY idx_scan DESC;

-- 数据库活动
SELECT 
    pid,
    usename,
    application_name,
    client_addr,
    state,
    query_start,
    state_change,
    query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY query_start;

-- 锁等待
SELECT 
    w.pid AS waiting_pid,
    w.usename AS waiting_user,
    l.pid AS locking_pid,
    l.usename AS locking_user,
    w.query AS waiting_query,
    l.query AS locking_query
FROM pg_stat_activity w
JOIN pg_locks l1 ON w.pid = l1.pid AND NOT l1.granted
JOIN pg_locks l2 ON l1.locktype = l2.locktype AND l1.relation = l2.relation AND l2.granted
JOIN pg_stat_activity l ON l2.pid = l.pid;
```

### 7.3 缓冲区和 I/O

```sql
-- 查看缓冲区使用情况
SELECT 
    c.relname,
    pg_size_pretty(count(*) * 8192) AS buffered,
    round(100.0 * count(*) / (SELECT setting FROM pg_settings WHERE name='shared_buffers')::integer, 1) AS buffer_percent,
    round(100.0 * count(*) * 8192 / pg_table_size(c.oid), 1) AS percent_of_relation
FROM pg_class c
INNER JOIN pg_buffercache b ON b.relfilenode = c.relfilenode
INNER JOIN pg_database d ON (b.reldatabase = d.oid AND d.datname = current_database())
GROUP BY c.oid, c.relname
ORDER BY 3 DESC
LIMIT 10;

-- 查看表 I/O 统计
SELECT 
    schemaname,
    tablename,
    heap_blks_read,
    heap_blks_hit,
    round(100.0 * heap_blks_hit / nullif(heap_blks_hit + heap_blks_read, 0), 2) AS hit_ratio
FROM pg_statio_user_tables
ORDER BY heap_blks_read DESC;
```

## 八、常见问题与优化

### 8.1 表膨胀问题

```sql
-- 查看表膨胀
SELECT 
    schemaname,
    tablename,
    n_live_tup,
    n_dead_tup,
    round(n_dead_tup::numeric / nullif(n_live_tup, 0) * 100, 2) AS dead_tuple_ratio
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY dead_tuple_ratio DESC;

-- 手动清理
VACUUM ANALYZE users;
VACUUM FULL users;  -- 会锁表，谨慎使用

-- 查看自动清理设置
SELECT * FROM pg_autovacuum WHERE relname = 'users';

-- 调整自动清理参数
ALTER TABLE users SET (
    autovacuum_vacuum_scale_factor = 0.1,
    autovacuum_analyze_scale_factor = 0.05
);
```

## 九、最佳实践总结

### 9.1 索引设计原则

1. **主键自动创建 B-tree 索引**
2. **外键列创建索引**
3. **频繁查询的列创建索引**
4. **避免在频繁更新的列上创建过多索引**
5. **复合索引将选择性高的列放在前面**
6. **考虑使用部分索引减少索引大小**
7. **大表考虑使用 BRIN 索引**
8. **JSONB 和数组使用 GIN 索引**
9. **定期监控索引使用情况，删除未使用的索引**

### 9.2 查询优化原则

1. **避免在索引列上使用函数**
2. **避免隐式类型转换**
3. **使用参数化查询防止 SQL 注入**
4. **只查询需要的列，避免 SELECT ***
5. **使用 EXISTS 代替 IN 进行子查询**
6. **使用 UNION ALL 代替 UNION（如果不需要去重）**
7. **大数据量分页使用键集分页**
8. **定期运行 ANALYZE 更新统计信息**
9. **使用 EXPLAIN ANALYZE 分析查询性能**

### 9.3 维护建议

1. **定期运行 VACUUM ANALYZE**
2. **监控表膨胀情况**
3. **定期重建索引（REINDEX）**
4. **监控慢查询并优化**
5. **合理配置 autovacuum 参数**
6. **定期备份数据库**
7. **监控磁盘空间和连接数**
