# PostgreSQL 备份与恢复

## 一、备份恢复工具概览

PostgreSQL 备份恢复常用工具可以分为逻辑备份、全局备份、物理备份和恢复工具。先理解工具用途，再选择具体命令，能避免把 SQL 文件、自定义格式备份和物理备份混在一起。

| 工具 | 主要作用 | 常见场景 | 注意事项 |
|-----|---------|---------|---------|
| `pg_dump` | 逻辑备份单个数据库，可导出 SQL、自定义、目录、tar 等格式 | 单库备份、对象迁移、导出部分表或 schema | 备份通常不阻塞读写，但恢复前要确认版本兼容和对象依赖 |
| `pg_dumpall` | 逻辑备份整个实例，也可只备份角色、表空间等全局对象 | 整实例迁移、备份账号和表空间定义 | 大实例耗时较长，恢复顺序和权限需要提前规划 |
| `pg_restore` | 恢复 `pg_dump` 生成的自定义、目录或 tar 格式备份 | 选择性恢复、并行恢复、自定义格式恢复 | 不用于恢复纯 SQL 文件，纯 SQL 通常用 `psql` 执行 |
| `pg_basebackup` | 从运行中的实例生成物理基础备份 | 搭建从库、制作物理备份、PITR 基础备份 | 会产生网络和磁盘 IO，需确认复制权限、备份目录和 WAL 获取方式 |

常见选择方式：

| 目标 | 推荐工具 |
|-----|---------|
| 只备份一个业务库 | `pg_dump` |
| 备份角色、表空间等全局对象 | `pg_dumpall --globals-only` |
| 恢复 `.sql` 文本备份 | `psql` |
| 恢复 `.dump`、目录格式或 tar 格式备份 | `pg_restore` |
| 搭建从库或做物理基础备份 | `pg_basebackup` |

## 二、备份类型

### 1.1 逻辑备份

```sql
-- 使用 pg_dump 备份单个数据库
pg_dump -h localhost -p 5432 -U myuser -d mydb > mydb_backup.sql

-- 备份为自定义格式（支持选择性恢复）
pg_dump -h localhost -U myuser -d mydb -Fc > mydb_backup.dump

-- 备份为目录格式
pg_dump -h localhost -U myuser -d mydb -Fd -f mydb_backup_dir

-- 备份为 tar 格式
pg_dump -h localhost -U myuser -d mydb -Ft > mydb_backup.tar

-- 只备份表结构
pg_dump -h localhost -U myuser -d mydb --schema-only > mydb_schema.sql

-- 只备份数据
pg_dump -h localhost -U myuser -d mydb --data-only > mydb_data.sql

-- 备份特定表
pg_dump -h localhost -U myuser -d mydb -t users -t orders > mydb_tables.sql

-- 排除特定表
pg_dump -h localhost -U myuser -d mydb -T temp_table > mydb_backup.sql

-- 备份特定模式
pg_dump -h localhost -U myuser -d mydb -n public > mydb_public.sql

-- 并行备份（目录格式）
pg_dump -h localhost -U myuser -d mydb -Fd -f mydb_backup_dir -j 4
```

### 1.2 全局备份

```bash
# 备份所有数据库（pg_dumpall）
pg_dumpall -h localhost -U postgres > all_databases.sql

# 只备份全局对象（角色、表空间）
pg_dumpall --globals-only > globals.sql

# 备份角色
pg_dumpall --roles-only > roles.sql

# 备份表空间
pg_dumpall --tablespaces-only > tablespaces.sql
```

### 1.3 物理备份

```bash
# 使用 pg_basebackup 进行基础备份
pg_basebackup -h localhost -U replicator -D /backup/pg_backup -Fp -Xs -P -v

# 参数说明
# -D: 备份目录
# -Fp: 普通格式（不是 tar）
# -Ft: tar 格式
# -Xs: 流式 WAL
# -Xf: 获取 WAL 文件（不是流式）
# -P: 显示进度
# -v: 详细输出
# -R: 创建 standby.signal（用于流复制）

# 压缩备份
pg_basebackup -h localhost -U replicator -D - -Ft -Xs -z > backup.tar.gz

# 增量备份（PostgreSQL 17+）
pg_basebackup -h localhost -U replicator -D /backup/pg_backup --incremental=/backup/manifest
```

## 三、恢复操作

### 2.1 逻辑恢复

```bash
# 恢复 SQL 格式备份
psql -h localhost -U myuser -d mydb < mydb_backup.sql

# 创建数据库并恢复
createdb -h localhost -U myuser newdb
psql -h localhost -U myuser -d newdb < mydb_backup.sql

# 使用 pg_restore 恢复自定义格式
pg_restore -h localhost -U myuser -d mydb mydb_backup.dump

# 恢复到新数据库
createdb -h localhost -U myuser newdb
pg_restore -h localhost -U myuser -d newdb mydb_backup.dump

# 只恢复表结构
pg_restore --schema-only -h localhost -U myuser -d mydb mydb_backup.dump

# 只恢复数据
pg_restore --data-only -h localhost -U myuser -d mydb mydb_backup.dump

# 恢复特定表
pg_restore -t users -t orders -h localhost -U myuser -d mydb mydb_backup.dump

# 并行恢复（目录格式）
pg_restore -h localhost -U myuser -d mydb -j 4 mydb_backup_dir

# 查看备份内容
pg_restore --list mydb_backup.dump

# 选择性恢复（使用列表文件）
pg_restore --list mydb_backup.dump > restore.list
# 编辑 restore.list，注释掉不需要恢复的对象
pg_restore -L restore.list -h localhost -U myuser -d mydb mydb_backup.dump
```

### 2.2 物理恢复

```bash
# 停止 PostgreSQL
sudo systemctl stop postgresql

# 备份当前数据目录（可选）
mv /var/lib/postgresql/data /var/lib/postgresql/data_old

# 恢复基础备份
cp -r /backup/pg_backup /var/lib/postgresql/data

# 设置权限
chown -R postgres:postgres /var/lib/postgresql/data
chmod 700 /var/lib/postgresql/data

# 创建 recovery.signal（PostgreSQL 12+）
touch /var/lib/postgresql/data/recovery.signal

# 编辑 postgresql.conf 配置恢复
# restore_command = 'cp /archive/%f %p'
# recovery_target_time = '2024-01-15 14:30:00'
# recovery_target_action = 'promote'

# 启动 PostgreSQL
sudo systemctl start postgresql

# 恢复完成后，recovery.signal 会自动删除
```

### 2.3 时间点恢复（PITR）

```bash
# 1. 准备基础备份
pg_basebackup -h localhost -U replicator -D /backup/base -Fp -Xs -P

# 2. 配置 WAL 归档（postgresql.conf）
archive_mode = on
archive_command = 'cp %p /archive/%f'
wal_level = replica

# 3. 恢复时
# 停止 PostgreSQL
sudo systemctl stop postgresql

# 清理数据目录
rm -rf /var/lib/postgresql/data/*

# 恢复基础备份
cp -r /backup/base/* /var/lib/postgresql/data/

# 创建恢复信号
touch /var/lib/postgresql/data/recovery.signal

# 配置恢复参数（postgresql.conf 或 postgresql.auto.conf）
cat >> /var/lib/postgresql/data/postgresql.conf << EOF
restore_command = 'cp /archive/%f %p'
recovery_target_time = '2024-01-15 14:30:00'
recovery_target_action = 'promote'
EOF

# 启动 PostgreSQL
sudo systemctl start postgresql

# 监控恢复进度
tail -f /var/log/postgresql/postgresql-16-main.log
```

## 四、备份策略

### 3.1 小型数据库策略

```bash
#!/bin/bash
# daily_backup.sh

DB_NAME="mydb"
BACKUP_DIR="/backup/postgresql"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/${DB_NAME}_${DATE}.dump"

# 创建备份目录
mkdir -p ${BACKUP_DIR}

# 执行备份
pg_dump -h localhost -U postgres -d ${DB_NAME} -Fc > ${BACKUP_FILE}

# 压缩备份
gzip ${BACKUP_FILE}

# 删除7天前的备份
find ${BACKUP_DIR} -name "${DB_NAME}_*.dump.gz" -mtime +7 -delete

# 发送通知（可选）
echo "Backup completed: ${BACKUP_FILE}.gz" | mail -s "PostgreSQL Backup" admin@example.com
```

### 3.2 中型数据库策略

```bash
#!/bin/bash
# backup_strategy.sh

DB_NAME="mydb"
BACKUP_DIR="/backup/postgresql"
DATE=$(date +%Y%m%d)
DAY_OF_WEEK=$(date +%u)

# 周日进行完整备份
if [ "$DAY_OF_WEEK" -eq 7 ]; then
    # 完整备份
    pg_basebackup -h localhost -U replicator -D ${BACKUP_DIR}/base_${DATE} -Fp -Xs -P
    
    # 删除旧的完整备份（保留4周）
    find ${BACKUP_DIR} -name "base_*" -type d -mtime +28 -exec rm -rf {} \;
else
    # 其他日子进行逻辑备份
    pg_dump -h localhost -U postgres -d ${DB_NAME} -Fc > ${BACKUP_DIR}/${DB_NAME}_${DATE}.dump
    gzip ${BACKUP_DIR}/${DB_NAME}_${DATE}.dump
    
    # 删除7天前的逻辑备份
    find ${BACKUP_DIR} -name "${DB_NAME}_*.dump.gz" -mtime +7 -delete
fi

# 每小时备份 WAL（通过 archive_command）
```

### 3.3 大型数据库策略

```bash
#!/bin/bash
# large_db_backup.sh

DB_NAME="mydb"
BACKUP_DIR="/backup/postgresql"
DATE=$(date +%Y%m%d)

# 使用并行备份
pg_dump -h localhost -U postgres -d ${DB_NAME} -Fd -f ${BACKUP_DIR}/${DB_NAME}_${DATE} -j 4

# 压缩备份目录
tar -czf ${BACKUP_DIR}/${DB_NAME}_${DATE}.tar.gz -C ${BACKUP_DIR} ${DB_NAME}_${DATE}

# 删除原始目录
rm -rf ${BACKUP_DIR}/${DB_NAME}_${DATE}

# 物理备份（每周）
if [ $(date +%u) -eq 7 ]; then
    pg_basebackup -h localhost -U replicator -D ${BACKUP_DIR}/base_${DATE} -Fp -Xs -P -j 4
fi

# 异地备份（使用 rsync）
rsync -avz ${BACKUP_DIR}/ remote_server:/backup/postgresql/
```

## 五、高可用方案

### 4.1 流复制

```bash
# 主服务器配置（postgresql.conf）
listen_addresses = '*'
wal_level = replica
max_wal_senders = 10
wal_keep_size = 1GB
max_replication_slots = 10
hot_standby = on

# 主服务器配置（pg_hba.conf）
host replication replicator 192.168.1.0/24 scram-sha-256

# 创建复制用户
CREATE USER replicator WITH REPLICATION ENCRYPTED PASSWORD 'password';

# 备服务器设置
# 1. 停止 PostgreSQL
sudo systemctl stop postgresql

# 2. 清理数据目录
rm -rf /var/lib/postgresql/data/*

# 3. 使用 pg_basebackup
pg_basebackup -h primary_host -U replicator -D /var/lib/postgresql/data -Fp -Xs -P -R

# 4. 启动 PostgreSQL
sudo systemctl start postgresql

# 5. 验证复制状态
# 在主服务器上
SELECT * FROM pg_stat_replication;

# 在备服务器上
SELECT * FROM pg_stat_wal_receiver;
```

### 4.2 同步复制

```sql
-- 主服务器配置
synchronous_commit = on
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
-- 或
synchronous_standby_names = 'ANY 1 (standby1, standby2)'

-- 查看同步复制状态
SELECT 
    application_name,
    state,
    sync_state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn
FROM pg_stat_replication;
```

### 4.3 逻辑复制

```sql
-- 发布（主服务器）
CREATE PUBLICATION my_publication FOR TABLE users, orders;

-- 发布所有表
CREATE PUBLICATION all_tables FOR ALL TABLES;

-- 查看发布
SELECT * FROM pg_publication;

-- 订阅（备服务器）
CREATE SUBSCRIPTION my_subscription
CONNECTION 'host=primary_host dbname=mydb user=replicator password=password'
PUBLICATION my_publication;

-- 查看订阅状态
SELECT * FROM pg_stat_subscription;

-- 删除订阅
DROP SUBSCRIPTION my_subscription;
```

### 4.4 故障转移

```bash
# 手动故障转移

# 1. 在主服务器上（如果可访问）
pg_ctl promote -D /var/lib/postgresql/data

# 2. 或者使用 pg_ctl 命令
/usr/lib/postgresql/16/bin/pg_ctl promote -D /var/lib/postgresql/data

# 3. 使用 repmgr（推荐）
# 安装 repmgr
sudo apt-get install postgresql-16-repmgr

# 注册主服务器
repmgr -f /etc/repmgr.conf primary register

# 注册备服务器
repmgr -f /etc/repmgr.conf standby register

# 手动故障转移
repmgr -f /etc/repmgr.conf standby promote

# 自动故障转移（repmgrd）
# 配置 repmgr.conf
failover=automatic
promote_command='repmgr standby promote -f /etc/repmgr.conf'
follow_command='repmgr standby follow -f /etc/repmgr.conf'
```

## 六、常见问题处理

### 5.1 备份失败处理

```bash
# 检查磁盘空间
df -h

# 检查 PostgreSQL 连接
pg_isready -h localhost -p 5432

# 检查权限
ls -la /backup/

# 查看错误日志
tail -f /var/log/postgresql/postgresql-16-main.log

# 测试备份命令
pg_dump -h localhost -U postgres -d mydb -v 2>&1 | head -20
```

### 5.2 恢复失败处理

```bash
# 检查备份文件完整性
pg_restore --list mydb_backup.dump

# 检查数据库是否存在
psql -l

# 创建数据库（如果不存在）
createdb -U postgres mydb

# 检查权限
psql -c "\du"

# 查看详细错误
pg_restore -h localhost -U postgres -d mydb mydb_backup.dump 2>&1
```

### 5.3 复制延迟处理

```sql
-- 查看复制延迟
-- 在主服务器上
SELECT 
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS replication_lag
FROM pg_stat_replication;

-- 在备服务器上
SELECT 
    status,
    receive_start_lsn,
    received_lsn,
    latest_end_lsn,
    pg_size_pretty(pg_wal_lsn_diff(received_lsn, latest_end_lsn)) AS lag
FROM pg_stat_wal_receiver;

-- 处理延迟
-- 1. 增加 wal_keep_size
-- 2. 使用复制槽
SELECT * FROM pg_replication_slots;

-- 3. 检查网络
-- 4. 检查备服务器性能
```

### 5.4 WAL 归档问题

```bash
# 检查归档状态
psql -c "SELECT archived_count, failed_count, last_archived_time, last_failed_time FROM pg_stat_archiver;"

# 手动归档
psql -c "SELECT pg_switch_wal();"

# 检查归档目录
ls -la /archive/

# 清理旧 WAL
# 使用 pg_archivecleanup（备服务器）
pg_archivecleanup /archive 000000010000000100000010

# 或使用归档管理工具
```

## 七、最佳实践

### 6.1 备份策略建议

1. **完整备份**：每周至少一次（pg_basebackup）
2. **增量备份**：每天（pg_dump）
3. **WAL 归档**：持续进行
4. **验证备份**：定期测试恢复
5. **异地存储**：备份文件存储在不同位置
6. **保留策略**：
   - 本地：7天
   - 网络：30天
   - 异地：90天或更久

### 6.2 恢复演练

```bash
#!/bin/bash
# recovery_test.sh

# 定期恢复演练
TEST_DB="recovery_test_db"
BACKUP_FILE="/backup/postgresql/mydb_latest.dump"

# 创建测试数据库
dropdb --if-exists ${TEST_DB}
createdb ${TEST_DB}

# 恢复备份
pg_restore -d ${TEST_DB} ${BACKUP_FILE}

# 验证数据完整性
psql -d ${TEST_DB} -c "SELECT COUNT(*) FROM users;"
psql -d ${TEST_DB} -c "SELECT pg_database_size('${TEST_DB}');"

# 运行一致性检查
psql -d ${TEST_DB} -c "SELECT * FROM pg_stat_user_tables ORDER BY n_tup_ins DESC LIMIT 5;"

# 清理测试数据库
dropdb ${TEST_DB}

echo "恢复演练完成"
```

### 6.3 监控建议

```sql
-- 监控备份状态
SELECT 
    start_time,
    label,
    backup_mode,
    wal_mode,
    pg_size_pretty(total_byte_size) AS backup_size
FROM pg_stat_basebackup;

-- 监控复制状态
SELECT 
    application_name,
    state,
    sync_state,
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS lag
FROM pg_stat_replication;

-- 监控归档状态
SELECT 
    archived_count,
    last_archived_wal,
    last_archived_time,
    failed_count,
    last_failed_wal,
    last_failed_time,
    stats_reset
FROM pg_stat_archiver;
```
