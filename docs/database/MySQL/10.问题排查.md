# 问题排查

## 慢 SQL

### 开启慢查询日志

```sql
-- 查看当前状态
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';

-- 临时开启（重启失效）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;          -- 超过 1s 记录
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
```

永久配置写入 `my.cnf`：

```ini
[mysqld]
slow_query_log = ON
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = ON
```

### 用 mysqldumpslow 分析日志

```bash
# 按平均执行时间排序，取前 10
mysqldumpslow -s at -t 10 /var/log/mysql/slow.log

# 按总执行次数排序
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log
```

### 用 EXPLAIN 分析执行计划

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100;
```

重点关注字段：

| 字段 | 说明 |
|------|------|
| `type` | 访问类型，`ALL` 为全表扫描，最差；`ref`/`range`/`const` 较好 |
| `key` | 实际使用的索引，`NULL` 表示未命中索引 |
| `rows` | 预估扫描行数，越小越好 |
| `Extra` | `Using filesort`、`Using temporary` 需要重点优化 |

### 常见优化手段

```sql
-- 1. 强制使用索引
SELECT * FROM orders FORCE INDEX (idx_user_id) WHERE user_id = 100;

-- 2. 分析表统计信息
ANALYZE TABLE orders;

-- 3. 查看索引使用情况
SELECT * FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE object_schema = 'mydb' AND object_name = 'orders';
```

常见慢 SQL 原因及解法：

- **未命中索引**：检查查询条件字段是否有索引，注意隐式类型转换、函数包裹列会导致索引失效
- **回表过多**：考虑覆盖索引
- **排序/分组无索引**：为 `ORDER BY`、`GROUP BY` 字段建联合索引
- **深分页**：`LIMIT 100000, 10` 改为游标分页或延迟关联

```sql
-- 延迟关联优化深分页
SELECT o.* FROM orders o
JOIN (SELECT id FROM orders ORDER BY id LIMIT 100000, 10) t ON o.id = t.id;
```

## 长事务

### 查找当前长事务

```sql
-- 查看运行超过 60s 的事务
SELECT
    trx_id,
    trx_started,
    TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_sec,
    trx_state,
    trx_rows_modified,
    trx_query
FROM information_schema.INNODB_TRX
WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 60
ORDER BY duration_sec DESC;
```

```sql
-- 关联 processlist 获取连接信息
SELECT
    t.trx_id,
    t.trx_state,
    TIMESTAMPDIFF(SECOND, t.trx_started, NOW()) AS duration_sec,
    t.trx_query,
    p.id AS process_id,
    p.user,
    p.host,
    p.db
FROM information_schema.INNODB_TRX t
JOIN information_schema.PROCESSLIST p ON t.trx_mysql_thread_id = p.id
WHERE TIMESTAMPDIFF(SECOND, t.trx_started, NOW()) > 60;
```

### 终止长事务

```sql
-- 找到 process_id 后 kill
KILL 12345;
```

### 预防长事务

- 业务层设置合理超时：`SET SESSION innodb_lock_wait_timeout = 10`
- 避免在事务中做耗时操作（HTTP 调用、大量计算）
- 开启 `autocommit`，只在必要时显式开启事务
- 监控告警：对 `INNODB_TRX` 中超阈值的事务发出告警

## 死锁

### 查看最近一次死锁

```sql
SHOW ENGINE INNODB STATUS\G
```

输出中找到 `LATEST DETECTED DEADLOCK` 部分，包含：
- 两个事务各自持有/等待的锁
- 被回滚的事务（MySQL 自动选择代价小的一方回滚）

### 开启死锁日志

```sql
-- 将死锁信息持久化到错误日志
SET GLOBAL innodb_print_all_deadlocks = ON;
```

### 通过 performance_schema 分析锁等待

```sql
-- 查看当前锁等待关系
SELECT
    r.trx_id AS waiting_trx,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx,
    b.trx_query AS blocking_query
FROM information_schema.INNODB_LOCK_WAITS w
JOIN information_schema.INNODB_TRX r ON w.requesting_trx_id = r.trx_id
JOIN information_schema.INNODB_TRX b ON w.blocking_trx_id = b.trx_id;
```

> MySQL 8.0 起 `INNODB_LOCKS` / `INNODB_LOCK_WAITS` 迁移至 `performance_schema`：

```sql
SELECT * FROM performance_schema.data_lock_waits\G
SELECT * FROM performance_schema.data_locks\G
```

### 死锁常见原因及解法

| 原因 | 解法 |
|------|------|
| 多事务以不同顺序访问同一批行 | 统一加锁顺序（按主键升序） |
| 间隙锁 + 插入意向锁冲突 | 降低隔离级别至 RC，或优化查询避免大范围间隙锁 |
| 索引缺失导致锁升级为表锁 | 为查询条件加索引 |
| 批量操作拆分不当 | 拆小批次，减少单次锁持有时间 |

### 应用层处理死锁

死锁是 MySQL 主动检测并回滚的，错误码为 **1213**，应用层捕获后重试：

```python
import pymysql
from time import sleep

def execute_with_retry(conn, sql, args=None, max_retries=3):
    for attempt in range(max_retries):
        try:
            with conn.cursor() as cur:
                cur.execute(sql, args)
            conn.commit()
            return
        except pymysql.err.OperationalError as e:
            if e.args[0] == 1213 and attempt < max_retries - 1:  # Deadlock
                conn.rollback()
                sleep(0.1 * (attempt + 1))
            else:
                raise
```
