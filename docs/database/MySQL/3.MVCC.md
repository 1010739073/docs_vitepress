# 什么是MVCC
MVCC，全称Multi-Version Concurrency Control，即多版本并发控制，是一种在数据库管理系统中实现对数据库并发访问的方法。它通过为每个数据行维护多个版本，使得不同事务可以在不同的时间点看到不同版本的数据，从而实现了非锁定的读取操作，提高了并发性能。

# 当前读和快照读
- **当前读**：像`select...lock in share mode`（共享锁）、`select...for update`（排他锁）、`update`、`insert`、`delete`这些操作都是当前读。当前读读取的是记录的最新版本，并且会对读取的记录进行加锁，保证其他并发事务不能修改当前记录。
- **快照读**：像不加锁的`select`操作就是快照读，即不加锁的非阻塞读。快照读的前提是隔离级别不是串行化级别，串行化级别下的快照读会退化成当前读。快照读基于MVCC实现，可能读到的并不一定是数据的最新版本，而有可能是之前的历史版本。

MVCC的主要目的是实现读-写冲突不加锁，这里的读指的是快照读，而非当前读。当前读实际上是一种加锁的操作，属于悲观锁的实现。

# MVCC实现原理

## 隐式字段
每行记录除了我们自定义的字段外，InnoDB存储引擎还隐式定义了以下字段：
- **DB_TRX_ID**：6字节，最近修改（插入或更新）事务的ID，记录创建这条记录或最后一次修改该记录的事务ID。
- **DB_ROLL_PTR**：7字节，回滚指针，指向这条记录的上一版本（存储于rollback segment中）。
- **DB_ROW_ID**：6字节，隐含的自增ID（隐藏主键），如果数据表没有主键，InnoDB会自动以DB_ROW_ID产生一个聚簇索引。
- **删除标记**：一个隐藏的删除标记字段，记录被更新或删除时，并不真正删除记录，而是设置这个删除标记为true。

## undo日志
- **insert undo log**：事务在插入新记录时产生的undo log，只在事务回滚时需要，事务提交后可以立即丢弃。
- **update undo log**：事务在进行更新或删除操作时产生的undo log，不仅在事务回滚时需要，在快照读时也需要，所以不能随便删除。只有当快照读或事务回滚不再涉及该日志时，对应的日志才会被purge线程统一清除。

## Read View
Read View（读视图）是MVCC实现的核心概念，它是事务进行快照读操作时生成的一个数据快照。Read View主要包含以下信息：
- **m_ids**：当前系统中活跃的事务ID集合。
- **min_trx_id**：活跃事务中最小的事务ID。
- **max_trx_id**：系统中下一个要分配的事务ID。
- **creator_trx_id**：创建该Read View的事务ID。

### Read View的工作原理
当事务进行快照读时，会根据以下规则判断记录的可见性：
1. 如果记录的DB_TRX_ID等于creator_trx_id，说明是当前事务修改的记录，可见。
2. 如果记录的DB_TRX_ID小于min_trx_id，说明该记录在当前事务开始前就已经提交，可见。
3. 如果记录的DB_TRX_ID在min_trx_id和max_trx_id之间，需要判断该事务ID是否在m_ids中：
   - 如果在，说明该事务是活跃的，记录不可见。
   - 如果不在，说明该事务已经提交，记录可见。
4. 如果记录的DB_TRX_ID大于等于max_trx_id，说明该记录是在当前事务开始后创建的，不可见。

## purge
- 为了实现InnoDB的MVCC机制，更新或删除操作都只是设置一个老记录的deleted_bit为true，并不真正将过时的记录删除。
- purge线程负责清理deleted_bit为true的记录。purge线程也维护了一个Read View（相当于系统中最老活跃事务的Read View）。如果某个记录的deleted_bit为true，并且其DB_TRX_ID相对于purge线程的Read View可见，那么这条记录可以被安全清除。

# MVCC在不同隔离级别下的表现
- **读未提交（READ UNCOMMITTED）**：不使用MVCC，直接读取最新版本的数据。
- **读已提交（READ COMMITTED）**：每次查询都会生成一个新的Read View，因此可以看到其他事务已提交的修改。
- **可重复读（REPEATABLE READ）**：事务开始时生成一个Read View，整个事务期间使用同一个Read View，因此可以保证同一事务内多次读取结果一致。
- **串行化（SERIALIZABLE）**：不使用MVCC，直接对所有操作加锁，保证事务的完全串行执行。

# MVCC的优势
1. **提高并发性能**：避免了读-写冲突，读操作不需要加锁，提高了系统的并发处理能力。
2. **减少死锁**：由于读操作不需要加锁，减少了死锁的发生概率。
3. **实现一致性读**：通过Read View机制，实现了不同隔离级别的一致性读。
4. **简化应用开发**：开发者不需要手动处理并发冲突，由数据库自动管理。

# MVCC的局限性
1. **增加存储开销**：需要维护多个版本的数据，增加了存储空间。
2. **增加CPU和内存开销**：需要管理和维护undo日志和Read View。
3. **可能导致数据不一致**：在某些情况下，快照读可能读到过时的数据。
4. **对长事务不友好**：长事务会导致undo日志积累，增加系统负担。
