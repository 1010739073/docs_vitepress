# 主从复制

## 主从状态检查

### 主库状态

```sql
-- 查看主库 binlog 位置（传统复制用于配置从库）
SHOW MASTER STATUS\G

-- 查看所有从库的复制状态
SHOW SLAVE HOSTS;

-- 查看 binlog 列表
SHOW BINARY LOGS;

-- 查看 binlog 事件（排查 binlog 内容）
SHOW BINLOG EVENTS IN 'mysql-bin.000001' LIMIT 20;
```

### 从库状态

```sql
-- 查看从库完整复制状态（最常用）
SHOW SLAVE STATUS\G
```

关键字段说明：

| 字段 | 说明 |
|------|------|
| `Slave_IO_Running` | IO 线程状态，正常应为 `Yes` |
| `Slave_SQL_Running` | SQL 线程状态，正常应为 `Yes` |
| `Seconds_Behind_Master` | 复制延迟秒数，`0` 表示已追上；`NULL` 表示 IO 线程未连接 |
| `Master_Host` | 当前连接的主库地址 |
| `Master_Log_File` | IO 线程正在读取的主库 binlog 文件 |
| `Read_Master_Log_Pos` | IO 线程已读到的 binlog 位置 |
| `Relay_Master_Log_File` | SQL 线程正在执行的主库 binlog 文件 |
| `Exec_Master_Log_Pos` | SQL 线程已执行到的主库 binlog 位置 |
| `Last_IO_Error` | IO 线程最近的错误信息 |
| `Last_SQL_Error` | SQL 线程最近的错误信息 |

> MySQL 8.0.22+ 推荐用 `SHOW REPLICA STATUS\G`，`SLAVE` 相关语法已废弃但仍兼容。

## 启动和停止从库同步

```sql
-- 启动所有复制线程（IO + SQL）
START SLAVE;

-- 只启动 IO 线程（只拉取 binlog，不执行）
START SLAVE IO_THREAD;

-- 只启动 SQL 线程（执行已拉取的 relay log）
START SLAVE SQL_THREAD;

-- 停止所有复制线程
STOP SLAVE;

-- 只停止 IO 线程
STOP SLAVE IO_THREAD;

-- 只停止 SQL 线程
STOP SLAVE SQL_THREAD;
```

> MySQL 8.0.22+ 对应命令为 `START REPLICA` / `STOP REPLICA`。

## 跳过复制错误

当 SQL 线程报错卡住时（如主从数据冲突），可跳过单条出错事务：

```sql
-- 停止 SQL 线程
STOP SLAVE SQL_THREAD;

-- 跳过 1 个错误事务（GTID 模式不适用，见下方）
SET GLOBAL sql_slave_skip_counter = 1;

-- 重新启动 SQL 线程
START SLAVE SQL_THREAD;
```

**GTID 模式下跳过错误：**

```sql
STOP SLAVE;

-- 将出错的 GTID 注入为空事务（替换为实际报错的 GTID）
SET GTID_NEXT = 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx:N';
BEGIN; COMMIT;
SET GTID_NEXT = 'AUTOMATIC';

START SLAVE;
```

## 变更主节点（CHANGE MASTER TO）

### 基于 binlog 位置（传统复制）

先在主库执行 `SHOW MASTER STATUS` 获取 `File` 和 `Position`，然后在从库执行：

```sql
STOP SLAVE;

CHANGE MASTER TO
  MASTER_HOST     = '192.168.1.100',
  MASTER_PORT     = 3306,
  MASTER_USER     = 'repl',
  MASTER_PASSWORD = 'repl_password',
  MASTER_LOG_FILE = 'mysql-bin.000010',   -- 主库 SHOW MASTER STATUS 的 File
  MASTER_LOG_POS  = 154;                  -- 主库 SHOW MASTER STATUS 的 Position

START SLAVE;
SHOW SLAVE STATUS\G
```

### 基于 GTID（推荐）

GTID 模式下无需指定 binlog 文件和位置，MySQL 自动定位续传点：

```sql
STOP SLAVE;

CHANGE MASTER TO
  MASTER_HOST         = '192.168.1.100',
  MASTER_PORT         = 3306,
  MASTER_USER         = 'repl',
  MASTER_PASSWORD     = 'repl_password',
  MASTER_AUTO_POSITION = 1;               -- 启用 GTID 自动定位

START SLAVE;
SHOW SLAVE STATUS\G
```

### MySQL 8.0.23+ 新语法

`SLAVE` 关键字已替换为 `REPLICA`：

```sql
STOP REPLICA;

CHANGE REPLICATION SOURCE TO
  SOURCE_HOST         = '192.168.1.100',
  SOURCE_PORT         = 3306,
  SOURCE_USER         = 'repl',
  SOURCE_PASSWORD     = 'repl_password',
  SOURCE_AUTO_POSITION = 1;

START REPLICA;
SHOW REPLICA STATUS\G
```

## 重置复制

```sql
-- 停止并清除所有复制配置和 relay log（从库执行）
STOP SLAVE;
RESET SLAVE ALL;     -- ALL 表示同时清除 master.info 中的连接信息

-- 清除主库 binlog（危险：会删除 binlog 文件，谨慎操作）
RESET MASTER;        -- 清除所有 binlog，GTID 模式下会重置 gtid_executed

-- 清除主库 binlog 到指定文件前
PURGE BINARY LOGS TO 'mysql-bin.000010';
PURGE BINARY LOGS BEFORE '2024-01-01 00:00:00';
```
