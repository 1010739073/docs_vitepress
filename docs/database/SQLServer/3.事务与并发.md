# SQL Server 事务与并发

## 1. 事务基础

### 什么是事务

事务是一组数据库操作的集合，这些操作要么全部成功，要么全部失败。事务具有以下四个特性（ACID）：

- **原子性（Atomicity）**：事务中的操作要么全部完成，要么全部不完成
- **一致性（Consistency）**：事务执行前后，数据库从一个一致性状态转变为另一个一致性状态
- **隔离性（Isolation）**：并发执行的事务之间互不干扰
- **持久性（Durability）**：事务提交后，对数据库的修改是永久的

### 事务示例

```sql
-- 开始事务
BEGIN TRANSACTION;

BEGIN TRY
    -- 扣除账户余额
    UPDATE Accounts SET Balance = Balance - 1000 WHERE AccountID = 1;
    
    -- 增加账户余额
    UPDATE Accounts SET Balance = Balance + 1000 WHERE AccountID = 2;
    
    -- 记录转账日志
    INSERT INTO TransactionLog (FromAccount, ToAccount, Amount, TransactionDate)
    VALUES (1, 2, 1000, GETDATE());
    
    -- 提交事务
    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    -- 回滚事务
    ROLLBACK TRANSACTION;
    
    -- 记录错误
    DECLARE @ErrorMessage NVARCHAR(4000) = ERROR_MESSAGE();
    PRINT 'Transaction failed: ' + @ErrorMessage;
END CATCH
```

## 2. 事务隔离级别

### 隔离级别概述

SQL Server 支持四种事务隔离级别：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 描述 |
|----------|------|------------|------|------|
| READ UNCOMMITTED | 是 | 是 | 是 | 读取未提交的数据 |
| READ COMMITTED | 否 | 是 | 是 | 只读取已提交的数据（默认） |
| REPEATABLE READ | 否 | 否 | 是 | 可重复读 |
| SNAPSHOT | 否 | 否 | 否 | 快照隔离 |
| SERIALIZABLE | 否 | 否 | 否 | 最高隔离级别 |

### 设置隔离级别

```sql
-- 设置会话隔离级别
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET TRANSACTION ISOLATION LEVEL SNAPSHOT;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 查看当前隔离级别
DBCC USEROPTIONS;
```

### 隔离级别示例

```sql
-- 使用 READ COMMITTED 隔离级别
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN TRANSACTION;
SELECT Balance FROM Accounts WHERE AccountID = 1;
-- 此时其他事务可以看到这个查询
COMMIT TRANSACTION;

-- 使用 SERIALIZABLE 隔离级别
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN TRANSACTION;
SELECT Balance FROM Accounts WHERE AccountID = 1;
-- 此时其他事务不能修改 AccountID = 1 的数据
COMMIT TRANSACTION;
```

## 3. 锁机制

### 锁类型

SQL Server 使用多种锁类型来控制并发访问：

| 锁类型 | 描述 | 兼容性 |
|--------|------|--------|
| 共享锁（S） | 用于读操作，允许其他事务读取 | 与 S、U、IX 兼容 |
| 更新锁（U） | 介于共享锁和排他锁之间 | 与 S 兼容 |
| 排他锁（X） | 用于写操作，完全锁定 | 不与任何锁兼容 |
| 意向锁（IS、IX、SIX） | 表示事务意图获取更细粒度的锁 | 相应兼容 |
| 架构锁（Sch-M、Sch-S） | 用于架构修改 | 极少兼容 |

### 查看锁信息

```sql
-- 查看当前锁
SELECT 
    resource_type,
    resource_description,
    request_mode,
    request_status,
    resource_associated_entity_id
FROM sys.dm_tran_locks
WHERE resource_database_id = DB_ID();

-- 查看阻塞事务
SELECT 
    blocking_session_id,
    session_id,
    wait_type,
    wait_time,
    wait_resource
FROM sys.dm_exec_requests
WHERE blocking_session_id <> 0;

-- 查看死锁信息
SELECT 
    xdr.value('@timestamp', 'datetime') AS timestamp,
    xdr.query('.') AS deadlock_graph
FROM (
    SELECT CAST(target_data AS XML) AS target_data
    FROM sys.dm_xe_session_targets st
    JOIN sys.dm_xe_sessions s ON s.address = st.event_session_address
    WHERE s.name = 'system_health'
      AND st.target_name = 'ring_buffer'
) AS Data
CROSS APPLY target_data.nodes('//RingBufferTarget/event') AS xdr(xdr)
WHERE xdr.xdr.value('@name', 'varchar(4000)') = 'xml_deadlock_report';
```

### 锁提示

```sql
-- 使用锁提示控制锁定行为
SELECT * FROM Users WITH (NOLOCK);  -- 等同于 READ UNCOMMITTED
SELECT * FROM Users WITH (HOLDLOCK);  -- 等同于 REPEATABLE READ
SELECT * FROM Users WITH (NOEXPAND);  -- 不使用索引视图
SELECT * FROM Users WITH (UPDLOCK);  -- 使用更新锁
SELECT * FROM Users WITH (XLOCK);  -- 使用排他锁
SELECT * FROM Users WITH (ROWLOCK);  -- 行级锁
SELECT * FROM Users WITH (PAGLOCK);  -- 页级锁
SELECT * FROM Users WITH (TABLOCK);  -- 表级锁
```

## 4. 死锁处理

### 死锁原因

死锁通常由以下原因引起：
- 事务以不同顺序访问资源
- 长时间持有锁
- 缺乏适当的索引
- 隔离级别设置不当

### 死锁示例

```sql
-- 连接 1
BEGIN TRANSACTION;
UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1;
-- 等待连接 2 释放 AccountID = 2 的锁
UPDATE Accounts SET Balance = Balance + 100 WHERE AccountID = 2;
COMMIT TRANSACTION;

-- 连接 2（并发执行）
BEGIN TRANSACTION;
UPDATE Accounts SET Balance = Balance - 200 WHERE AccountID = 2;
-- 等待连接 1 释放 AccountID = 1 的锁
UPDATE Accounts SET Balance = Balance + 200 WHERE AccountID = 1;
COMMIT TRANSACTION;
```

### 死锁处理策略

```sql
-- 设置死锁优先级
SET DEADLOCK_PRIORITY LOW;  -- 当前事务作为死锁受害者
SET DEADLOCK_PRIORITY HIGH;  -- 当前事务优先
SET DEADLOCK_PRIORITY NORMAL;  -- 默认优先级

-- 使用 TRY-CATCH 处理死锁
BEGIN TRY
    BEGIN TRANSACTION;
    -- 你的事务代码
    UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1;
    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    IF ERROR_NUMBER() = 1205  -- 死锁错误
    BEGIN
        ROLLBACK TRANSACTION;
        -- 重试逻辑
        WAITFOR DELAY '00:00:01';
        -- 重新执行事务
    END
    ELSE
    BEGIN
        ROLLBACK TRANSACTION;
        THROW;
    END
END CATCH
```

## 5. Go 事务示例

```go
package main

import (
    "context"
    "database/sql"
    "fmt"
    "log"
    
    _ "github.com/microsoft/go-mssqldb"
)

func main() {
    connString := "server=localhost;user id=sa;password=YourPassword;database=TestDB"
    db, err := sql.Open("sqlserver", connString)
    if err != nil {
        log.Fatal(err)
    }
    defer db.Close()
    
    // 开始事务
    tx, err := db.BeginTx(context.Background(), nil)
    if err != nil {
        log.Fatal(err)
    }
    
    // 使用 defer 处理事务回滚
    defer func() {
        if p := recover(); p != nil {
            tx.Rollback()
            panic(p)  // 重新抛出 panic
        } else if err != nil {
            tx.Rollback()
        } else {
            err = tx.Commit()
            if err != nil {
                log.Fatal(err)
            }
        }
    }()
    
    // 执行事务操作
    _, err = tx.Exec(`
        UPDATE Accounts SET Balance = Balance - @Amount WHERE AccountID = @FromID
    `, sql.Named("Amount", 1000), sql.Named("FromID", 1))
    if err != nil {
        return
    }
    
    _, err = tx.Exec(`
        UPDATE Accounts SET Balance = Balance + @Amount WHERE AccountID = @ToID
    `, sql.Named("Amount", 1000), sql.Named("ToID", 2))
    if err != nil {
        return
    }
    
    fmt.Println("事务执行成功")
}
```

## 6. 并发控制最佳实践

### 避免死锁

```sql
-- 1. 统一访问顺序
-- 好的做法：总是先更新账户1，再更新账户2
BEGIN TRANSACTION;
UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1;
UPDATE Accounts SET Balance = Balance + 100 WHERE AccountID = 2;
COMMIT TRANSACTION;

-- 2. 保持事务简短
BEGIN TRANSACTION;
-- 只包含必要的操作
UPDATE Users SET LastLogin = GETDATE() WHERE UserID = 1;
COMMIT TRANSACTION;

-- 3. 使用合适的隔离级别
-- 大多数情况使用 READ COMMITTED
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 4. 避免用户交互
-- 不要在事务中等待用户输入
```

### 优化锁策略

```sql
-- 1. 为查询添加适当的索引
-- 避免全表扫描导致的锁升级
CREATE INDEX IX_Accounts_AccountID ON Accounts(AccountID);

-- 2. 使用更小的隔离级别
-- 如果允许脏读，可以使用 NOLOCK
SELECT * FROM Users WITH (NOLOCK) WHERE Age > 25;

-- 3. 控制事务时间
-- 设置锁等待超时
SET LOCK_TIMEOUT 5000;  -- 5秒

-- 4. 使用快照隔离
-- 启用快照隔离
ALTER DATABASE TestDB SET ALLOW_SNAPSHOT_ISOLATION ON;
SET TRANSACTION ISOLATION LEVEL SNAPSHOT;
```

## 7. 高并发场景处理

### 乐观并发控制

```sql
-- 添加版本号列
ALTER TABLE Products ADD VersionNumber ROWVERSION;

-- 更新时检查版本号
UPDATE Products 
SET Price = @NewPrice, 
    VersionNumber = @NewVersion
WHERE ProductID = @ProductID 
  AND VersionNumber = @OldVersion;

-- 检查是否更新成功
IF @@ROWCOUNT = 0
BEGIN
    PRINT '版本冲突，数据已被其他用户修改';
END
```

### 悲观并发控制

```sql
-- 使用 UPDLOCK 锁定记录
BEGIN TRANSACTION;
SELECT * FROM Products WITH (UPDLOCK) WHERE ProductID = 1;

-- 在应用层处理业务逻辑
-- ...

-- 更新数据
UPDATE Products SET Price = 100 WHERE ProductID = 1;

COMMIT TRANSACTION;
```

## 8. 性能监控

### 监控事务

```sql
-- 查看当前活动事务
SELECT 
    transaction_id,
    transaction_begin_time,
    transaction_type,
    transaction_state,
    transaction_name
FROM sys.dm_tran_active_transactions;

-- 查看事务锁
SELECT 
    t.transaction_id,
    l.resource_type,
    l.request_mode,
    l.request_status
FROM sys.dm_tran_active_transactions t
JOIN sys.dm_tran_locks l ON t.transaction_id = l.request_owner_id;

-- 查看死锁统计
SELECT 
    cntr_value AS Deadlocks
FROM sys.dm_os_performance_counters
WHERE counter_name = 'Number of Deadlocks/sec';
```

## 9. 事务配置

### 数据库事务设置

```sql
-- 设置数据库为单一用户模式
ALTER DATABASE TestDB SET SINGLE_USER WITH ROLLBACK IMMEDIATE;

-- 设置数据库为多用户模式
ALTER DATABASE TestDB SET MULTI_USER;

-- 设置数据库为只读模式
ALTER DATABASE TestDB SET READ_ONLY;

-- 设置数据库为读写模式
ALTER DATABASE TestDB SET READ_WRITE;

-- 启用快照隔离
ALTER DATABASE TestDB SET ALLOW_SNAPSHOT_ISOLATION ON;

-- 启用读提交快照隔离
ALTER DATABASE TestDB SET READ_COMMITTED_SNAPSHOT ON;
```

## 10. 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 死锁 | 事务访问顺序不一致 | 统一访问顺序，使用锁提示 |
| 阻塞 | 锁等待时间过长 | 优化查询，添加索引，减少锁持有时间 |
| 超时 | 事务执行时间过长 | 优化 SQL，使用合适的隔离级别 |
| 幻读 | 隔离级别过低 | 提高隔离级别到 SERIALIZABLE |
| 锁升级 | 行锁升级到表锁 | 优化查询，使用分区表 |
